## 坑 1：用 16 位节拍变量做按键计时会在几十秒后卡死

> 第八届国赛 · 发现者：INT姐

### 现象

程序刚上电时按键一切正常，按了几次之后却可能整机卡死，后面按什么都没反应。这个问题不一定立刻出现，所以很容易被误判成“偶发小 bug”。

### 原因

`uwTick` 被定义成了 `int`，不是 `long int`。在 C51 里，16 位节拍变量只能撑几十秒，溢出后基于时间差的按键判定、超时逻辑就会一起失真，最后表现成整机卡死或按键完全失效。

```c
// 错误写法：毫秒节拍只有 16 位，运行几十秒后就会溢出
unsigned int uwTick;
```

### 解决方法

所有“系统节拍”“时间戳”“超时比较”变量统一改成 32 位整型，别让核心计时逻辑站在 16 位的小细腿上。

```c
// 正确写法：心跳改到 32 位
unsigned long int uwTick;
```

### 举一反三

- 只要变量单位是 `ms`，又要跨几十秒甚至几分钟，就别用 16 位类型
- 去抖、长按、双击、超时状态机，本质都依赖时间差，底层节拍一溢出会一起陪葬
- 竞赛里“过一会儿才出问题”的 bug，优先排查计数器位宽

| 变量类型 | 作用领域 | 存在周期 |
| --- | --- | --- |

| 静态局部变量 | 函数 / 代码块内部（与局部变量相同） | 程序整个运行期间（与全局变量相同） |

> 关联模块：数码管

### 整理记录

* **问题描述**：
* **问题背景**：哔哩哔哩课程蓝桥杯模块部分，采用蓝桥杯单片机实物板
* **问题分析**：delay12ms使用的是stcisp生成的stc89的

```
void Delay12us(void) //@12.000MHz
{
	unsigned char data i;
	nop();
	i = 3;
	while (--i);
}
```

* **解决方案**：

```
void Delay12us(void) //@12.000MHz
{
	unsigned char data i;
	nop();
	nop();
```

```c
if (key_down == 4) {

// 处理S6 - 无法进入
}
// 修改后（正确）
if (key_down == 4) {
// 处理S4
}
if (key_down == 6) {
// 处理S6 - 可以正常进入
}
```

## 坑 47：组合按键判断位置错误

> 来源：[063_蓝桥杯单片机第二十二周周报（二）.md](./单片机答疑数据库/063_蓝桥杯单片机第二十二周周报（二）.md)
> 周期：蓝桥杯单片机第二十二周周报（二）

### 整理记录

- **问题描述：** S8和S9组合按键清空继电器计数不成功
- **问题背景：** 第16届省赛
- **问题分析：** 在 `Key_Read()` 矩阵键盘扫描函数中，S8+S9的组合按键判断 `if(P30 == 0 && P32 == 0) temp = 89;` 被放在了该列单键扫描的**中间位置**。扫描时代码从上往下执行，S8（`if(P30 == 0) temp = 9;`）先被匹配到并赋值 `temp = 9`，后续的组合判断虽然条件也满足，但已经晚了一步（或者根据具体代码结构，组合判断可能被跳过）
```c
// 错误写法：组合判断夹在单键扫描中间
if(P30 == 0) temp = 9;
if(P30 == 0 && P32 == 0) temp = 89; // ← 放在这里
if(P30 == 0) temp = 11;
// ...
// 正确写法：组合判断放在该列扫描的最后
if(P30 == 0) temp = 9;
if(P30 == 0) temp = 11;
// ...其他单键扫描
if(P30 == 0 && P32 == 0) temp = 89; // ← 放最后，覆盖单键值
```
- **解决方案：** 将组合按键判断移到该列所有单键扫描的**最后面**，利用后赋值覆盖前赋值的特性，确保两键同时按下时 `temp` 最终被赋为组合键值89
> **规律总结：** 矩阵键盘中实现组合按键时，组合判断必须放在单键判断之后，利用"后覆盖"机制保证组合键优先级高于单键。

## 坑 52：switch-default 无按键时误触发导致 L3 常亮 + EEPROM 保存时机错误（2处修复）

> 来源：[066_蓝桥杯单片机第二十三周周报.md](./单片机答疑数据库/066_蓝桥杯单片机第二十三周周报.md)
> 周期：蓝桥杯单片机第二十三周周报

### 整理记录

- **问题描述：** 4T 测评中 9 个测试点未通过，其中 7 个为指示灯状态错误（#13、#15、#18、#19、#20、#21、#23），2 个为 EEPROM 存储值错误（#11、#25）。学员反馈"LED 模块的代码检查了几次都没找到问题，底层模块也没问题"
- **问题背景：** 第十一届省赛，涉及电压阈值计数、LED 错误指示、EEPROM 参数保存
- **问题分析：**
- [a] **switch-default 在无按键时被反复触发（核心问题）：** 按键扫描每 10ms 执行一次，无按键按下时 `Key_Down = 0`。`Key_Down = 0` 进入 `switch` 后不匹配 12、13、16、17 中的任何一个 `case`，于是落入 `default` 分支执行 `Error_Key++`。仅 30ms 后 `Error_Key >= 3`，触发 `ucLed[2] = 1`，L3（错误指示灯）几乎永远亮着。学员在 LED 模块中反复查找问题，但根因不在 LED 模块，而在按键处理的 `default` 分支
- [b] **EEPROM 保存时机错误：** 学员在 S12 按键处理中先判断 `Seg_Disp_Mode == 2` 再 `++Seg_Disp_Mode`，意图是"离开计数界面时保存"。但题目要求的保存时机是"离开参数界面、进入计数界面时保存"。正确逻辑应先 `++Seg_Disp_Mode`，再判断切换后 `== 2` 则保存。测试点 #25 期望 EEPROM 值为 25，实际读到 15（上一次的旧值），正是因为保存发生在错误的时机
- **解决方案：**
- [a] 在 `switch` 前加 `if(Key_Down == 0) return;`，排除无按键状态，避免 `default` 被误触发：
```c
if(Key_Down == 0) return; // 没有按键按下，直接返回
switch(Key_Down)
{
case 12: ... break;
case 13: ... break;
case 16: ... break;
case 17: ... break;
default:
Error_Key++;
break;
}
```
- [b] 将模式切换放到 EEPROM 保存判断之前，"先切换，再判断是否需要保存"：
```c
case 12:
Error_Key = 0;
if(++Seg_Disp_Mode == 3) Seg_Disp_Mode = 0; // 先切换模式
if(Seg_Disp_Mode == 2) // 再判断：切换后是否进入计数界面
{
V_Pra_10 = V_Pra * 10;
EEPROM_Write(&V_Pra_10, 0, 1); // 此时保存的是刚修改完的新参数
}
```
> **附注：** 本例中学员反复检查 LED 模块和底层代码均未找到问题，是因为根因不在 LED 本身——LED 模块忠实地执行了 `ucLed[2] = (Error_Key >= 3)` 的逻辑，问题出在上游按键处理中 `Error_Key` 被错误累加。排查"显示/输出异常"时，除了检查显示模块本身，还要沿数据流向上追溯到数据源头。

## 坑 53：长按计时器 else 分支持续清零导致无法触发长按

> 来源：[069_蓝桥杯单片机第二十四周周报（二）.md](./单片机答疑数据库/069_蓝桥杯单片机第二十四周周报（二）.md)
> 周期：蓝桥杯单片机第二十四周周报（二）

### 整理记录

- **问题描述：** 学员自己实现的长短按功能，只能触发短按，长按始终无响应
- **问题背景：** 按键长短按检测（定时器中断中计时）
- **问题分析：** 学员原代码在定时器中断中写成：
```c
if(Long_Press_Flag) // 触发长按
{
if(++Time_15M >= 1500)
Time_15M = 1501; // 防止溢出
else
Time_15M = 0; // ← 问题在这里
}
```
`else` 分支的含义是：只要 `Time_15M` 还没累加到 1500，就立刻清零。这意味着 `Time_15M` 每次自增到 1 后下一轮就被清回 0，永远无法达到 1500，长按条件永远不成立
- **解决方案：** 去掉 `if-else` 中的清零分支，改为仅在按键未按下时才清零：
```c
if(Long_Press_Flag) // 按键处于按下状态
{
if(Time_15M < 1500)
{
Time_15M++; // 持续累加直到 1500，不清零
}
}
else // 按键未按下或已弹起
{
Time_15M = 0; // 只有松开时才清零
}
```

> 来源：[069_蓝桥杯单片机第二十四周周报（二）.md](./单片机答疑数据库/069_蓝桥杯单片机第二十四周周报（二）.md)

- **问题背景：** 第十七届模拟赛第三次题目（四键同时按下开始比赛、检测谁先松手）
- **问题分析：** 学员的按键底层采用编号累加方式识别按键，无法正确处理多键同时按下的场景。该题要求四键同时按下开始比赛并检测谁先松手，必须用位掩码方式逐位判断每个按键的状态。正确做法是通过位运算精确判断哪个键松开：
```c
case 1: key_up_can = 4; // bit0 松开 → S4

case 4: key_up_can = 6; // bit2 松开 → S6
```

- **解决方案：** 将按键底层改为位掩码方式，每个按键对应一个独立的 bit 位，通过位运算分别检测各键的按下/松开状态，确保多键组合场景下能准确区分每个按键

## 坑 59：第十四届省赛按键无反应三连 — 滴答溢出/数码管残留/按下抬起写反

> 来源：[070_蓝桥杯单片机第二十五周周报.md](./单片机答疑数据库/070_蓝桥杯单片机第二十五周周报.md)
> 周期：蓝桥杯单片机第二十五周周报

### 整理记录

- **问题描述：** 学员刚开始做第十四届省赛题目，按下按键 4 切换界面，但数码管毫无变化；后续补充按键任务执行后 Led、Seg 等任务也完全停摆
- **问题背景：** 第十四届省赛题目，多界面按键切换+调度器任务调度框架
- **问题分析：** 学员代码同时中了三枪：
- [a] **滴答时钟溢出导致调度器"死锁"（致命）：** `uwTick` 定义为 `unsigned char`（8 位，最大值 255），每 1ms 中断加 1，到 255 后会清零重新计数。但调度器结构体 `task_t` 中 `rate_ms` 和 `last_ms` 都是 `unsigned long int`（32 位）。开机经过 255ms 后，比如数码管任务（100ms 周期）`last_ms` 变成 200，下一次需要运行的时间是 200+100=300。此时判断条件 `if(now_time >= (Scheduler[i].rate_ms + Scheduler[i].last_ms))` 变成 `if(now_time >= 300)`。因为 `now_time` 最大只能跟随 `uwTick` 达到 255（溢出回零后在 0~255 徘徊），这个条件永远不会成立。调度器彻底瘫痪，按键和数码管任务都不会再执行了
- [b] **显示问题：数码管没有清除旧状态：** 假设按键生效了，`Seg_Show_Mode` 从 0 变成了 1。`Seg_Proc` 中模式 0 只赋值了 `Seg_Buf[3]`，模式 1 只赋值了 `Seg_Buf[4]`，切换模式时没有把另一个位置清空（恢复成 10 熄灭状态）。结果就是按键按下后第 3 位和第 4 位都亮了，且一直亮着，视觉上看起来就像"数码管没变化"
- [c] **逻辑隐患：按下与抬起判断反了：** 蓝桥杯开发板按键通常是"按下为 0，松开为 1"。学员的边沿检测代码 `Key_Down = Key_Val & (Key_Val ^ Key_Old)` 检测的是 0→1（释放），`Key_Up = ~Key_Val & (Key_Val ^ Key_Old)` 检测的是 1→0（按下），实际体验变成松手才切换
- **解决方案：**
- [a] 将 `uwTick` 改为 `unsigned long int`，或调度器判断改用 `(uwTick - Scheduler[i].last_ms) >= Scheduler[i].rate_ms`（差值判断，天然抗溢出），推荐后者（更省 RAM）
- [b] `Seg_Proc` 中每个 `case` 分支先清零所有 `Seg_Buf` 位，再赋值对应位置
- [c] 互换 Key_Down 和 Key_Up 的检测公式，或在 `Key_Read()` 内部对键值取反
- **经验教训：** 调度器时间基准类型必须与比较运算兼容，8 位变量参与 32 位运算 + 溢出回零时极易触发"永不满足"的死锁；显示切换类逻辑务必保证"切走就清"，不留残留

## 坑 60：长按计时器 else 分支持续清零导致无法触发长按

> 来源：[071_蓝桥杯单片机第二十五周周报（二）.md](./单片机答疑数据库/071_蓝桥杯单片机第二十五周周报（二）.md)
> 周期：蓝桥杯单片机第二十五周周报（二）

### 整理记录

- **问题描述：** 在 `while(1)` 主循环中调用按键处理函数，按下指定按键（`Key_Down == 4`）后没有任何作用，数码管模式（`Seg_Mode`）无法正常切换。
- **问题背景：** 单片机（STC15F2K60S2）的非阻塞式按键扫描框架开发，涉及三行代码按键状态机（通过异或运算提取下降沿/上升沿）与定时器减速消抖。
- **问题分析：** 该问题主要是由函数内部执行时序错误导致，状态稍纵即逝被拦截：
- [a] **核心漏洞（执行顺序倒置）：** 在 `Key_Proc()` 中，按键的读取与历史值更新操作（刷新 `Key_Old`）放在了减速拦截语句（`if(Key_Slow_Down<100) return;`）的上方。
- [b] **下降沿状态丢失：** 主循环执行速度极快（微秒级）。按键按下的瞬间，系统计算出 `Key_Down = 4` 并立刻更新了 `Key_Old`。但此时减速变量未满 100，程序直接 `return`，动作未能执行。
- [c] **判断条件扑空：** 当经历了几千次循环，`Key_Slow_Down` 终于计满 100 并放行时，由于按键一直被按着，`Key_Val` 与 `Key_Old` 相等，此时算出的 `Key_Down` 早就变成了 0。导致底部的 `if(Key_Down==4)` 永远无法成立。
- [d] **参数设置不合理（体验问题）：** 消抖阈值设定为 100ms 偏大，即使逻辑正确，过长的扫描间隔也会导致按键响应严重迟钝，出现漏按的情况。
- **解决方案：**
- [a] **前置减速拦截：** 将 `if(Key_Slow_Down < 10) return; Key_Slow_Down = 0;` 移至 `Key_Proc()` 函数的**最开头**（第一行）。让整个按键的读取、状态计算全部受控于该定时器。
- [b] **缩短消抖周期：** 将减速阈值由 100 优化修改为 10 或 20（即每 10ms~20ms 扫描一次），既能利用软件定时器完美滤除机械抖动，又能保证高频响应灵敏度。

- [b] **重构为稳态电平判断（方式二/更优）：** 针对“按住生效，松开恢复”的功能要求，直接判断按键的实时稳态。将条件改为判断 `if(Key_Old == 17)`，按键只要保持按下状态就置 1，否则走 `else` 清 0，逻辑更契合实际需求。
