## 坑 1：频率计数时频繁关开中断会把 1 秒闸门时间拉歪

> 未知题目 · 发现者：左岚

### 现象

用 `NE555` 做频率计数时，测出来的结果和 `4T` 差得比较远，而且现象不稳定: 有时候正常，有时候异常，看起来像玄学漂移。

### 原因

这类频率测量本质上依赖“精确的 1 秒计数窗口”。如果在计数过程中频繁关中断、开中断，就会拖慢定时器轮询或 `1s` 闸门翻转的时刻，最终不是计数器坏了，而是“这一秒”本身被拉长或压短了。

```c
// 错误写法：在计数窗口中途频繁关开中断
EA = 0;
/* 其他处理逻辑 */
EA = 1;
```

### 解决方法

计数窗口内尽量保持中断连续运行，不要为了别的逻辑去打断它。真要做临界区保护，也只在结果快照或状态切换时短暂处理，不要碰闸门计时主链路。

```c

// 每1000ms（1秒）执行一次
if (++Time_1s == 1000)
{
    Time_1s = 0; // 1秒计时器清零
    // 读取定时器0的计数值，即为过去1秒内的脉冲数（频率Hz）
    Freq = (TH0 << 8) | TL0;
    // 清零定时器0，准备下一个周期的计数
    TH0 = TL0 = 0;
}
```

### 举一反三

- 凡是“固定时间窗计数”的题，精度先看时间窗准不准，再看计数值对不对
- 关中断不只会丢事件，还会悄悄改写时间基准
- 遇到“有时正常有时异常”的频率题，先查有没有在测量窗口里做过临时关中断

* **问题分析**：

* **解决方案**：

  + 将`xms`变量类型扩大，延时时间恢复正确

```
/*
学员原先是 void Delay(unsigned char xms)
这样在 xms在255以上时会自动转为unsigned char 比如传入500会转成500-255=245的延时 导致对不上
*/
void Delay(unsigned int xms)
{
	unsigned char data i, j;
	while(xms--)
    {
        i = 2;
        j = 239;
        do
        {
            while (--j);
        } while (--i);
    }
}
```

* **问题分析**：

  + 首先可以根据按键选择模式 比如：按键一按下选择模式一，以此类推
  + 流水可以开个`time_500ms`变量在定时器中断不断加加直到500执行函数
  + 执行函数可以根据mode选择左移还是右移

  + 两个灯流水这种类似的交给学员自己写，这边提供了一个流水方式

```
void key_proc()
{
	if(key_slow) return;
	key_slow=1;
	key_val=Key_Read();
	key_down=key_val&(key_val^key_old);
	key_up=~key_val&(key_val^key_old);
	key_old=key_val;
	/* 开始 */
    //按下1切换成模式1以此类推
	if(key_down)
		mode=key_down;
}

//定时器代替delay
void Timer0Server() interrupt 1
{
	TL0 = 0x18;
	TH0 = 0xFC;
	if(++key_slow ==10 ) key_slow=0;
	/* 流水灯相关 */
	if(++time_500ms==500) 
	{
		time_500ms=0;
        if(mode==0)return;//初始什么都不点亮也不流水
        if(P1=0xFF)
        	P1=0xFE;
        else 
        {
            switch(mode)
            {
                case 1://右移动
                    P1=_cror_(P1,1);
                break;

                case 2://左移动
                    P1=_crol_(P1,1);
                break;
            }
        }
    }
}
```

## 坑 13：定时器溢出导致系统卡死

> 来源：[019_蓝桥杯单片机第七周周报（二）.md](./单片机答疑数据库/019_蓝桥杯单片机第七周周报（二）.md)
> 周期：2024年10月27日 — 2024年11月02日

### 整理记录

* **问题描述**：如果time减到0了，是不是就是全亮了。
* **问题背景**：无具体课程信息。
* **问题分析**：

  + 无符号整数减到0后继续减会溢出
  + 溢出后变成最大值65535，导致延时时间过长
  + 系统会一直卡在延时状态
* **解决方案**：

  + 不会全亮，减到0后再继续减会溢出，变成65535，就会一直延时卡住。需要在代码中添加溢出判断，避免无符号整数下溢。

```
void Timer0_ISR() interrupt  // 缺少中断号

    // 中断处理代码
}

void Timer0_ISR() interrupt 1  // 添加中断号1(Timer0)
{
    // 中断处理代码
}
```

## 坑 29：频率测量 Freq 始终为 0（V4T项目，5处修复）

> 来源：[061_蓝桥杯单片机第二十一周周报（三）.md](./单片机答疑数据库/061_蓝桥杯单片机第二十一周周报（三）.md)

- [b] `unsigned char temp` 改为 `unsigned char temp = 0`

- [e] 改为 `Seg_Pos = (Seg_Pos + 1) % 8`

## 坑 30：按键扫描干扰 T0 计数器导致频率偏大 100

> 来源：[062_蓝桥杯单片机第二十二周周报.md](./单片机答疑数据库/062_蓝桥杯单片机第二十二周周报.md)
> 周期：蓝桥杯单片机第二十二周周报

### 整理记录

- **问题描述：** 频率界面每次显示都比真实值大 100
- **问题背景：** 第十五届省赛题目
- **问题分析：** 按键扫描函数末尾执行 `P3=P3|0xEF;`，`0xEF = 1110 1111`，OR 操作未恢复 P3.4（bit4 保持原值）。而 P3.4 正是 T0 计数器的外部输入引脚，每次按键扫描过程中对 P3 口的操作会在 P3.4 上产生下降沿，被 T0 计数器误计为一次脉冲。定时器 10ms 中断扫描按键，每秒扫描 100 次，因此频率测量值恰好偏大 100
- **解决方案：** 将 `P3=P3|0xEF;` 改为 `P3=P3|0xFF;`，确保 P3.4 在按键扫描结束后恢复为高电平，避免干扰 T0 计数器输入

## 坑 31：定时器中断周期异常导致倒计时速度不对

> 来源：[062_蓝桥杯单片机第二十二周周报.md](./单片机答疑数据库/062_蓝桥杯单片机第二十二周周报.md)
> 周期：蓝桥杯单片机第二十二周周报

### 整理记录

- **问题描述：** Proteus 仿真中数码管减速已设为 500ms，但倒计时跳变非常快；把减速值调到 1000 或 2000 后基本不跳了
- **问题背景：** 日常课程练习（仿真环境）
- **问题分析：** `Timer0Init()` 中漏敲了 `TMOD |= 0x01;` 这一行，`TMOD` 未被配置为模式 1（16 位定时器），Timer0 工作在默认的模式 0（13 位定时器），最大计数值只有 8192 而非 65536，初始值 `TH0=0xFC, TL0=0x18` 在 13 位模式下实际定时远小于预期的 1ms，导致中断频率大幅偏快，所有基于中断计数的时间都相应加快
- **解决方案：** 补上 `TMOD |= 0x01;`，将 Timer0 配置为模式 1（16 位定时器模式），中断周期恢复为正确的 1ms

## 坑 32：中断时序冲突导致通信失败

> 来源：[063_蓝桥杯单片机第二十二周周报（二）.md](./单片机答疑数据库/063_蓝桥杯单片机第二十二周周报（二）.md)
> 周期：蓝桥杯单片机第二十二周周报（二）

### 整理记录

- **问题描述：** 温度读取失效 / DS1302数据错误
- **问题背景：** 第13届省赛（调度器顺序问题）、第14届省赛（ISR浮点运算）
- **问题分析：**
- DS18B20 OneWire协议的位读取只有4μs采样窗口，1ms定时器中断（ISR执行50-100μs）落入该窗口时采样报废
- DS1302通信通过P2.3(SDA)进行，ISR中数码管扫描的 `P2 = temp | 0xe0` 会篡改P2.3状态
- ISR中执行浮点运算（`0.0444*Freq+1.1111`）耗时800-1300μs，超过1ms定时器周期，主循环几乎得不到CPU时间
```c
// ISR中浮点运算——致命！
shidu_j[chufa_num] = (unsigned int)((0.0444 * Freq + 1.1111) * 10 + 0.5);
// 8051无FPU，此运算耗时 > 1ms定时器周期
```
- **解决方案：**
- OneWire/DS1302通信前关中断：`EA = 0;` ... 通信 ... `EA = 1;`
- 将浮点运算移出ISR，改用整数运算：`44UL * Freq + 1111) / 100` 或 `(Freq - 200) * 4 / 9 + 100`
> **规律总结：** 8051单片机ISR中绝对不要做浮点运算；OneWire、DS1302等时序敏感通信必须加关中断保护。

> 来源：[063_蓝桥杯单片机第二十二周周报（二）.md](./单片机答疑数据库/063_蓝桥杯单片机第二十二周周报（二）.md)

## 坑 34：频率测量误读 TH1/TL1 导致 Timer1 被清零，按键迟钝、数码管流水

> 来源：[066_蓝桥杯单片机第二十三周周报.md](./单片机答疑数据库/066_蓝桥杯单片机第二十三周周报.md)
> 周期：蓝桥杯单片机第二十三周周报

### 整理记录

- **问题描述：** 按键响应极慢，数码管出现流水灯效果，界面切换也很慢
- **问题背景：** 频率测量功能，Timer0 做计数器、Timer1 做定时中断
- **问题分析：** 学员在 Timer1 的 ISR 中读频率时写成了 `Freq = (TH1<<8) | TL1`，随后又 `TH1 = TL1 = 0` 清零。但频率计数器用的是 Timer0（TH0/TL0），Timer1 是定时中断的定时器。读 TH1/TL1 读到的是 Timer1 自己的重装值（不是频率），清零 TH1/TL1 后 Timer1 的计数起点被改为 0，需要从 0 计到 65535 才能溢出触发下一次中断，周期从原来的 1ms 变成约 65ms。这导致：
- 数码管扫描频率从 1000Hz 降到约 15Hz，肉眼可见逐位扫描（流水灯效果）
- 按键扫描和调度器全部变慢，响应极其迟钝
- **解决方案：** 将 `TH1`/`TL1` 改为 `TH0`/`TL0`：
```c
// 错误
Freq = (TH1 << 8) | TL1;
TH1 = TL1 = 0;
// 正确
Freq = (TH0 << 8) | TL0;
TH0 = TL0 = 0;
```

- [b] **设置参数与比赛参数未分离：** 设置界面修改的参数和比赛界面使用的参数共用同一个变量，比赛倒计时直接递减设置值，导致下一轮比赛初值异常，且设置界面显示被改坏

## 坑 36：按键底层多写 P3 &= 0xef 导致 T0 计数输入被拉低，频率恒为 0

> 来源：[066_蓝桥杯单片机第二十三周周报.md](./单片机答疑数据库/066_蓝桥杯单片机第二十三周周报.md)
> 周期：蓝桥杯单片机第二十三周周报

### 整理记录

- **问题描述：** 数码管肉眼观察显示正常，但 4T 测评全部错误：频率界面始终显示 `F 0`，周期界面显示 `N4967295`（溢出值）
- **问题背景：** 频率/周期测量题目，Timer0 做外部计数器
- **问题分析：** 按键扫描底层函数末尾多了一行 `P3 &= 0xef;`。`0xef = 1110 1111b`，该操作将 **P3.4 强制拉低**。P3.4 正是 T0（Timer0 外部计数输入）引脚，每次按键扫描（每 10ms）都会将 T0 输入拉低一次，外部频率信号无法正常进入计数器，Timer0 计数值始终为 0：
- 频率 = 0 → 频率界面显示 `F 0`
- 周期 = 1000000 / 0 → 整数除零溢出，显示 `N4967295`
学员肉眼测试时，因为没有外接信号源或不关注频率值，误认为"显示正常"，直到 4T 自动测评注入频率信号后才暴露问题
- **解决方案：** 删除按键底层中多余的 `P3 &= 0xef;`：
```c
// 修改前（P3.4 被拉低，破坏 T0 计数输入）
P3 &= 0xef; // ← 删除此行
return temp;
}
// 修改后
return temp;
}
```
> **附注：** P3 口在 STC15 上承担多种复用功能（T0、T1、INT0、INT1、RXD、TXD 等），按键扫描操作 P3 时必须只动按键相关的引脚位，不能用掩码误伤其他功能引脚。写底层时建议逐位操作或仔细核对掩码值，避免影响同端口的其他外设。

## 坑 37：计时器 else 分支无条件清零（本周最高频，4次）

> 来源：[067_蓝桥杯单片机第二十三周周报（二）.md](./单片机答疑数据库/067_蓝桥杯单片机第二十三周周报（二）.md)
> 周期：蓝桥杯单片机第二十三周周报（二）

### 整理记录

- **问题描述：** LED超时判断永远不成立 / 长按检测永远不触发 / L4闪烁条件永远不满足
- **问题背景：** 第11届省赛（2次）、第12届省赛、第15届省赛
- **问题分析：** 在定时器中断中，计时变量的累加逻辑被 `else` 分支在每次中断中清零，导致计数器永远无法达到目标值：
```c
// 典型错误模式（出现4次，写法几乎一模一样）
if(++Timer_5s >= 5000)
Timer_5s = 5001; // 到达目标，限幅
else
Timer_5s = 0; // ← 致命！每次中断都清零
// 执行追踪：
// 中断1: Timer_5s: 0→1, 1>=5000? 否 → Timer_5s=0
// 中断2: Timer_5s: 0→1, 1>=5000? 否 → Timer_5s=0
// ... 永远到不了5000
```
另一种变体——长按检测中的相同模式：
```c
// 第12届：Time_Tick 用于1秒长按检测
if(Long_Press_Detection)
{
if(++Time_Tick >= 1000)
Time_Tick = 1001;
else
Time_Tick = 0; // ← 每次中断都归零，永远到不了1000
}
```
- **解决方案：** 删除 `else Timer = 0;`，将清零操作移到条件不满足（如Low_Flag为假）时执行：
```c
if(Low_Flag)
{
if(++Timer_5s >= 5000)
Timer_5s = 5000; // 限幅防溢出即可
}
else
Timer_5s = 0; // 条件不成立时才清零
```
> **规律总结：** 这是本周出现频率最高的Bug模式（4次），且几乎所有学员犯的错误写法一模一样。根本原因是对 `if-else` 语义的误解：`else` 分支在条件不成立时**每次都执行**，而非"只在第一次不成立时执行"。建议将此模式作为赛前代码自检的第一项。

## 坑 38：计时变量被多个功能共用

> 来源：[067_蓝桥杯单片机第二十三周周报（二）.md](./单片机答疑数据库/067_蓝桥杯单片机第二十三周周报（二）.md)
> 周期：蓝桥杯单片机第二十三周周报（二）

### 整理记录

- **问题描述：** 频率测量值始终为0，数码管频率显示为0
- **问题背景：** 第13届国赛
- **问题分析：** `timer_1000ms` 同时用于两个功能：Timer1 ISR 中的1秒频率采样门控 和 主循环中的按键长按检测。两个功能互相争抢同一变量：
```c
// Timer1 ISR 中：频率测量需要 timer_1000ms 累积到 1000
if(++timer_1000ms == 1000)
{
timer_1000ms = 0;
freq = (TH0 << 8) | TL0; // 读取频率
TH0 = TL0 = 0;
}
// key_proc() 中（每10ms调用一次）：
if(timer_1000ms >= 1000) // 长按判断
{ /* 长按逻辑 */ }
else
{ timer_1000ms = 0; } // ← 每10ms就清零！
```
key_proc 每10ms调用一次，此时 timer_1000ms 最多累积到10就被清零，ISR中的 `==1000` 永远不成立，频率永远读不出。
- **解决方案：** 为频率测量和长按检测使用**独立的计时变量**：
```c
unsigned int timer_freq = 0; // 频率专用
unsigned int timer_long_press = 0; // 长按专用
```
> **规律总结：** 不同功能模块的计时变量必须独立，绝不能共用。即使两个功能恰好都需要"1秒定时"，也应该分开，否则任何一方的清零操作都会破坏另一方。

## 坑 40：第十二届省赛（第二次）周期计算截断误差导致周期页全部失分

> 来源：[069_蓝桥杯单片机第二十四周周报（二）.md](./单片机答疑数据库/069_蓝桥杯单片机第二十四周周报（二）.md)
> 周期：蓝桥杯单片机第二十四周周报（二）

### 整理记录

- **问题描述：** 4T 测评 64 分，频率页 4 项全部正确，但周期页 4 项全部错误，周期值均偏小（如标准 3330 显示 3289、标准 2000 显示 1972）
- **问题背景：** 第十二届省赛第二次题目（频率/周期测量）
- **问题分析：** 程序将周期值写成 `period = 1000000 / Freq;`，频率测量本身存在约 1% 的小误差，频率页尚可通过，但周期页对频率做倒数运算后误差被进一步放大，导致连续失分。此外还存在两个隐患：
- [a] **长按 S7 计时逻辑写错：** 与问题（2）同类错误，计时变量未达阈值前被反复清零，长按无法触发
- [b] **ADC 换算公式缺少比例系数：** `Volt_100x = Ad_Read(0x01) * 100` 少了 `/51`，电压值偏大
- **解决方案：**
- 周期计算加入四舍五入：`Period = (1000000UL + Freq / 2) / Freq;`
- 修正长按计时逻辑，按下时持续累加、松手时判断并清零
- ADC 换算补上比例系数：`Volt_100x = Ad_Read(0x01) * 100 / 51;`

## 坑 41：串口 printf 浮点发送偶发小数点错位——定时器中断打断浮点格式化

> 来源：[069_蓝桥杯单片机第二十四周周报（二）.md](./单片机答疑数据库/069_蓝桥杯单片机第二十四周周报（二）.md)
> 周期：蓝桥杯单片机第二十四周周报（二）

### 整理记录

- **问题描述：** 串口发送浮点数据时，正确值如 `10.00` 偶尔会发送成 `100.0`，小数点位置错乱
- **问题背景：** 串口通信功能，使用 `printf("%.1f"/"%.2f")` 格式化发送温度/浓度值
- **问题分析：** 8051 的 `printf` 浮点转字符串（`%f`）非常慢（几毫秒级），执行过程中 Timer1 中断会打断它。中断服务函数中的 `Seg_Disp` 数码管扫描操作使用的寄存器与 `printf` 内部使用的寄存器冲突，中断返回后 `printf` 拿到的是被改过的寄存器值，导致小数点位置错乱
- **解决方案：** 在执行 `printf` 前后临时关闭/恢复 Timer1 中断，防止浮点格式化过程被打断：
```c
ET1 = 0; // 关闭 Timer1 中断
printf("C:%.1f\r\n", Temperature);
ET1 = 1; // 恢复 Timer1 中断
```

## 坑 42：第十四届 4T 测评未触发即产生历史数据导致计数多 1

> 来源：[070_蓝桥杯单片机第二十五周周报.md](./单片机答疑数据库/070_蓝桥杯单片机第二十五周周报.md)
> 周期：蓝桥杯单片机第二十五周周报

### 整理记录

- **问题描述：** 第十四届省赛 4T 测评中，在第一次有效光敏触发之前，温度/湿度回显界面就已经显示了历史数据，时间回显界面显示 `F01` 而非 `F00`
- **问题背景：** 第十四届省赛题目，暗光触发采集 → 温湿度采样 → 历史记录入库，涉及多步状态流转
- **问题分析：** `Get_Temperature()` 被定时任务周期调用，但在执行采集和入库前没有判断 `Trigger_Flag` 是否已置 1。上电后定时器先更新了 `Freq`，`Get_Temperature()` 再次运行时发现湿度已有效，在未触发光敏下降沿的情况下就执行了 `Trigger_Count++` 操作，导致历史记录被提前写入一次。另外 `AD_DA()` 中每次检测到下降沿就重新清零采集标志，在 3 秒采集显示期间存在重复触发隐患

- [b] `AD_DA()` 中仅在未触发时才允许响应下降沿触发：`if(Trigger_Flag==0 && temp_ad<80 && Voltage_Light_Old>80)` 才能进入触发
- [c] 触发显示的 3 秒期间禁止再次触发，直到采集完成

## 坑 43：第十五届省赛一频率校准与参数状态分散

> 来源：[070_蓝桥杯单片机第二十五周周报.md](./单片机答疑数据库/070_蓝桥杯单片机第二十五周周报.md)
> 周期：蓝桥杯单片机第二十五周周报

### 整理记录

- **问题描述：** 校准 -100 时 `F1500` 显示成 `F1615`（应为 `F1400`）、DAC 电压从 1.8V/2.3V 偏至 2.0V/2.6V、历史最大值 `HF250` 变成 `HF152`、历史回显时间不对（如 `HA133445` 变成别的时间）
- **问题背景：** 第十五届省赛一题目，涉及 Timer0 频率采集、校准参数加减、DAC 输出、历史最大值记录、RTC 回显等
- **问题分析：** 学员代码存在 4 个核心缺陷：
- [a] **负校准值计算写反：** `lookup_para` 本身为负数，代码写的是 `freq_value_cal=freq_value_real-lookup_para`，当 `lookup_para=-100` 时实际变成 `freq_value_real+100`，导致校准值往反方向偏移。正负校准应统一为 `freq_value_real + lookup_para`
- [b] **两套参数分离不生效：** 参数界面改的是 `limit_ctrl / lookup_ctrl`（显示值），真正参与计算的是 `limit_para / lookup_para`（运行值），需要退出参数界面才能提交更新。4T 测评节奏快，刚改完参数就立刻查看 DAC 和历史值，此时旧参数仍在运行，导致 `HF250→HF152`、`DAC 2.3→2.6` 等偏差
- [c] **最大值更新放在显示函数中：** `freq_max` 和 `ucRtc_maxf` 的更新写在 `seg_proc()` 开头，依赖"什么时候刚好刷新显示"来决定何时保存。参数刚改完、频率刚变化或界面刚切换时，保存时机容易漂移
- [d] **main() 中强刷 RTC 覆盖测评时间：** `set_time(ucRtc)` 每次上电都把时钟设为 `23:59:50`，若测评环境已预置了测试时间，则会被直接覆盖，导致所有时间相关数据出错
- **解决方案：**
- [a] 校准公式统一为 `freq_value_cal = freq_value_real + lookup_para`，不再按正负校准拆成两套判断逻辑
- [b] 去掉 `limit_ctrl / lookup_ctrl` 两套暂存参数，参数界面直接改正式参数 `limit_para / lookup_para`，修改后立即生效
- [c] 新增独立的 `freq_proc()` 函数，专门负责合成最终频率、判断错误标志、超限判断、保存最大频率和最大时间。`seg_proc()` 只负责显示，不再修改业务变量
- [d] `main()` 中移除 `set_time(ucRtc)` 调用，不再上电强刷 RTC；`Timer1_Isr()` 中只保留数码管扫描和每秒锁存实测频率，不再做校准业务判断

## 坑 44：第十五届省赛一频率题LED管理混乱与五位显示缺失

> 来源：[070_蓝桥杯单片机第二十五周周报.md](./单片机答疑数据库/070_蓝桥杯单片机第二十五周周报.md)
> 周期：蓝桥杯单片机第二十五周周报

### 整理记录

- **问题描述：** 频率主界面 L1 闪烁不稳定、超限时 L2 没按 0.2s 闪烁、负频率错误态 L2 没常亮、五位频率 `15280` 被显示成 `A692`、峰值回显 `HF16680` 显示成 `HF10033`
- **问题背景：** 第十五届省赛一频率采集题，涉及频率计数、校准加减、超限判断、峰值记录、LED 闪烁指示
- **问题分析：**
- [a] **L2 被多套逻辑控制互相覆盖：** `Led_Proc()` 里同时判断超限闪烁和错误常亮，`Timer1_Isr()` 里又直接翻转 `ucLed[1]`，导致超限闪烁和错误常亮互相抢状态，最终态不是"一个地方算完"
- [b] **L1 离开主界面未复位：** L1 闪烁在中断里做翻转，离开频率主界面时没有清零计数器和闪烁标志，导致其他页面残留闪烁状态，切回时起始相位不稳定
- [c] **五位频率按四位逻辑拆分：** 显示只写了 `Seg_pingshu/1000%10` 取千位，当频率 >=10000 时千位得到 15，字模下标 15 对应 `A`，所以 `15280` 显示成 `A692`
- [d] **峰值记录链路不统一：** `Seg_pingshu` 定义为无符号数，错误态和峰值判断容易被一起带偏；峰值、错误判断、DAC 输出没有基于同一个最终频率值
- [e] **S8/S9 参数修改未限制界面：** 参数加减没有严格限制在参数界面，其它页面也可能被误改
- **解决方案：**
- [a] `Seg_pingshu` 改为 `long int` 带符号类型，统一作为"最终频率值"。负值时进入错误态，正值时才更新峰值
- [b] 新增 `ping_l()` 频率统一计算入口：`Seg_pingshu = Freq + Seg_jiaozhong;` → 判断 `diping` 错误标志 → 更新 `Seg_dapin` 峰值和时间
- [c] 重写 `Led_Proc()`，将 L1/L2 的"是否开始闪烁"和"当前亮灭状态"分开管理：用 `Lx_led_start` 控制闪烁启用，用 `Lx_led_200_flow` 控制亮灭。L2 优先级：错误态常亮 > 超限闪烁 > 熄灭
- [d] 频率主界面改为 5 位显示：`Seg_Buf[3] = (Seg_pingshu>=10000)?万位:10; Seg_Buf[4..7] = 千/百/十/个位`
- [e] S8/S9 参数加减加界面判定保护：`if(Seg_xianshi==1 && Key_Down==8)` 才允许执行

## 坑 45：第十五届省赛频率题频率超限LED闪烁状态未退出与比较口径不统一

> 来源：[070_蓝桥杯单片机第二十五周周报.md](./单片机答疑数据库/070_蓝桥杯单片机第二十五周周报.md)
> 周期：蓝桥杯单片机第二十五周周报

### 整理记录

- **问题描述：** 当超限频率 >= 当前频率时，LED0 没有恢复为稳定熄灭状态，而是停在常亮或常灭不变，看起来像"卡住不闪烁"
- **问题背景：** 第十五届省赛频率采集题，LED0 在"当前输出频率 > 超限频率"时按 0.2s 周期闪烁
- **问题分析：** 核心不是"0.2s 闪烁写错了"，而是状态机没有收口：
- [a] **超限判断使用了原始频率而非调整后频率：** 显示看的是 `Freq + Freq_cail_crtol`（校准后的最终频率），但 LED 判断写的是 `Freq_limter < Freq`（原始测量值）。校准值修改后显示变了但 LED 行为不一致
- [b] **停止闪烁时缺少复位动作：** 不超限时只写了 `Timer_flag = 0`，没有清零 `Time_200ms`、没有清零 `led_flag`、没有明确 `ucLed[0] = 0`。退出闪烁时 `led_flag` 保留退出前的最后一次值，导致 LED 可能常亮或常灭
- [c] **等于情况未处理：** 分别写了 `if(Freq_limter < Freq)` 和 `if(Freq_limter > Freq)`，当 `Freq_limter == Freq` 时两边都不进入，LED 保持上一拍结果
- [d] **调整后频率使用无符号变量存在下溢风险：** `Freq_disp = Freq + Freq_cail_crtol` 中校正值可为负，无符号变量下溢成大数
- [e] **最大频率记录用了原始值：** 记录峰值时用 `Freq` 而非调整后的最终频率，口径不一致
- **解决方案：**
- [a] 统一计算最终频率作为"唯一真相"：新增 `Freq_Final = (long int)Freq + Freq_cail_crtol`，显示、比较、记录最大值全部使用该变量
- [b] LED 闪烁退出时完整复位：不超限时同步清零 `Timer_flag`、`Time_200ms`、`led_flag`，并设置 `ucLed[0] = 0`
```c
if(Freq_Final > Freq_limter)
Timer_flag = 1;
else {
Timer_flag = 0;
Time_200ms = 0;
led_flag = 0;
}
ucLed[0] = led_flag;
```
- [c] 用 `>` 判断超限，用 `else` 兜底覆盖等于和小于两种情况，不再分别写 `<` 和 `>` 导致悬空
- [d] 最终频率改为 `long int` 带符号类型，避免负校准值导致下溢
- [e] 新增 `Freq_Proc()` 函数统一计算最终频率、更新最大值及时间，作为独立任务周期执行（20ms），确保 LED/显示始终使用一致的数据
- **经验教训：** 状态进入和退出必须成对设计，退出时要把相关标志位和计时变量一起复位；显示、判断、记录必须统一使用同一个"最终有效量"，不要让不同逻辑用不同口径的数据

## 坑 46：NE555 频率测量架构认知

> 来源：[071_蓝桥杯单片机第二十五周周报（二）.md](./单片机答疑数据库/071_蓝桥杯单片机第二十五周周报（二）.md)
> 周期：蓝桥杯单片机第二十五周周报（二）

### 整理记录

- **问题描述：** 学员疑惑 NE555 频率测量的代码是应该直接写在系统调度器（定时器 1）的中断里，还是需要单独开一个中断来处理。
**问题背景：** 蓝桥杯单片机基础外设（NE555 频率测量）考点。该功能的标准实现需要统筹使用两个硬件资源：一个用作“计数器”统计外部脉冲个数，另一个用作“定时器”提供 1 秒的时间闸门。
**问题分析：** 这是一个典型的对底层硬件工作模式分工不清晰导致的架构疑惑：
- [a] **外设职责混淆：** 学员看到模板中计算频率的代码 `Freq=(TH0<<8)|TL0;` 写在了 `Timer1_Isr`（调度器中断）里，就误以为 NE555 完全归属调度器管辖。实际上，闸门卡点在定时器 1，但真实的脉冲收集是由定时器 0 完成的。
- [b] **“开中断”与“初始化”概念混淆：** 作为外部脉冲计数器，定时器 0 的核心在于“独立初始化”（配置为计数模式，引脚设为 P3.4），它在后台依靠纯硬件默默累加脉冲，通常不需要为其单独写一个中断服务函数（除非需要处理超量程溢出）。
**解决方案：**
- [a] **剥离初始化逻辑：** 明确告知学员，定时器 0 必须单独写一个初始化函数（配置为计数器模式，启动 `TR0`，不开启中断使能）。
- [b] **调度器负责读取：** 真正的频率计算动作（即读取 `TH0` 和 `TL0` 并清零准备下一次统计），则顺理成章地放在定时器 1（调度器）中，以 1 秒为周期去定时执行。两相配合，各司其职。
