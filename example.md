# 踩坑文档格式范例

本文件展示了本系列文档的标准写法，供新建文档时参照。

---

## 目录结构

每个硬件/功能模块对应一个文件，所有该模块的踩坑都写在同一文件里：

```
蓝桥杯 单片机 踩坑/
├── README.md
├── CLAUDE.md
├── example.md
├── led.md        # LED 亮灭、闪烁
├── seg.md        # 数码管显示
├── key.md        # 按键检测
├── eeprom.md     # EEPROM 读写
├── da.md         # DA 输出
├── ad.md         # AD 采集
├── ultrasonic.md # 超声波测距
└── uart.md       # 串口通信
```

## 文档模板

复制以下模板新建或追加内容：

```markdown
# [模块名]模块踩坑

---

## 坑 N：[问题简要描述]

> [届次] · 发现者：[姓名]（若不清楚可省略）

### 现象

（描述观察到的异常现象，越具体越好）

### 原因

（解释为什么会出现这个问题）

\```c
// 错误写法（附注释说明问题所在）
wrong_code_here;
\```

### 解决方法

（给出修正方案，优先给出推荐方案）

\```c
// 正确写法
correct_code_here;
\```

### 举一反三

（这个问题还可能在哪些场景出现？有什么通用的预防规则？）
```

## 完整示例

以下是一篇虚构的完整踩坑文档：

---

### `uart.md`

---

# 串口模块踩坑

---

## 坑 1：波特率设置后串口乱码

> 通用 · 发现者：不详

### 现象

串口助手收到的数据全是乱码，明明设置了 9600 波特率。

### 原因

定时器 1 工作在方式 2（8 位自动重装），SMOD 位影响波特率计算，初始化时忽略了 PCON 的设置：

```c
// 错误写法：SMOD 默认为 0，波特率实际是期望值的一半
TMOD = 0x20;
TH1 = 0xFD;
TL1 = 0xFD;
TR1 = 1;
```

### 解决方法

确认 PCON 中 SMOD 的值，或直接用查表法确认 TH1 初值：

```c
// 正确写法（12MHz，9600波特率，SMOD=0）
PCON &= 0x7F;   // 确保 SMOD = 0
TMOD = 0x20;
TH1 = 0xFD;
TL1 = 0xFD;
TR1 = 1;
```

### 举一反三

- 不同晶振频率下 TH1 初值不同，务必对照波特率表
- SMOD=1 时波特率加倍，调试时可用于验证

## 写作建议

1. **现象要具体**：不要只写"输出错误"，要写"输出值为 65486 而不是预期的 50"
2. **原因要解释机制**：说清楚底层原理，而不只是"类型问题"
3. **代码要对比**：错误写法和正确写法放在一起，一眼看出区别
4. **举一反三要实用**：列出同类场景，让读者能触类旁通
