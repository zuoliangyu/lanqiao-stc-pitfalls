## 三、本周重点问题详解
### （1）计时器 else 分支无条件清零（本周最高频，4次）
- **问题描述：** LED超时判断永远不成立 / 长按检测永远不触发 / L4闪烁条件永远不满足
- **问题背景：** 第11届省赛（2次）、第12届省赛、第15届省赛
- **问题分析：** 在定时器中断中，计时变量的累加逻辑被 `else` 分支在每次中断中清零，导致计数器永远无法达到目标值：
```c
// 典型错误模式（出现4次，写法几乎一模一样）
if(++Timer_5s >= 5000)
Timer_5s = 5001; // 到达目标，限幅
else
Timer_5s = 0; // ← 致命！每次中断都清零
// 执行追踪：
// 中断1: Timer_5s: 0→1, 1>=5000? 否 → Timer_5s=0
// 中断2: Timer_5s: 0→1, 1>=5000? 否 → Timer_5s=0
// ... 永远到不了5000
```
另一种变体——长按检测中的相同模式：
```c
// 第12届：Time_Tick 用于1秒长按检测
if(Long_Press_Detection)
{
if(++Time_Tick >= 1000)
Time_Tick = 1001;
else
Time_Tick = 0; // ← 每次中断都归零，永远到不了1000
}
```
- **解决方案：** 删除 `else Timer = 0;`，将清零操作移到条件不满足（如Low_Flag为假）时执行：
```c
if(Low_Flag)
{
if(++Timer_5s >= 5000)
Timer_5s = 5000; // 限幅防溢出即可
}
else
Timer_5s = 0; // 条件不成立时才清零
```
> **规律总结：** 这是本周出现频率最高的Bug模式（4次），且几乎所有学员犯的错误写法一模一样。根本原因是对 `if-else` 语义的误解：`else` 分支在条件不成立时**每次都执行**，而非"只在第一次不成立时执行"。建议将此模式作为赛前代码自检的第一项。
### （2）函数声明与定义参数类型不匹配（bit vs unsigned char）
- **问题描述：** 数码管所有位的小数点全亮，无法单独控制
- **问题背景：** 第11届省赛
- **问题分析：** `seg.h` 中函数声明使用 `bit` 类型参数，而 `seg.c` 中定义使用 `unsigned char`：
```c
// seg.h 声明
void Seg_Disp(unsigned char wela, unsigned char dula, bit point);
// seg.c 定义
void Seg_Disp(unsigned char wela, unsigned char dula, unsigned char point);
```
在 8051 架构中，`bit` 类型通过进位标志（CY）传递，`unsigned char` 通过寄存器（R5/R7等）传递。调用者（main.c）包含 `seg.h`，按 `bit` 方式将 0/1 设入CY；但 `seg.c` 编译时按 `unsigned char` 从寄存器读取，读到的是寄存器中的**残留值**（通常非零），导致 `if(point) P0 &= 0x7f;` 几乎总是成立。
**注意：** Keil C51 对此不会报编译错误或警告，因为 `.h` 和 `.c` 分别编译，链接器不检查参数类型一致性。
- **解决方案：** 统一参数类型，将 `seg.c` 中的 `unsigned char point` 改为 `bit point`
> **规律总结：** 8051 的 `bit` 类型有独特的传参机制（通过CY而非寄存器），这与标准C的 `char/int` 传参完全不同。`.h` 和 `.c` 之间的类型不匹配在普通平台上可能只是警告，但在8051上会导致完全错误的数据传递。**写完函数后务必逐参数核对声明和定义。**
### （3）EEPROM 写入时机不符题意
- **问题描述：** 评测系统检查EEPROM地址0的值与预期不符
- **问题背景：** 第11届省赛（本次对话）、第13届国赛
- **问题分析：** 题目明确要求"**从参数界面退出时**，将电压参数VP×10保存到EEPROM"，但代码在每次按S16/S17调整参数时都立即写入：
```c
// 错误：每次按键都写EEPROM
case 16:
Voltage_Standard += 5;
EEPROM_Write(&Voltage_Standard, 0, 1); // ← 不应在此处！
break;
```
另一个变体更严重——在主循环中无条件写入：
```c
// 第13届国赛：led_proc() 中每次调用都写
void led_proc()
{
// ...
EEPROM_Write(&delay_j, 0, 1); // 每次主循环迭代都写！
}
// 后果：EEPROM寿命被快速消耗 + 主循环被阻塞~50ms/次
```
- **解决方案：** 仅在按S12退出参数界面时执行一次写入：
```c
case 1: // 参数界面
if(Key_Down == 12)
{
EEPROM_Write(&Voltage_Standard, 0, 1); // 退出时才写
Seg_Disp_Mode = 2; // 切换到计数界面
}
// S16/S17只修改变量，不写EEPROM
```
> **规律总结：** EEPROM写入时机必须严格按题意执行。常见错误有两种：（1）每次修改参数就写，（2）在主循环中无条件写。正确做法是"修改时只改RAM变量，退出/切换时才写EEPROM"。
### （4）BTN 独立按键模式误用矩阵扫描
- **问题描述：** 所有按键功能完全失效，但初始界面显示正常（得了19.1分纯靠不需要按键的测试点）
- **问题背景：** 第13届国赛
- **问题分析：** 题目要求 J5 配置为 **BTN 独立按键模式**，但代码使用了矩阵键盘的列扫描逻辑。在BTN模式下，S4-S7 直接连接 P3.3-P3.0 与 GND，按下时 P3.x 恒为低电平，与列线无关。矩阵扫描时三次列遍历都能检测到，后赋值覆盖前赋值：
```
按下S7 (P3.0=0):
第1次列扫描 P44=0: if(P30==0) temp=7; ← 正确
第2次列扫描 P42=0: if(P30==0) temp=11; ← 覆盖！
第3次列扫描 P35=0: if(P30==0) temp=15; ← 再次覆盖！
→ 最终返回15，代码检查 Key_Down==7 永远不匹配
```
所有按键的实际返回值都偏移了8：S4→12, S5→13, S6→14, S7→15
- **解决方案：** BTN模式下直接读P3端口，不做列扫描：
```c
unsigned char Key_Read()
{
unsigned char temp = 0;
if(P33 == 0) temp = 4;
if(P32 == 0) temp = 5;
if(P31 == 0) temp = 6;
if(P30 == 0) temp = 7;
return temp;
}
```
> **规律总结：** 赛前必须确认题目要求的J5跳线模式。BTN模式用独立读取，KBD模式用矩阵扫描，两套代码不能混用。建议准备两个版本的 `Key_Read()` 模板。
### （5）DS18B20 通信失败的复合原因
- **问题描述：** 上电后温度始终显示 55.9 或 59.9，与实际温度无关
- **问题背景：** 第11届省赛（多名学生）
- **问题分析：** 这个问题需要**同时修复两个bug**，只改一个仍然不行：
**Bug A — 1T单片机延时未补偿：**
```c
void Delay_OneWire(unsigned int t)
{
//t *= 12; // ← 被注释掉了！
while(t--);
}
```
STC15F2K60S2 是 1T 单片机，每时钟周期执行一条指令，比传统 12T 的 8051 快 12 倍。所有 OneWire 时序（复位脉冲、读写位）都快了约12倍，DS18B20 根本无法正确响应。
**Bug B — 发完转换命令后立即复位读取：**
```c
float rd_temperature()
{
init_ds18b20();
Write_DS18B20(0xcc);
Write_DS18B20(0x44); // 发起温度转换（需要750ms）
init_ds18b20(); // ← 立即复位！中断了转换！
Write_DS18B20(0xcc);
Write_DS18B20(0xbe); // 读取的是转换被中断后的垃圾值
// ...
}
```
**为什么显示55.9/59.9？** 通信完全失败时，总线保持高电平，读回全1（0xFFFF），0xFFFF / 16.0 ≈ 4095.9，对于 DS18B20 的 12 位有效数据，实际截取低12位后为 0x0FFF = 4095，但受符号位影响，0x07FF / 16 = 127.9 或特定截取方式得到 55.9/59.9。
- **解决方案：** 两个bug必须同时修复：
```c
// 1. 取消注释 t *= 12
void Delay_OneWire(unsigned int t) { t *= 12; while(t--); }
// 2. 改为"先读上次结果，再发起下次转换"
float rd_temperature()
{
unsigned char low, high;
init_ds18b20();
Write_DS18B20(0xcc);
Write_DS18B20(0xbe); // 先读上次结果
low = Read_DS18B20();
high = Read_DS18B20();
init_ds18b20();
Write_DS18B20(0xcc);
Write_DS18B20(0x44); // 再发起下次转换
return ((high << 8) | low) / 16.0;
}
```
同时确保调用间隔 ≥ 750ms（调度器周期设为1000ms）。
> **规律总结：** DS18B20 问题几乎每周都有，但本周的特点是**两个bug叠加**，修一个不够。赛前模板中应预置正确的"先读后转"模式和 `t *= 12` 补偿。
### （6）计时变量被多个功能共用
- **问题描述：** 频率测量值始终为0，数码管频率显示为0
- **问题背景：** 第13届国赛
- **问题分析：** `timer_1000ms` 同时用于两个功能：Timer1 ISR 中的1秒频率采样门控 和 主循环中的按键长按检测。两个功能互相争抢同一变量：
```c
// Timer1 ISR 中：频率测量需要 timer_1000ms 累积到 1000
if(++timer_1000ms == 1000)
{
timer_1000ms = 0;
freq = (TH0 << 8) | TL0; // 读取频率
TH0 = TL0 = 0;
}
// key_proc() 中（每10ms调用一次）：
if(timer_1000ms >= 1000) // 长按判断
{ /* 长按逻辑 */ }
else
{ timer_1000ms = 0; } // ← 每10ms就清零！
```
key_proc 每10ms调用一次，此时 timer_1000ms 最多累积到10就被清零，ISR中的 `==1000` 永远不成立，频率永远读不出。
- **解决方案：** 为频率测量和长按检测使用**独立的计时变量**：
```c
unsigned int timer_freq = 0; // 频率专用
unsigned int timer_long_press = 0; // 长按专用
```
> **规律总结：** 不同功能模块的计时变量必须独立，绝不能共用。即使两个功能恰好都需要"1秒定时"，也应该分开，否则任何一方的清零操作都会破坏另一方。
### （7）C语言未定义行为与数组越界破坏内存
- **问题描述：** 温度控制值被莫名覆盖清零，界面模式跳变
- **问题背景：** 第14届省赛
- **问题分析：** 这个项目中同时存在两个隐蔽的致命bug：
**Bug A — 自增赋值的未定义行为：**
```c
Seg_Show_Mode = (++Seg_Show_Mode) % 3; // ← 未定义行为！
```
`++Seg_Show_Mode` 和 `Seg_Show_Mode = ...` 在同一表达式中对同一变量做了两次修改，没有序列点，编译器可能产生任意结果。实际表现为模式可能从0直接跳到2（跳过1），导致界面切换时参数备份/恢复逻辑被绕过。
**Bug B — 数组越界覆盖相邻变量：**
```c
idata unsigned char Triggered_Rtc[3] = {0, 0, 0}; // 下标 0, 1, 2
Triggered_Rtc[3] = 0; // ← 越界！写入数组之外的内存！
```
在 idata 区域中，`Triggered_Rtc[3]` 正好覆盖了内存中紧邻的 `Temperature_Para` 变量，导致温度控制值被清零。由于 idata 变量按声明顺序连续分配，越界一个字节就能精准命中下一个变量。
- **解决方案：**
```c
// Bug A 修复：拆开自增和赋值
Seg_Show_Mode = (Seg_Show_Mode + 1) % 3;
// Bug B 修复：使用正确的下标范围
Triggered_Rtc[0] = 0;
Triggered_Rtc[1] = 0;
Triggered_Rtc[2] = 0;
```
> **规律总结：** 8051 没有内存保护机制，数组越界不会触发异常，而是静默地覆盖相邻变量，表现为"某个变量莫名其妙被改了"。排查此类问题的线索是：检查 idata 区域中被损坏变量的**前一个声明的数组**是否存在越界访问。

## 六、经验教训与注意事项
1. **计时器模式：** `if(++timer >= target)` 的 else 分支**绝对不要**无条件清零。清零应放在"外部条件不满足"的分支中（如 `if(!Low_Flag) timer = 0;`）
2. **函数签名核对：** 写完驱动函数后，逐参数对照 `.h` 声明和 `.c` 定义，尤其注意 8051 特有的 `bit` 类型
3. **EEPROM写入纪律：** 只在"值变化 + 特定时机（退出界面/切换模式）"时写入，绝不放在主循环或高频调度任务中
4. **按键模式模板：** 准备 BTN 和 KBD 两套 `Key_Read()`，赛前根据题目要求直接切换
5. **DS18B20 模板：** 预置"先读后转"模式 + `t *= 12` 补偿，作为固定模板不再手写
6. **变量独立原则：** 不同功能模块的计时变量、标志位必须独立声明，即使功能相似也不共用
7. **数组边界检查：** 声明 `arr[N]` 后，所有访问下标必须在 `0` 到 `N-1` 范围内；8051无内存保护，越界会静默覆盖相邻变量
