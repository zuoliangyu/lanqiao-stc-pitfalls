## 三、本周重点问题详解
### （1）第十五届省赛（第二次）温控题 4T 测评多项不通过
- **问题描述：** 10 项测评中 4 项错误：指示灯存在误亮（序号1）、DAC 输出为 0.0（序号3/7）、L4 闪烁 3s 后未熄灭（序号5）、温度变化后 2s 保持状态异常（序号9）
- **问题背景：** 第十五届省赛第二次题目（温度控制综合题）
- **问题分析：** 该问题由多个缺陷共同导致，非单一错误：
- [a] **调度器判断条件写错：** `if(task_rate + last_ms)` 只要不为 0 就成立，导致所有任务几乎每次循环都执行，整个程序失去节拍控制
- [b] **首次采样未做保护：** 上电时旧温度初始化为 0，第一次读到 23℃ 被误判为"升温"，导致 L2 等状态灯误亮
- [c] **L2/L3 显示变量与计时变量不是同一套逻辑：** 2s 定时结束后，真正参与显示的标志未被清除，灯不会按题意熄灭
- [d] **L4 触发条件混入历史状态：** 把 L4 触发写成依赖旧的上升/下降报警标志，而非直接由"本次温差 > 1℃"触发，导致闪烁时序错乱
- [e] **DAC 输出时序错误：** 温度尚未稳定时 DAC 变量初值为 0，但 DAC 输出函数已先执行，再加上调度器失效，测评时测到 0.0V
- **解决方案：**
- [a] 调度器改为 `if(now_time >= last_ms + rate_ms)`，正确比较当前时刻与下次执行时间
- [b] 第一次采样只记录温度不比较，从第二次开始做升降判断（`temperature_sample_count` 保护）
- [c] L2/L3 改为"事件保持灯"模式：检测到升/降温时启动 2s 计时器，显示时根据计时值决定亮灭
- [d] L4 独立做成"闪烁窗口"：仅由"本次温差绝对值 > 1℃"直接触发，0.2s 翻转，3s 后强制熄灭
- [e] 主函数上电后先 `while(rd_temperature() == 85);` 等待温度传感器返回有效值，再启动定时器和调度
### （2）长按计时器 else 分支持续清零导致无法触发长按
- **问题描述：** 学员自己实现的长短按功能，只能触发短按，长按始终无响应
- **问题背景：** 按键长短按检测（定时器中断中计时）
- **问题分析：** 学员原代码在定时器中断中写成：
```c
if(Long_Press_Flag) // 触发长按
{
if(++Time_15M >= 1500)
Time_15M = 1501; // 防止溢出
else
Time_15M = 0; // ← 问题在这里
}
```
`else` 分支的含义是：只要 `Time_15M` 还没累加到 1500，就立刻清零。这意味着 `Time_15M` 每次自增到 1 后下一轮就被清回 0，永远无法达到 1500，长按条件永远不成立
- **解决方案：** 去掉 `if-else` 中的清零分支，改为仅在按键未按下时才清零：
```c
if(Long_Press_Flag) // 按键处于按下状态
{
if(Time_15M < 1500)
{
Time_15M++; // 持续累加直到 1500，不清零
}
}
else // 按键未按下或已弹起
{
Time_15M = 0; // 只有松开时才清零
}
```
### （3）第十二届省赛（第二次）周期计算截断误差导致周期页全部失分
- **问题描述：** 4T 测评 64 分，频率页 4 项全部正确，但周期页 4 项全部错误，周期值均偏小（如标准 3330 显示 3289、标准 2000 显示 1972）
- **问题背景：** 第十二届省赛第二次题目（频率/周期测量）
- **问题分析：** 程序将周期值写成 `period = 1000000 / Freq;`，频率测量本身存在约 1% 的小误差，频率页尚可通过，但周期页对频率做倒数运算后误差被进一步放大，导致连续失分。此外还存在两个隐患：
- [a] **长按 S7 计时逻辑写错：** 与问题（2）同类错误，计时变量未达阈值前被反复清零，长按无法触发
- [b] **ADC 换算公式缺少比例系数：** `Volt_100x = Ad_Read(0x01) * 100` 少了 `/51`，电压值偏大
- **解决方案：**
- 周期计算加入四舍五入：`Period = (1000000UL + Freq / 2) / Freq;`
- 修正长按计时逻辑，按下时持续累加、松手时判断并清零
- ADC 换算补上比例系数：`Volt_100x = Ad_Read(0x01) * 100 / 51;`
### （4）第十三届省赛（第二次）页面状态机错误+DAC 线性公式整数除法
- **问题描述：** 4T 测评多项失分，界面切换后显示内容与预期不符，DAC 输出在中间段异常
- **问题背景：** 第十三届省赛第二次题目（电压测量 + 超声测距 + 参数设置综合题）
- **问题分析：** 该问题涉及多处逻辑缺陷：
- [a] **界面切换顺序错误：** 实际为 `U→P→L→U`，题目要求 `U→L→P→U`，导致切换后显示和 LED 全部错位
- [b] **参数界面和测距界面显示代码写反：** `case 1` 写了参数界面、`case 2` 写了测距界面，与页面定义不统一
- [c] **S5/S6/S7 有效界面判断错误：** 按键有效条件绑定了错误的 Mode 值，页面改正后按键也需同步修改
- [d] **DAC 中间段线性公式整数除法为 0：** `1/15*(Wave-20)` 中 `1/15` 为整数除法结果为 0，导致 20~80cm 区间 DAC 输出始终约为 1V
- [e] **测距未启动时 DAC 未清零：** 题目要求未启动测距时输出 0V，程序缺少 `Da_Write(0)`
- [f] **参数提交逻辑多加了题目未要求的限制：** 只有 `AD_up > AD_down` 才保存，但题目要求退出参数界面即生效
- **解决方案：**
- [a] 页面定义统一为 `0=U, 1=L, 2=P`，S4 切换顺序改为 `U→L→P→U`
- [b] `Seg_proc()` 中 `case 1` 对应测距界面 L、`case 2` 对应参数界面 P
- [c] S5/S6/S7 有效条件同步改为仅在参数界面（`Mode_show == 2`）有效
- [d] 线性公式避免整数除法，改为先乘后除：如 `(Wave - 20) * 4 / 60 + 1`（单位 V）
- [e] 测距未启动时显式执行 `Da_Write(0)`
- [f] 去掉额外的上下限比较限制，退出参数界面时直接生效
### （5）第十三届省赛（第一次）温度显示跳变
- **问题描述：** 功能基本都能实现，但温度显示会出现跳变现象
- **问题背景：** 第十三届省赛第一次题目
- **问题分析：** 温度读取操作未独立封装，可能与其他外设操作存在总线冲突或时序干扰，导致偶发读取到异常值，表现为显示跳变
- **解决方案：** 将温度读取单独封装为独立函数，确保读取时序完整不被打断
### （6）第十三届省赛（第一次）继电器不动作——三层问题叠加
- **问题描述：** 采集温度大于参数温度时，继电器旁的 L10 不亮
- **问题背景：** 第十三届省赛第一次题目（温度控制+时间控制综合题）
- **问题分析：** 该问题经历三轮排查，由三层缺陷叠加导致：
- [a] **温度控制参数未在模式切换时传递：** 程序定义了 `Temperatue_Parameter`（参数界面用）和 `Temperatue_Ctrol_Parameter`（控制逻辑用）两个变量，但模式切换时未互相赋值，导致 `Temperatue_Ctrol_Parameter` 始终为初始值 23，控制逻辑拿不到用户设定的参数。应在切换到温度控制模式（模式一）时将参数值赋给控制变量，切换到参数模式（模式二）时将控制变量回写给参数变量
- [b] **继电器底层驱动写法错误：** 学员自己写的底层直接操作 P0：
```c
if(Relay_enable == 1)
P0 = (P0 & 0xef) | 0x10;
else if(Relay_enable == 0)
P0 = P0 & 0xef;
```
这段代码只改了 P0 总线上的瞬时电平，没有选通 Y5 锁存器将数据写入继电器/L10 所在的外设组。正确的 `Relay()` 函数应通过缓存变量 `temp_2` 维护整组 Y5 外设状态（蜂鸣器/电机/继电器），修改对应位后整字节写入 P0，再选通 Y5（`P2 = (P2 & 0x1f) | 0xa0;`）完成锁存，最后恢复 P2。直接操作 P0 不锁存，且会与中断中的 LED 扫描等共享总线操作冲突
- [c] **开发板跳线帽未接到继电器位置：** 以上软件问题全部修正后继电器仍不动作，最终发现是硬件问题——开发板上的跳线帽未切换到继电器对应位置
- **解决方案：**
- [a] 模式切换时双向同步两个温度参数变量
- [b] 使用标准的 `Relay()` 底层驱动函数，通过缓存变量+Y5 锁存方式操作继电器，避免直接读改写 P0
- [c] 将开发板跳线帽接到继电器对应位置
### （7）第十三届省赛（第一次）4T 测评多项未通过——状态控制全面失调
- **问题描述：** 4T 测评多项未通过：温度控制模式下指示灯状态错误、L3 闪烁周期错误（多项）、温度低于参数值时继电器误吸合、时间控制模式下继电器应吸合却断开、参数界面显示异常
- **问题背景：** 第十三届省赛第一次题目（温度控制+时间控制综合题）
- **问题分析：** 非单一错误，而是状态控制、闪烁节拍、温度采样、按键逻辑四部分同时存在问题，叠加导致大面积失分：
- [a] **L3 闪烁周期写错：** 题目要求 L3 以 0.1s 间隔闪烁，但程序把翻转节拍写成 1000ms（应为 100ms），完整明灭周期变成 2s，所有与"继电器吸合时 L3 闪烁"相关的测试全部失败
- [b] **时间控制模式 5 秒窗口未完整复位：** 整点触发继电器吸合 5s 后，只关闭了继电器，但未清除"开始标志"和计数器，导致下一次整点到来时旧状态覆盖新触发，出现"应吸合却断开"
- [c] **温度采样不稳定：** 读取 DS18B20 时未关闭中断，且采样周期过快，中断打断单总线时序导致偶发读到异常高值，温度比较结果出错，继电器误动作
- [d] **按键功能跨页面耦合：** 时间显示页面中混入了"减参数"逻辑，导致按键在不同页面承担了不该有的功能，参数界面行为混乱

- **问题背景：** 第十二届省赛题目
- **问题分析：** 存在两个问题：
- [a] **底层驱动 `Delay_OneWire` 时序被破坏（主要原因）：** 学员可能为适配 12MHz 主频，在延时函数中加了 `t *= 12`，导致 OneWire 协议中所有时序（复位脉冲、读写时隙）放大为原来的 12 倍。DS18B20 对时序要求非常严格，时序一乱通信就失败，读出的数据自然是错的：
```c
void Delay_OneWire(unsigned int t) {
unsigned char i;
t *= 12; // ← 导致所有延时放大 12 倍
while (t--) {
for (i = 0; i < 12; i++);
}
}
```
- [b] **温度采集放在了 `Seg_Proc()` 显示函数中（结构问题）：** `Seg_Proc()` 执行周期为 500ms，而 DS18B20 在 12 位精度下一次转换需要 750ms，采集间隔不满足转换时间要求，且采集与显示逻辑耦合
- **解决方案：**
- [a] 删除 `Delay_OneWire` 中的 `t *= 12`，恢复为官方原版底层驱动。底层驱动不要随意修改，如需适配不同主频应调整循环内部迭代次数，而非对参数做倍乘
- [b] 将温度采集抽离为独立函数 `Temp_Proc()`，使用独立减速计数器，间隔设为 800ms，`Seg_Proc()` 只负责显示
### （19）串口 printf 浮点发送偶发小数点错位——定时器中断打断浮点格式化
- **问题描述：** 串口发送浮点数据时，正确值如 `10.00` 偶尔会发送成 `100.0`，小数点位置错乱
- **问题背景：** 串口通信功能，使用 `printf("%.1f"/"%.2f")` 格式化发送温度/浓度值
- **问题分析：** 8051 的 `printf` 浮点转字符串（`%f`）非常慢（几毫秒级），执行过程中 Timer1 中断会打断它。中断服务函数中的 `Seg_Disp` 数码管扫描操作使用的寄存器与 `printf` 内部使用的寄存器冲突，中断返回后 `printf` 拿到的是被改过的寄存器值，导致小数点位置错乱
- **解决方案：** 在执行 `printf` 前后临时关闭/恢复 Timer1 中断，防止浮点格式化过程被打断：
```c
ET1 = 0; // 关闭 Timer1 中断
printf("C:%.1f\r\n", Temperature);
ET1 = 1; // 恢复 Timer1 中断
```
### （20）第十四届省赛 温度界面值跳变导致最大值记录错误——中断打断单总线时序
- **问题描述：** 温度界面的值会偶发跳变，导致最大值记录被异常值污染
- **问题背景：** 第十四届省赛题目
- **问题分析：** DS18B20 单总线通信对时序要求非常严格，读取过程中如果被定时器中断打断，时序被破坏会导致偶发读到异常值，表现为温度显示跳变，异常高值被记录为最大值
- **解决方案：** 在 `rd_temperature()` 函数中，单总线通信前后加入开关总中断保护（`EA = 0` / `EA = 1`），确保读取过程不被中断打断：
```c
float rd_temperature() {
unsigned char low, high;
EA = 0;
init_ds18b20();
Write_DS18B20(0xcc);
Write_DS18B20(0x44);
Delay_OneWire(200);
init_ds18b20();
Write_DS18B20(0xcc);
Write_DS18B20(0xbe);
low = Read_DS18B20();
high = Read_DS18B20();
EA = 1;
return (float)(high << 8 | low) * 0.0625;
}
```
### （21）第十三届国赛 EEPROM 上电能读但写不成功——保存策略设计不合理
- **问题描述：** 上电后能从 EEPROM 读出继电器动作次数，但运行中次数变化后掉电再上电仍为旧值，写入不成功
- **问题背景：** 第十三届国赛题目
- **问题分析：** 问题不在 EEPROM 底层驱动，而在上层保存逻辑，三个原因叠加：
- [a] **旧值更新过早：** 程序在真正写 EEPROM 之前就把 `Relay_Num_Old = Relay_Num` 执行了，如果本次写入因锁标志校验未通过而没有执行，系统也不会再认为"数据发生了变化"，失去重试机会
- [b] **锁标志被当成运行时写入门禁：** 每次保存前都要先读取并校验锁标志，一旦锁标志异常，后续写入就会长期被阻断。锁标志更适合用于"上电判断数据是否有效"，不适合在运行时拦截每次写操作
- [c] **保存周期偏慢：** 保存任务是周期性执行（500ms），不是继电器变化后立刻保存，如果数据刚变化还没轮到保存任务就掉电，也会丢失
- **解决方案：**
- [a] 只在 EEPROM 写入成功后才同步旧值：先写数据，再写锁标志，最后才 `Relay_Num_Old = Relay_Num`
- [b] 运行时不再用锁标志拦截写入，只要数据变化就直接写，写完后顺带维护锁标志；锁标志仅在上电初始化时用于判断历史数据是否有效
- [c] 改进后的保存函数：
```c
void EEPROM_Save(void)
{
if(Relay_Num_Old != Relay_Num)
{
EEPROM_Write(&Relay_Num, 0, 1);
EEPROM_Write(&EEPROM_Lock, 22, 1);
Relay_Num_Old = Relay_Num;
}
}
```
### （22）第十五届国赛 程序阻塞卡死——多处致命错误叠加
- **问题描述：** 学员反馈代码逻辑与参考代码基本一致，但存在阻塞现象——要么切换不了界面，要么卡死在运行模式
- **问题背景：** 第十五届国赛题目（运动控制综合题）
- **问题分析：** 共发现 6 个问题，其中 3 个致命：
- [a] **Timer0 初始化 AUXR 操作符写错（致命）：** 将 `AUXR |= 0x80` 误写为 `AUXR &= 0x80`，`&=` 会清除 bit0~bit6，Timer0 没有进入 1T 模式，还可能影响串口等其他外设初始化
- [b] **Ctrl_Proc 调度周期太长（致命）：** 运动控制函数设为 20ms 执行一次（应为 1ms），位置更新频率太低，Dt 值大 20 倍导致浮点累积误差大，运动模拟异常无法正确到达目的地
- [c] **S4 按键缺少运行模式处理（致命/卡死根因）：** S4 只处理了空闲→运行和等待→运行，完全没有处理运行中按 S4 切换到等待模式的情况。一旦进入运行模式且速度为 0，设备永远不会到达目的地也不会自动切换状态，彻底卡死
- [d] **S9/S12/S13 按键没有限制在参数界面（严重）：** 这些按键在任何界面下都会生效，在位置/速度界面按下时会悄悄修改参数，引起不可预期的行为
- [e] **进入运行模式时没有清除标志位（中等）：** 未清除 `Get_Coordinate_Flag` 和 `Reach_Flag`，回到空闲状态后按 S4 会用旧坐标再次进入运行模式
- [f] **多余的 idata 变量浪费内存（中等）：** 声明了多个未使用的 `idata` 变量（共约 7 字节），8051 的 idata 仅 256 字节且与栈共享，挤压栈空间可能导致 `sqrt()`、`sscanf()` 等库函数调用时栈溢出，表现为程序随机卡死、界面切换失灵
- **解决方案：**
- [a] `AUXR &= 0x80` 改为 `AUXR |= 0x80`
- [b] `Ctrl_Proc` 调度周期从 20ms 改为 1ms
- [c] S4 按键处理中补充 `Status == 1` 时切换到 `Status = 2`（等待模式）
- [d] S9/S12/S13 处理加 `Seg_Mode == 2` 前置判断，仅在参数界面有效
- [e] 进入运行模式时同时清除 `Get_Coordinate_Flag = 0` 和 `Reach_Flag = 0`
- [f] 删除所有未使用的 idata 变量，释放栈空间
