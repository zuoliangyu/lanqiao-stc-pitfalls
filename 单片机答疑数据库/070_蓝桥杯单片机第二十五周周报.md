## 三、本周重点问题详解
### （1）DS18B20 时序期间数码管闪烁
- **问题描述：** 数码管第 5、6 位会不时闪烁一下
- **问题背景：** 第十二届省赛题目，DS18B20 温度读取
- **问题分析：** `rd_temperature()` 函数执行 DS18B20 单总线通信（`init_ds18b20()`、`Write_DS18B20()`、`Read_DS18B20()`），这些函数内部包含精确时序延迟（微妙级），而主循环或定时器中开启了定时器中断。定时器中断用于数码管动态扫描，在 DS18B20 通信期间中断被打断，数码管扫描时序错乱，导致部分位闪烁
- **解决方案：** 在进入 DS18B20 时序前关闭中断（`EA = 0`），时序结束后恢复中断（`EA = 1`），确保单总线通信不被中断打断，同时保证数码管扫描时序完整
```c
EA = 0;
init_ds18b20();
Write_DS18B20(0xcc);
Write_DS18B20(0x44);
// ... 其余时序
EA = 1;
```
### （2）第十三届 4T 测评多模块逻辑未闭环
- **问题描述：** 第十三届省赛题目（4T），测评项大量失败：温度大于参数值时继电器不吸合、整点继电器未吸合、L3 闪烁周期错误、U2/U3 界面切换后数码管仍显示 U1、参数加减后界面未刷新等
- **问题背景：** 第十三届省赛题目，涉及三个界面（U1 温度 / U2 时间 / U3 参数）、两种工作模式（温度控制 / 时间控制）、整点 5 秒事件、继电器控制等
- **问题分析：** 程序存在 4 个核心缺陷：
- [a] **继电器控制逻辑未闭环：** 工作模式变量只控制了 L2 指示灯，没有真正决定继电器状态；温度参数 ST 只参与显示和加减，未参与"温度大于参数时继电器吸合"的判断
- [b] **整点 5 秒事件缺乏独立变量管理：** 整点事件包含 L1 亮 5 秒、时间控制模式下继电器吸合 5 秒、继电器吸合时 L3 以 0.1s 闪烁三件事，仅靠 `if(分钟==0 && 秒==0)` 无法维持 5 秒状态
- [c] **S17 功能理解错误：** 题目要求在时间界面下，按住 S17 显示"分-秒"，松开恢复"时-分"，但学员写成了长按 2 秒切换的思路
- [d] **数码管刷新过慢：** 界面切换和参数修改后，数码管缓冲区更新太慢，测评系统抓取到的是旧界面数据
- **解决方案：**
- [a] 新增 `Relay_Flag` 变量统一管理继电器状态，在温度控制模式下按 `DT > ST` 判断，在时间控制模式下按整点事件判断
- [b] 新增 `Hour_Flag`、`Hour_Once`、`Hour_Time` 三个变量：`Hour_Flag` 标记整点 5 秒保持状态，`Hour_Once` 防止同一整点反复触发，`Hour_Time` 计数保持 5 秒；新增 `Led3_Flag`、`Led3_Time` 控制 0.1s 闪烁
- [c] 将 S17 改为按住即显示分秒（`S_Flag = 1`），松开即恢复时分（`S_Flag = 0`），移除长按计时逻辑
- [d] 将数码管刷新周期调快至 100ms，将温度采样从显示函数中分离（新增 `Temp_Proc()` 独立采样），确保界面切换时显示能即时刷新
### （3）第十一届 EEPROM 变量类型与保存时机错误引发连锁失败
- **问题描述：** 第十一届省赛 4T 测评中，参数界面的加减显示和数值比较全部异常：进入参数界面后显示 `P 6.00` 而非 `P 1.00`，越限判断、越线计数、状态指示灯全部与预期不符
- **问题背景：** 第十一届省赛题目，EEPROM 存储电压阈值参数用于越线检测与越限判断
- **问题分析：** 第一份代码存在两个根本错误：
- [a] **EEPROM 存储变量与运行阈值混用：** `voltage_disp_10x` 定义为 `unsigned int`（2 字节），但 EEPROM 仅读写 1 字节内容，在 C51 环境下导致初值读歪，阈值从正常的 `10`（即 1.0V）变成了异常大值（约 6V），后续所有比较和越线计数均基于错误阈值运行
- [b] **EEPROM 保存时机错误：** 题目要求退出参数界面时保存（P → N），但代码写在了切回数据显示界面时保存（N → U），导致 4T 检测 EEPROM 地址 0 时读到的仍为旧值
- [c] **连锁反应：** 参数初值错误直接导致越限判断（L1 状态）、越线计数（Num）、清零测试等后续测评项全部失败，并非独立新 bug
- **解决方案：**
- [a] 将 EEPROM 读写变量改为 `unsigned char`（1 字节），同时新增独立的运行阈值变量（如 `Voltage_Control_100x`），实现"存储值、显示值、运行值"三者分离
- [b] 将 EEPROM 写回时机改为退出参数界面时（`Seg_Disp_Mode == 2` 时），而非切回数据显示界面时
- [c] 上电时先读 EEPROM 到显示变量，再拷贝到运行阈值变量：`Voltage_Control_100x = Voltage_Disp_10x * 10;`，后续采样比较统一使用运行阈值
### （4）第十四届 4T 测评采集状态流未走顺
- **问题描述：** 第十四届省赛 4T 测评大量失分：历史界面尚未采集就出现数据、触发采集后未进入 `E xx-yy` 页面、温度/湿度最大值一直不更新、湿度非法时 `AA` 无法显示、时间回显界面显示当前时钟而非最后一次采集时间
- **问题背景：** 第十四届省赛题目，暗光触发采集 → 温湿度采样 → 历史记录入库，涉及多步状态流转
- **问题分析：** 学员代码的采集数据链存在 7 处缺陷：
- [a] **暗光触发旧值未保存：** 触发暗光采集时保存了上一次光照值，但每次采样后没有更新旧值，导致触发条件实际很难按预期成立
- [b] **未采集就提前写入历史数据：** 历史记录更新条件太宽（仅 `error_flag==0 && entry_data==0`），在温湿度实际完成采集前就已写入假数据
- [c] **湿度采样标志写反：** 判断条件写成了"已经采完了才去采"，导致湿度采样根本不会执行，湿度始终为 0
- [d] **非法频率判断逻辑错误：** `freq<200 && freq>2000` 使用了 `&&` 而非 `||`，条件永远不成立，`AA` 无法显示
- [e] **最大值更新写反：** 把"当前值赋给当前值"而非"当前值赋给最大值变量"，导致最大值长期保持 0
- [f] **L4 判断使用旧温度值：** 比较温度参数时使用的是上一轮的旧值，导致 L4 闪烁状态慢一拍
- [g] **LED 状态未完整刷新：** 部分 LED 只在满足条件时赋值，不满足时不主动清零，遗留上一轮状态
- **解决方案：**
- [a] 每次光照采样后更新旧值：`voltage_value = temp;`，触发条件改为 `temp<80 && voltage_value>80`（由亮变暗才触发）
- [b] 修改入库条件为"温度和湿度都完成触发后才允许写入一次"：`if(temper_trigger_over==1 && hum_trigger_over==1 && entry_data==0)`
- [c] 湿度采样标志判断纠正为"未完成时去采"：`if(hum_trigger_over==0)` 进入采集
- [d] 非法频率判断改为 `||`：`if(freq<200 || freq>2000)` 判非法，显示 `AA`
- [e] 最大值更新修正为 `temper_max = temper_val;` / `hum_max = hum_val;`
- [f] L4 比较使用最新采样值：`l4_flag = (temper_val > temper_para);`
- [g] LED 处理函数中每次完整赋值所有 LED 状态，不满足条件时显式清零
### （5）超声波运动检测多状态混淆与计数错误
- **问题描述：** 超声波运动检测项目（test-32）4T 测评大量失分：L1-L4 状态指示灯错误、继电器次数统计异常、运动界面显示 `L0` 而非正确状态、S8+S9 组合键清零无效、距离值整体偏小
- **问题背景：** 涉及超声波测距、接近判定、运动状态判定（静止/慢速/快速）、温度+高温双条件继电器控制、S8+S9 组合键清零等
- **问题分析：** 学员代码存在 7 个核心问题：
- [a] **接近判定与运动状态锁定混用同一变量：** 变量 `suo` 同时表示"已接近"和"状态锁定 3 秒"，物体接近时 `jiejing` 只在进入瞬间为 1，随后被清空，导致 LED 和继电器"闪一下就灭"
- [b] **L1-L4 未在未接近时清零：** LED 处理只有"接近时怎么亮"，没有"未接近时怎么灭"，上次亮过的灯一直保留
- [c] **继电器次数按循环次数累加：** 只要满足"接近+高温"就 `jici++`，统计的是循环执行次数而非吸合次数，导致 NC 数值暴增
- [d] **S8+S9 组合键判断逻辑不可能成立：** `Key_Down==8 && Key_Down==9` 永远为假，且底层按键驱动也不支持组合键返回值
- [e] **运动界面显示变量未更新：** 显示的是 `moshi` 变量，但从未正确更新，导致频繁显示 `L0 xxx`

- **问题描述：** 校准 -100 时 `F1500` 显示成 `F1615`（应为 `F1400`）、DAC 电压从 1.8V/2.3V 偏至 2.0V/2.6V、历史最大值 `HF250` 变成 `HF152`、历史回显时间不对（如 `HA133445` 变成别的时间）

- **问题描述：** 光照由亮变暗触发后，温湿度"是否都比上一次触发时更高"的判断结果不稳定，甚至触发后比较失真

- [a] **温湿度 old 值保存时机错误：** 学员在本次触发采集完成后，立刻执行了 `Temperature_Value_old = Temperature_Value; Humidity_Value_old = Humidity_Value;`，导致本次触发值刚算出来，`old` 就被覆盖成这一次的值。后续比较时 `current` 和 `old` 已经一样，"当前值大于上一次值"这个条件失真
- [b] **比较位置放错：** 温湿度比较逻辑写在了 `Get_Temperature()` 温度采集函数里，但采集函数只是周期读取当前温度，不是"触发事件真正成立的时刻"。比较应该在"亮到暗触发成立且本次温湿度已锁存完成"后进行
- [c] **三种旧值变量类型混淆：** `AD_old` 表示"上一轮轮询光照值"（每轮更新是对的），但 `Temperature_Value_old` 和 `Humidity_Value_old` 表示"上一次有效触发时锁存的值"，不能像 `AD_old` 一样每轮直接覆盖
- **解决方案：**
- [a] 温湿度 old 值更新放到触发处理函数（`AD_DA()`）末尾，顺序必须是：**先比较当前值与 old 值 → 更新上升标志位 → 最后再更新 old 值**，即 "先比较，再更新 old"
- [b] 将温湿度比较从 `Get_Temperature()` 移出，收回到触发处理函数中的正确位置：亮到暗触发成立 → 锁存温湿度 → 与上一次触发值比较 → 得出 `Temperature_Humidity_Rise_Flag` → 最后更新 old
- [c] `Get_Temperature()` 只负责更新当前温度值：`Temperature_now = rd_temperature();`，不再做旧值比较
- 修改原则：`old` 是"上一轮扫描值"则每轮循环后更新；`old` 是"上一次有效事件值"则在"事件处理结束后"再更新

### （8）十届国赛 S16/S17 参数界面运算符优先级错配
- **问题描述：** 参数界面中按 `S16`（减 2）和 `S17`（加 2），数码管显示的参数值无变化；`S18/S19` 同样无效
- **问题背景：** 第十届国赛题目，S16/S17 控制温度参数以步长 2 变化，S18/S19 控制距离参数以步长 5 变化
- **问题分析：**
- [a] **运算符优先级错误（根因）：** 参数增减语句写成 `if(Temperature_Para-=2<=0)`，由于 `<=` 运算符优先级高于 `-=`，编译器先计算 `2<=0` 得到 `0`，再执行 `Temperature_Para -= 0`，等价于没有变化。同理 `+=2>=99`、`-=5<=0`、`+=5>=99` 均存在相同问题
- [b] **参数变量未赋初值：** 参数变量只定义不赋初值，默认为 `0`，在错误表达式存在时 `0 -= 0` 和 `0 += 0` 均不改变值，进一步放大现象
- (c) **以前没遇到此问题的原因：** 常见正确写法要么加括号明确优先级 `if((Temperature_Para-=2)<=0)`，要么用三目运算符 `Temperature_Para = (Temperature_Para <= 2) ? 0 : Temperature_Para - 2`，不会产生歧义
- **解决方案：**
- [a] 重写参数修改逻辑，使用三目运算符避免优先级歧义：
```c
Temperature_Para = (Temperature_Para <= 1) ? 0 : Temperature_Para - 2;
Distance_Para = (Distance_Para <= 4) ? 0 : Distance_Para - 5;
```
- [b] 给参数变量设置明确初值：`Temperature_Para = 30; Distance_Para = 35;`
- **经验教训：** "赋值运算"和"比较运算"写在同一行必须特别警惕，最好改成"先判断、再赋值"或用三目运算符

- [a] 修正赋值：`Distance_Old=Distance;` 和 `Distance_Count=1;`；补全运动状态锁定逻辑：新增 `Exercise_Lock_Flag` 和 `Time_3s`，状态变化后锁定 3 秒不重新判定

- [c] 继电器计数改为上升沿统计：仅在 `Relay_enable==0` 时吸合才 `Relay_Count++`
- [d] 组合键清零改为 `if(Key_Val==89)` 持续计时、满 2 秒清零、松手重置计时

- [f] `Get_Distance` 调度周期改为 1000ms
- **经验教训：** `==` 和 `=` 是最难肉眼发现的 Bug 之一，调试困难时应开启编译器警告（`-Wparentheses` 或 `-Wall`）；参数界面锁定是本题的必考点，进入时需冻结所有输出状态
### （25）参数界面长按按键无法连续加减导致数值只能变化一次
- **问题描述：** 参数设置界面中，按住加键后数值只能加一次，无法长按连续递增
- **问题背景：** 通用参数界面题目，矩阵键盘下长按加减键实现连续增减功能
- **问题分析：** 按键短按正常，问题出在长按计时变量被提前清零：
- [a] **长按计时被后续分支提前清零：** 程序按顺序先判断加键长按（进入后持续执行加法），但代码继续往下执行减键处理分支。由于当前按下的不是减键，减键分支的 `else` 把长按计时变量 `Count_Start` 和 `Count` 清零，长按状态刚建立就被打断
- [b] 执行链：`加键按下 → Count_Start=1 → Count 计时到>500 → 加一次 → 继续执行减键逻辑 → else 清零 Count → 下次扫描长按条件不再成立`
- [c] **短按清零位置不完整：** 减键的短按时钟重置也不够完整，可能导致后续加减键逻辑风格不统一、计时状态残留
- **解决方案：** 重构按键处理为"先统计按下状态、再分短按/长按处理"的结构，确保长按计时不会被其他分支误清零：
- [a] 先统一判断按键 14/15 是否按下启动计时：`if(Key_Down==14 || Key_Down==15) Count_Start=1;`
- [b] 按 Count 分为短按和长按两支处理：`Count<500` 处理短按（Key_Up），`Count>=500` 处理长按（Key_Old）；长按分支最后用 `else` 统一清零
- [c] 加减键使用相同结构，确保短按/长按逻辑风格一致
- **经验教训：** 长按类按键处理中，多个按键的判断分支必须互不干扰；长按计时清零应放在"松手或无按键时"的统一路径中，不能放在其他按键的 `else` 分支里
### （26）温度题DAC温控模式默认错误、温度单位错用与短按长按复用风险
- **问题描述：** 4T 测评中 DAC 温度控制项目失败：温度 30℃、校准值 -5 时期望输出 3.5V，实际为 0.0；温度 13℃、校准值 -5 时期望输出 2V，实际为 0.0。温度显示和校准加减均正常
- **问题背景：** 第十五届省赛题目，DAC 自动温控输出与手动控制切换，温度→电压公式：10℃ 对应 2V，40℃ 对应 5V，区间线性插值
- **问题分析：** 失分不在显示，而在 DAC 控制逻辑有两层错误：
- [a] **DAC 工作模式默认状态不合理：** 默认是手动模式而非温控模式，测评流程未先切换到温控模式时，温控公式代码根本不会被执行，DAC 输出始终为 0.0V
- [b] **温控公式使用错误的温度单位：** 温度保存为"10 倍值"（25℃→250），但 DAC 计算时直接拿 250 与 10/40 比较。25℃ 实际判断的是 250，8℃（80）被误判为 ">=40" 输出 5V。公式完全失真
- [c] **短按/长按功能复用风险：** 同一按键同时用于 DAC 模式切换、长按计时、继电器控制，处理位置不统一，测评脚本时序与代码状态机不一致时容易模式切换失败
- **解决方案：**
- [a] DAC 默认模式改为温控模式（`Dac_Work_Mode = 0`），符合测评预期
- [b] 温度计算时统一使用"实际温度值"而非 10 倍值：`rd_temperature()` 加校准后得到真实摄氏温度，直接与 10℃/40℃ 比较：
```c
if (temp <= 10.0f)
Dac_Digit_Temperature = 2 * 51; // 2V
else if (temp >= 40.0f)
Dac_Digit_Temperature = 5 * 51; // 5V
else
Dac_Digit_Temperature = (unsigned char)((temp / 10.0f + 1.0f) * 51.0f); // 线性插值
```
- [c] 统一 DAC 输出函数：分别算出温控值和手动值，再根据模式选择写入，确保任何模式下都执行 `Da_Write()`
- [d] 短按/长按按"按下计时、松开判断"结构统一处理，避免功能互相干扰
- **经验教训：** 变量存储格式（10 倍值/真实值）在整个计算链中必须保持一致，不要"存的时候放大、算的时候忘记还原"；DAC/PWM 等输出类功能不管模式如何都应周期性调用 `Da_Write()`，用变量决定输出值而非决定要不要输出
### （27）第十五届省赛频率题频率超限LED闪烁状态未退出与比较口径不统一
- **问题描述：** 当超限频率 >= 当前频率时，LED0 没有恢复为稳定熄灭状态，而是停在常亮或常灭不变，看起来像"卡住不闪烁"
- **问题背景：** 第十五届省赛频率采集题，LED0 在"当前输出频率 > 超限频率"时按 0.2s 周期闪烁
- **问题分析：** 核心不是"0.2s 闪烁写错了"，而是状态机没有收口：
- [a] **超限判断使用了原始频率而非调整后频率：** 显示看的是 `Freq + Freq_cail_crtol`（校准后的最终频率），但 LED 判断写的是 `Freq_limter < Freq`（原始测量值）。校准值修改后显示变了但 LED 行为不一致
- [b] **停止闪烁时缺少复位动作：** 不超限时只写了 `Timer_flag = 0`，没有清零 `Time_200ms`、没有清零 `led_flag`、没有明确 `ucLed[0] = 0`。退出闪烁时 `led_flag` 保留退出前的最后一次值，导致 LED 可能常亮或常灭
- [c] **等于情况未处理：** 分别写了 `if(Freq_limter < Freq)` 和 `if(Freq_limter > Freq)`，当 `Freq_limter == Freq` 时两边都不进入，LED 保持上一拍结果
- [d] **调整后频率使用无符号变量存在下溢风险：** `Freq_disp = Freq + Freq_cail_crtol` 中校正值可为负，无符号变量下溢成大数
- [e] **最大频率记录用了原始值：** 记录峰值时用 `Freq` 而非调整后的最终频率，口径不一致
- **解决方案：**
- [a] 统一计算最终频率作为"唯一真相"：新增 `Freq_Final = (long int)Freq + Freq_cail_crtol`，显示、比较、记录最大值全部使用该变量
- [b] LED 闪烁退出时完整复位：不超限时同步清零 `Timer_flag`、`Time_200ms`、`led_flag`，并设置 `ucLed[0] = 0`
```c
if(Freq_Final > Freq_limter)
Timer_flag = 1;
else {
Timer_flag = 0;
Time_200ms = 0;
led_flag = 0;
}
ucLed[0] = led_flag;
```
- [c] 用 `>` 判断超限，用 `else` 兜底覆盖等于和小于两种情况，不再分别写 `<` 和 `>` 导致悬空
- [d] 最终频率改为 `long int` 带符号类型，避免负校准值导致下溢
- [e] 新增 `Freq_Proc()` 函数统一计算最终频率、更新最大值及时间，作为独立任务周期执行（20ms），确保 LED/显示始终使用一致的数据
- **经验教训：** 状态进入和退出必须成对设计，退出时要把相关标志位和计时变量一起复位；显示、判断、记录必须统一使用同一个"最终有效量"，不要让不同逻辑用不同口径的数据
### （28）第十四届省赛按键无反应三连 — 滴答溢出/数码管残留/按下抬起写反
- **问题描述：** 学员刚开始做第十四届省赛题目，按下按键 4 切换界面，但数码管毫无变化；后续补充按键任务执行后 Led、Seg 等任务也完全停摆
- **问题背景：** 第十四届省赛题目，多界面按键切换+调度器任务调度框架
- **问题分析：** 学员代码同时中了三枪：
- [a] **滴答时钟溢出导致调度器"死锁"（致命）：** `uwTick` 定义为 `unsigned char`（8 位，最大值 255），每 1ms 中断加 1，到 255 后会清零重新计数。但调度器结构体 `task_t` 中 `rate_ms` 和 `last_ms` 都是 `unsigned long int`（32 位）。开机经过 255ms 后，比如数码管任务（100ms 周期）`last_ms` 变成 200，下一次需要运行的时间是 200+100=300。此时判断条件 `if(now_time >= (Scheduler[i].rate_ms + Scheduler[i].last_ms))` 变成 `if(now_time >= 300)`。因为 `now_time` 最大只能跟随 `uwTick` 达到 255（溢出回零后在 0~255 徘徊），这个条件永远不会成立。调度器彻底瘫痪，按键和数码管任务都不会再执行了
- [b] **显示问题：数码管没有清除旧状态：** 假设按键生效了，`Seg_Show_Mode` 从 0 变成了 1。`Seg_Proc` 中模式 0 只赋值了 `Seg_Buf[3]`，模式 1 只赋值了 `Seg_Buf[4]`，切换模式时没有把另一个位置清空（恢复成 10 熄灭状态）。结果就是按键按下后第 3 位和第 4 位都亮了，且一直亮着，视觉上看起来就像"数码管没变化"
- [c] **逻辑隐患：按下与抬起判断反了：** 蓝桥杯开发板按键通常是"按下为 0，松开为 1"。学员的边沿检测代码 `Key_Down = Key_Val & (Key_Val ^ Key_Old)` 检测的是 0→1（释放），`Key_Up = ~Key_Val & (Key_Val ^ Key_Old)` 检测的是 1→0（按下），实际体验变成松手才切换
- **解决方案：**
- [a] 将 `uwTick` 改为 `unsigned long int`，或调度器判断改用 `(uwTick - Scheduler[i].last_ms) >= Scheduler[i].rate_ms`（差值判断，天然抗溢出），推荐后者（更省 RAM）
- [b] `Seg_Proc` 中每个 `case` 分支先清零所有 `Seg_Buf` 位，再赋值对应位置
- [c] 互换 Key_Down 和 Key_Up 的检测公式，或在 `Key_Read()` 内部对键值取反
- **经验教训：** 调度器时间基准类型必须与比较运算兼容，8 位变量参与 32 位运算 + 溢出回零时极易触发"永不满足"的死锁；显示切换类逻辑务必保证"切走就清"，不留残留
