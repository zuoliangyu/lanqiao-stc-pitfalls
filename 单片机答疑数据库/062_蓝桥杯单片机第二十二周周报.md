## 三、本周重点问题详解
### （1）按键扫描干扰 T0 计数器导致频率偏大 100
- **问题描述：** 频率界面每次显示都比真实值大 100
- **问题背景：** 第十五届省赛题目
- **问题分析：** 按键扫描函数末尾执行 `P3=P3|0xEF;`，`0xEF = 1110 1111`，OR 操作未恢复 P3.4（bit4 保持原值）。而 P3.4 正是 T0 计数器的外部输入引脚，每次按键扫描过程中对 P3 口的操作会在 P3.4 上产生下降沿，被 T0 计数器误计为一次脉冲。定时器 10ms 中断扫描按键，每秒扫描 100 次，因此频率测量值恰好偏大 100
- **解决方案：** 将 `P3=P3|0xEF;` 改为 `P3=P3|0xFF;`，确保 P3.4 在按键扫描结束后恢复为高电平，避免干扰 T0 计数器输入
### （2）DAC 输出值不正确（int 截断 float）
- **问题描述：** DAC 输出电压与预期值不符
- **问题背景：** 第十五届省赛题目
- **问题分析：** `freq_dat` 声明为 `idata int freq_dat = 0;`，但 DAC 输出的计算公式涉及浮点运算，int 类型会截断小数部分，导致精度丢失，输出电压产生偏差
- **解决方案：** 将 `freq_dat` 改为 `float` 类型，或调整计算顺序采用先乘后除的整数运算方式避免精度丢失
### （3）DAC 界面显示与按键逻辑误绑 ad_mode 导致测评失败
- **问题描述：** 板子上手动测试现象正确，但 4T 自动测评 DAC 相关步骤全部报错——数码管始终显示 A 102，按键加减无效
- **问题背景：** 第十五届省赛题目
- **问题分析：** 存在两处逻辑错误：
- [a] **显示内容搞混：** DAC 界面（`seg_mode == 1`）按 `ad_mode` 区分了两种显示——自动模式显示 `da_show`，手动模式显示 `ad`。但题目要求 DAC 界面始终显示 DAC 数字量（即 `ad`），与输出模式无关。测评时处于自动模式，界面显示的是温度自动计算出的 102，而非初始值 100
- [b] **按键加了多余判断：** S16/S17 调整 DAC 数字量时加了 `if(ad_mode)` 判断，导致只有手动模式下才能修改 `ad` 值。题目要求只要在 DAC 界面，按键就能调整数字量，与输出模式无关。测评时按加减键值根本不变
- **解决方案：** DAC 界面的"显示"和"按键调整"均不应受 `ad_mode` 限制。`ad_mode` 仅决定实际 DAC 输出走自动还是手动，不影响界面显示内容和按键操作。去掉显示和按键中对 `ad_mode` 的条件判断即可
### （4）串口乱码 + ADC 电压最大只到 1.30V（底层代码笔误）
- **问题描述：** 串口打印出现乱码，同时电位器 ADC 读数最大只能到 1.30V
- **问题背景：** 第十四届省赛模拟题2
- **问题分析：** 存在两处底层代码笔误：
- [a] **串口乱码：** `putchar()` 函数中 `while(TI == 0)` 后面漏写分号，导致 `TI = 0;` 被当作 while 循环体。TI 置 0 后循环条件重新成立，形成死循环，串口发送时序被破坏，输出乱码
```c
// 错误写法
while(TI == 0) // ← 缺少分号
TI = 0; // 被当作循环体，死循环
```
- [b] **ADC 上限异常：** `ad_da()` 中 ADC 换算公式写成了 `ad_read(0x43) * 51.0f / 100`（乘以 51），正确应为除以 51。ADC 返回值 0-255 对应 0-5.00V，换算公式应为 `ad_read(0x43) / 51.0f * 100`，乘以 51 导致结果远超正常范围，显示时被截断
- **解决方案：**
- [a] `while(TI == 0)` 后补上分号：`while(TI == 0);`
- [b] 将 `* 51.0f / 100` 改为 `/ 51.0f * 100`（或等价写法 `* 100 / 51.0f`）
### （5）运动状态显示错误 + 参数界面 L8 闪烁失效（定时器同步与状态冻结）
- **问题描述：** 跑动状态下数码管应显示 L3，实际显示 L1（静止）；进入参数设置界面后 L8 应保持之前的闪烁状态，但实际停止闪烁
- **问题背景：** 第十六届省赛真题。学员使用 DeepSeek、豆包等 AI 工具反复调试仍无法解决
- **问题分析：** 存在两处设计缺陷：
- [a] **运动状态被立刻覆盖（测评28、49、61）：** `Get_Distance` 调度周期为 1000ms，运动状态锁定时间为 3000ms。1000 整除 3000，导致第 4 次采样恰好卡在锁定到期瞬间，锁刚解除就立刻重新判定状态，此时连续两次距离值相近（物体已停下），状态被立刻覆盖为"静止"（L1），跑动状态（L3）无法维持
- [b] **L8 闪烁缺少状态冻结（测评22、63）：** 中断里闪烁条件只判断 `Motion_State==3` 才驱动闪烁计时器，但进入参数界面后 `Motion_State` 可能被新的距离采样覆盖为 1（静止），闪烁随之停止。代码缺少状态冻结机制——题目要求进入参数界面时 LED 状态"锁定"，但没有保存进入前的运动状态快照
- **解决方案：**
- [a] 采样周期改为 999ms（错开整除关系），锁定时间改为 3050ms（增加 50ms 余量防止边界竞争）。**教训：** 定时器设计中要避免周期之间存在整除关系，否则会产生同步问题
- [b] 新增 `Run_State`、`Stay_State`、`Paralysis_State` 三个冻结变量，在非锁定时实时更新，锁定时保持不变；中断中闪烁条件改为 `Motion_State==3 || (Led_Lock && Run_State)`；`Led_Proc` 锁定分支中用冻结变量控制 L8。**教训：** 题目说"状态锁定"时，需用额外变量快照当前状态，不能直接依赖实时变化的原始变量
### （6）DS18B20 温度始终显示 0（底层延时函数被注释）
- **问题描述：** 温度显示界面始终为 0，烧录课程例程源码也同样为 0，怀疑例程代码有误
- **问题背景：** DS18B20 程序设计课程
- **问题分析：** `Delay_OneWire()` 函数中关键的延时校准语句 `t *= 12;` 被注释掉了。DS18B20 采用单总线（OneWire）协议，对时序要求极其严格——复位脉冲、写时隙、读时隙都依赖精确的微秒级延时。`t *= 12` 是针对当前单片机主频的延时校准系数，注释掉后实际延时远小于协议要求，导致 DS18B20 初始化失败、数据读写全部异常，温度值始终为 0
```c
void Delay_OneWire(unsigned int t)
{
// t *= 12; // ← 被注释掉，延时不足，OneWire 时序全部失败
while(t--);
}
```
- **解决方案：** 取消 `t *= 12;` 的注释即可恢复正常温度读取
> **附注：** 该校准系数与单片机主频相关。蓝桥杯竞赛板使用 STC15F2K60S2（11.0592MHz），底层驱动代码中的延时系数需要匹配实际主频，不可随意注释或修改
### （7）函数名大小写不一致导致编译警告
- **问题描述：** Keil 编译时报 `warning C206: 'System_Init': missing function-prototype`
- **问题背景：** 日常课程练习
- **问题分析：** 头文件（.h）中声明函数时使用小写 `System_init()`，但 `main.c` 中调用时写成了大写 `System_Init()`。C 语言严格区分大小写，`System_init` 与 `System_Init` 是两个不同的标识符，编译器在当前作用域找不到 `System_Init` 的原型声明，因此报出 missing function-prototype 警告
- **解决方案：** 统一函数名大小写。将调用处 `System_Init()` 改为与头文件声明一致的 `System_init()`，或反过来修改头文件声明
### （8）在压缩包内直接打开 Keil 工程导致路径错误
- **问题描述：** 打开 Keil 工程后弹窗报错 `C:\Users...\AppData\Local\Temp\360zip$Temp...\Driver\Key.c contains an incorrect path`，工程无法正常使用
- **问题背景：** 学员打开群文件中的大模板工程
- **问题分析：** 学员没有先解压压缩包，而是在 360 压缩的预览窗口中直接双击 `.uvproj` 文件打开了工程。压缩软件会将文件临时释放到 `AppData\Local\Temp\360zip$Temp` 目录下，这是一个临时路径，Keil 工程中的相对路径引用（如 `Driver\Key.c`）在该临时目录下无法正确解析，导致路径错误
- **解决方案：** 先将压缩包完整解压到一个固定目录下（如桌面或项目文件夹），再从解压后的目录中打开 `.uvproj` 工程文件
### （9）上电数码管先显示 85（DS18B20 初始化读取方式不当）
- **问题描述：** 已添加延时函数，但上电后数码管仍然先显示 85 再跳变为正常温度
- **问题背景：** 第十一届省赛题目
- **问题分析：** 学员在 `main()` 初始化阶段调用了 `Get_Temperature()` 来预读温度、消除 85。但 `Get_Temperature()` 是带业务逻辑的封装函数，内部包含调度计时判断，首次调用时条件不满足会直接 `return`，实际并未执行底层的温度转换与读取，因此 DS18B20 的上电默认值 85°C 没有被刷掉
```c
// 错误做法：调用带逻辑判断的封装函数，首次会被 return 跳过
Get_Temperature();
Delay750ms();
```
- **解决方案：** 初始化阶段直接调用底层读取函数 `rd_temperature()` 而非 `Get_Temperature()`，确保真正执行一次温度转换和读取，将 DS18B20 内部寄存器的上电默认值 85 刷掉
```c
// 正确做法
while(rd_temperature() == 85); // 循环读取直到不再是上电默认值
```
### （10）EEPROM 加锁机制导致 4T 测评不通过
- **问题描述：** 4T 测评中其他功能全部正确，唯独 EEPROM 存储项全部报错——期望读出 10，实际读出 20/15 等错误值。加锁后得了 20 多分，去掉锁后只剩 EEPROM 不通过
- **问题背景：** 第十一届省赛题目
- **问题分析：** 学员在 EEPROM 读写中添加了自定义的校验/加锁机制。4T 自动测评系统按照题目约定的地址直接读写 EEPROM，不会配合用户自定义的锁逻辑，导致测评系统写入的数据被锁机制干扰，读出值与预期不符。这是 4T 测评的已知兼容性问题，米醋论坛上已有相关反馈
- **解决方案：** 4T 测评时不要在 EEPROM 中加锁/校验机制，直接按题目约定的地址进行读写即可。自定义校验逻辑仅适用于实际产品场景，不适用于自动评测环境
### （11）闹钟按键关闭逻辑嵌套在 AM/PM 分支内导致部分时段无法关闭
- **问题描述：** 闹钟响后按任意按键应关闭闹钟，但实际按键无法关闭
- **问题背景：** 模拟题2，闹钟功能
- **问题分析：** `if(Key_Down) Alarm_Flag = 0;` 这句关闭闹钟的代码被错误地嵌套在了 `if(Seg_Disp_Hour < 12) ... else { ... }` 的 else 分支内部，即只有当 `Seg_Disp_Hour >= 12`（下午时段）时按键才能关闭闹钟。如果闹钟在上午时段（Hour < 12）触发，代码根本执行不到关闭语句，按键无效
```c
// 错误：关闭逻辑嵌套在 else（下午）分支内
if(Seg_Disp_Hour < 12) {
// 上午 LED 逻辑...
} else {
// 下午 LED 逻辑...
if(Key_Down) Alarm_Flag = 0; // ← 只有下午才执行
}
```
- **解决方案：** 将 `if(Key_Down) Alarm_Flag = 0;` 移到 `if/else` 分支之外、`if(Alarm_Flag)` 块的末尾，确保无论 AM/PM 都能响应按键关闭闹钟
```c
if(Alarm_Flag) {
// AM/PM LED 逻辑...
if(Key_Down) Alarm_Flag = 0; // ← 移到这里，AM/PM 都能生效
}
```
### （12）数码管闪烁无法实现（Seg_Slow 刷新周期过长）
- **问题描述：** 数码管应实现闪烁效果，但实际始终常亮，看不到闪烁
- **问题背景：** 日常课程练习
- **问题分析：** 定时器中断中 `Seg_Slow` 的归零阈值设为了 500，即 `if(++Seg_Slow == 500) Seg_Slow = 0;`。定时器 1ms 中断，`Seg_Slow` 每 500ms 才归零一次，`Seg_Proc()` 开头 `if(Seg_Slow) return;` 导致数码管内容每 500ms 才刷新一次。而闪烁依赖 `LED_Flag`（250ms 翻转），数码管刷新周期远大于闪烁周期，`Seg_Buf` 中的闪烁值（在显示值和熄灭码 10 之间切换）无法被及时刷新到数码管上，视觉上看不到闪烁效果
- **解决方案：** 将 `Seg_Slow` 的归零阈值从 500 改为 10，即 `if(++Seg_Slow == 10) Seg_Slow = 0;`，使数码管每 10ms 刷新一次，远快于 250ms 的闪烁周期，闪烁效果即可正常显示
### （13）Key_Val/Key_Old 来回抖动（temp 未初始化 + Seg_Slow_Down 类型溢出）
- **问题描述：** 按下按键后 `Key_Val` 和 `Key_Old` 一直在 0 和按键序号之间来回跳变（如按1键，值在 0 和 1 之间反复抖动）
- **问题背景：** 日常课程练习（普中 A2 单片机）
- **问题分析：** 存在两处错误：
- [a] **temp 未初始化：** `Key_read()` 中 `unsigned char temp;` 没有赋初值 0。当没有按键按下时，四个 `if` 条件都不成立，`temp` 保持未初始化的随机值并返回，导致 `Key_Val` 不稳定，与 `Key_Old` 异或后产生虚假的 `Key_Down` 事件，表现为反复抖动
- [b] **Seg_Slow_Down 类型溢出：** `unsigned char Seg_Slow_Down;` 声明为 `unsigned char`（最大值 255），但中断中归零阈值设为 500。计数器永远到不了 500，`Seg_Slow_Down` 永远非零，`Seg_Proc()` 开头 `if(Seg_Slow_Down) return;` 始终成立，数码管刷新函数永远被跳过
- **解决方案：**
- [a] 将 `unsigned char temp;` 改为 `unsigned char temp = 0;`
- [b] 将 `Seg_Slow_Down` 类型改为 `unsigned int`，或将归零阈值改为 255 以内的合理值（如 10）
### （14）定时器中断周期异常导致倒计时速度不对
- **问题描述：** Proteus 仿真中数码管减速已设为 500ms，但倒计时跳变非常快；把减速值调到 1000 或 2000 后基本不跳了
- **问题背景：** 日常课程练习（仿真环境）
- **问题分析：** `Timer0Init()` 中漏敲了 `TMOD |= 0x01;` 这一行，`TMOD` 未被配置为模式 1（16 位定时器），Timer0 工作在默认的模式 0（13 位定时器），最大计数值只有 8192 而非 65536，初始值 `TH0=0xFC, TL0=0x18` 在 13 位模式下实际定时远小于预期的 1ms，导致中断频率大幅偏快，所有基于中断计数的时间都相应加快
- **解决方案：** 补上 `TMOD |= 0x01;`，将 Timer0 配置为模式 1（16 位定时器模式），中断周期恢复为正确的 1ms
### （15）按键误触发跳转界面（频率测量跳线帽未拔除）
- **问题描述：** 按下改变参数的按键时，界面会意外跳转到计数界面
- **问题背景：** 日常课程练习
- **问题分析：** 竞赛板上频率测量的跳线帽未拔除。该跳线帽将信号发生器输出连接到按键共用的 P3 口引脚，按键操作时电平变化通过跳线帽耦合到其他引脚，产生虚假的按键事件，触发了意外的界面跳转
- **解决方案：** 拔掉频率测量的跳线帽即可恢复正常。不使用频率测量功能时，应将对应跳线帽摘除，避免信号干扰按键扫描
### （16）EEPROM 掉电数据丢失（unsigned int 只写了 1 字节 + 局部变量未初始化）
- **问题描述：** EEPROM 存储参数后掉电重启，数据未正确恢复
- **问题背景：** 日常课程练习
- **问题分析：** 存在两处错误：
- [a] **写入字节数不足：** `Voltage_Compare` 声明为 `idata unsigned int`（2 字节），但 `EEPROM_Write(&Voltage_Compare, 0, 1)` 第三个参数写了 1，只写入了低字节（1 字节），高字节丢失。读取时 `EEPROM_Read(&Voltage_Compare, 0, 1)` 同样只读了 1 字节，高字节为随机值，导致恢复的数据与存入时不一致
- [b] **局部变量未初始化：** `EEPROM_Temp` 未初始化，校验判断时使用随机值，可能导致校验意外通过或失败
- **解决方案：**
- [a] `unsigned int` 占 2 字节，读写长度参数应改为 2：`EEPROM_Write(&Voltage_Compare, 0, 2)` 和 `EEPROM_Read(&Voltage_Compare, 0, 2)`
- [b] 局部变量声明时务必初始化：`unsigned char EEPROM_Temp = 0;`
### （17）流水灯一次性跑完无动画效果（for 循环误用）
- **问题描述：** 流水灯模式下 LED 没有逐个移动的效果，直接跳到最终状态后立即切换模式
- **问题背景：** 流水灯程序设计
- **问题分析：** `LED_Proc()` 中用 `for(i=1;i<8;i++)` 在单次函数调用中将所有 LED 状态一次性移完。循环体内 `Led_Buf[i]=1; Led_Buf[i-1]=0;` 连续执行 7 轮，中间没有延时，所有赋值在同一帧完成，视觉上只能看到最终状态（仅 `Led_Buf[7]=1`）。此外，循环结束后 `i` 必然等于 8，`if(i==8) Led_Mode=1;` 每次都成立，模式被立即切换，流水灯动画完全无法呈现
```c
// 错误写法：一次调用中 for 循环跑完所有步
for(i=1; i<8; i++)
{
Led_Buf[i] = 1;
Led_Buf[i-1] = 0; // 7 轮连续执行，只看到最终态
}
if(i == 8) Led_Mode = 1; // i 必然等于 8，模式立即切换
```
- **解决方案：** 引入 `Step_Count` 变量记录当前移动步数，每次定时到达时只执行一次数组左移操作（`Led_Buf[i]=Led_Buf[i+1]`），步数累计到 8 时才切换模式
```c
// 正确写法：每次调用只移一格
if(Timer >= Run_Switch_Time[Led_Mode])
{
Timer = 0;
for(i = 0; i < 7; i++)
Led_Buf[i] = Led_Buf[i+1]; // 数组整体左移一格
Led_Buf[7] = 0;
Step_Count++;
if(Step_Count >= 8)
{
Step_Count = 0;
Led_Mode = 1;
}
}
```
**核心理解：** 流水灯每一帧只移动一格，帧与帧之间靠定时器间隔产生动画效果，不能在一次调用中用循环完成所有帧
### （18）无效按键计数未按界面区分导致错误指示灯不亮（4T 测评 #21/#23）
- **问题描述：** 4T 测评得分 67，测评 #21（无效按键触发 3 次后 L3 应亮）和 #23（功能按键触发后 L1/L2/L3 应熄灭）始终未通过，错误状态指示灯逻辑异常
- **问题背景：** 第十一届省赛题目
- **问题分析：** `Key_Proc()` 中用一个 `switch(Key_Down)` 直接判断按键编号，S4/S5/S8/S9 不管在哪个界面按下都会清零 `Key_Error`。但题目要求的是：**按键在当前界面没有功能时，才算无效按键**。例如 S5 的功能是清零计数，仅在计数界面（界面3）有效；在数据监测界面（界面1）按 S5 应算一次无效按键，`Key_Error` 应 +1。但代码直接把 `Key_Error` 清零了，导致错误计数永远涨不到 3，L3 永远不会亮
```c
// 错误写法：不区分界面，所有按键都清零错误计数
switch(Key_Down)
{
case 4: Key_Error = 0; /* ... */ break;
case 5: Key_Error = 0; /* ... */ break;
// S4/S5/S8/S9 在任何界面按下都清零，无效计数永远累加不起来
}
```
- **解决方案：** 改为**先判断当前界面，再判断按键**的双层 switch 结构。每个界面只有该界面的有效按键才清零错误计数，其余按键走 `default` 分支累加 `Key_Error`
```c
// 正确写法：外层按界面分，内层按按键分
switch(Seg_Mode) // 先判断当前界面
{
case 0: // 界面0
switch(Key_Down)
{
case 4: Key_Error = 0; /* 界面0的有效操作 */ break;
// ...
default: Key_Error++; break; // 该界面无效的按键，累加错误
}
break;
case 2: // 计数界面
switch(Key_Down)
{
case 5: Key_Error = 0; /* 清零计数 */ break;
// ...
default: Key_Error++; break;
}
break;
}
```
