## 三、本周重点问题详解
### （1）S6/S7 按键缓存功能实现错误（编号写重、功能搞反、误用 EEPROM）
- **问题描述：** 学员不清楚按下 S6/S7 按键保存数据的正确写法，代码存在多处错误
- **问题背景：** 省赛题目，S6 为通道3电压"缓存"按键，S7 为频率"缓存"按键
- **问题分析：**
- [a] 按键编号写重：两个 `if` 都写成了 `Key_Down==7`，第一个应该是 `Key_Down==6`
- [b] S6/S7 功能搞反：S6 应存通道3电压，S7 应存频率，学员第一个 `if` 里存的是频率，应该存电压
- [c] 误用 EEPROM：题目说的"缓存"只是保存到内存变量中，用于和实时数据比较来控制 LED 亮灭，不是掉电保存，不需要写 EEPROM
- **解决方案：** 正确写法非常简洁，只需变量赋值：
```c
if(Key_Down == 6)
Vola_RB2_100x_Cahe = Vola_RB2_100x; // S6：缓存通道3电压
if(Key_Down == 7)
Freq_Cahe = Freq; // S7：缓存频率
```
缓存变量后续在 `Led_Proc()` 中用于比较：
```c
ucLed[0] = (Vola_RB2_100x > Vola_RB2_100x_Cahe); // L1：实时电压 > 缓存电压
ucLed[1] = (Freq > Freq_Cahe); // L2：实时频率 > 缓存频率
```
### （2）Led_Disp 锁存器冲突导致电压显示错误及 LED 状态异常（4处修复）
- **问题描述：** 4T 测评中通道3电压显示 2.50V（应为 3.50V）、LED 状态错误、LED 功能开关失效，共 4 个测试点未通过
- **问题背景：** 第十二届省赛（第二批）
- **问题分析：**
- [a] **Led_Disp 锁存器冲突（核心问题）：** `Led_Disp` 在主循环中调用，`Seg_Disp` 在定时器 ISR 中调用，两者共用 74HC573 锁存器。ISR 随时可能打断主循环的 `Led_Disp`，导致锁存器状态混乱，数码管段码串到 LED 上，电压显示偶发错误
- [b] **S7 长按逻辑错误：** `Key_Down==7 && Key_Up==7` 同时判断，但 `Key_Down`（按下沿）和 `Key_Up`（松开沿）是互斥事件，不可能同时为真，导致 LED 开关功能完全失效
- [c] **AD 地址多加 0x40 DAC 使能位：** `Ad_Read(0x43)` 中 0x40 为 DAC 使能位，纯 AD 采集不需要，属多余操作
- [d] **进入电压界面未重置通道：** 从周期界面按 S4 进入电压界面时未重置 `Seg_AD_Mode = 0`，不符合题目"每次进入电压界面均显示通道1"的要求
- **解决方案：**
- [a] 将 `Led_Disp` 移到定时器 ISR 中紧跟 `Seg_Disp` 之后调用，所有 573 锁存器操作集中在 ISR 中避免竞争
- [b] 长按检测改为"按下启动计时，松开时判断时长"：`Key_Down` 时启动计时标志，`Key_Up` 时判断 `Time_1_1s >= 1000` 则为长按
- [c] `0x43` 改为 `0x03`，`0x41` 改为 `0x01`，去掉 DAC 使能位；`51.0` 改为 `51`，避免浮点运算
- [d] 切入电压界面时添加 `Seg_AD_Mode = 0`
### （3）整数除法精度丢失导致 DA 输出电压卡在 1V
- **问题描述：** 模式1下温度≥20°C 时，DA 输出电压始终卡在 1V，不随温度线性变化
- **问题背景：** 第十二届省赛（第一批），DA 线性输出模式
- **问题分析：** `dat = (Temperature_10x * 3 / 200 - 2) * 51;` 中 `Temperature_10x * 3 / 200` 是整数除法，会截断小数。手动验算：20.0°C → `200*3/200=3`，21.0°C → `210*3/200=3`，25.0°C → `250*3/200=3`，26.6°C → `266*3/200=3`，结果在 20~26.6°C 范围内 `/200` 截断后始终为 3，`3-2=1`，`1*51=51`，电压恒为 1V。直到 26.7°C（`267*3/200=4`）才跳变到 2V
- **解决方案：** 调整运算顺序，先乘大再除，将 `*51` 提前合并：
```c
// 原来（精度丢失严重）
dat = (Temperature_10x * 3 / 200 - 2) * 51;
// 修改后（先乘大再除，保留精度）
dat = (long)Temperature_10x * 153 / 200 - 102;
// 等价于 (T*3*51)/200 - 2*51，加 (long) 防溢出
```
### （4）DAC 反算截断误差导致电压显示差 0.01V
- **问题描述：** 4T 测评 68.6/70 分，35 个测试点通过 34 个，#32 未通过：期望 A 3.10，实际显示 A 3.09
- **问题背景：** 第十二届省赛（第一批）
- **问题分析：** 学员先算 DAC 寄存器值 `dat`，再用 `DA_100x = dat * 100 / 51` 反算电压用于数码管显示。`dat * 100 / 51` 是整数除法反算，存在截断误差，DAC 寄存器的量化误差直接传递到了显示值上。正确做法应该"先算显示值，再量化输出"，显示值从源头直接计算，不经过反算
- **解决方案：** 改为先算电压值 `DA_100x`（直接精确），再正向量化成寄存器值：
```c
// 原来（先量化再反算 → 截断误差影响显示）
dat = 255;
Da_Write(dat);
DA_100x = dat * 100 / 51;
// 改后（先算电压再量化 → 显示精确）
DA_100x = 500;
dat = DA_100x * 51 / 100;
Da_Write(dat);
```
同时将温度精度从 ×10 改为 ×100 以匹配两位小数显示要求
### （5）初始化函数名拼写不一致导致板子无反应
- **问题描述：** 学员刚写完底层代码，在 `Led_Proc` 和 `Seg_Proc` 中写了测试代码验证数码管和 LED 显示，但板子上电后无任何反应
- **问题背景：** 底层搭建阶段，验证数码管和 LED 基本功能
- **问题分析：** 函数名拼写不一致导致链接时实际调用了错误的函数：
- `init.h` 声明的是 `Syetem_init()`（拼写错误：Syetem）
- `init.c` 定义的是 `System_Init()`（正确拼写但大小写不同）
- `main.c` 调用的是 `Syetem_init()`（与头文件声明匹配，但与定义不匹配）
Keil 链接器找不到 `Syetem_init()` 的实际定义，初始化函数未被正确执行，板子硬件未被正确配置，导致数码管和 LED 均无输出
- **解决方案：** 统一函数名拼写，将 `init.h` 声明、`init.c` 定义、`main.c` 调用三处的函数名改为一致的 `System_Init()`
### （6）底层驱动被修改 + 主逻辑多处 Bug 导致大面积丢分（13处修复）
- **问题描述：** 4T 测评原始得分约 30/62，时钟显示完全错乱、温湿度数据异常、LED 状态残留，多个测试点未通过
- **问题背景：** 第十四届省赛（第一批）
- **问题分析：**
- **底层驱动问题（致命）：**
- [a] DS1302 缺少 BCD 转换：直接写十进制数，芯片要求 BCD 编码，导致时钟显示错乱（如应显示 `13-46-50` 却显示 `19-70-85`）
- [b] DS1302 缺少中断保护：SDA 接 P2^3，ISR 中 `Seg_Disp` 也操作 P2，冲突导致 RTC 通信失败
- [c] DS18B20 缺少中断保护和转换等待：ISR 打断单总线时序导致读取失败，发送 0x44 后未等待就读取得到脏数据
- [d] IIC 与 DS1302 的 `sbit SDA` 命名冲突：两个文件定义同名 `sbit` 指向不同引脚（P2^1 vs P2^3），链接器随机选择导致通信失效
- **主逻辑问题：**
- [e] `Humidity_Get` 中多余的 `rd_temperature()` 调用，两次读取间隔不足导致脏数据
- [f] `Touch_Count` 在两个函数中重复自增，每次触发计数加2
- [g] `Tempera_Sum` 在两个函数中重复累加，平均温度翻倍
- [h] 清除数据时遗漏 `L5_Flag`、`Humi_Dat_Flag`、`L4_Blink_Flag` 标志位重置，LED 残留点亮
- [i] 温湿度上升判断 `Touch_Count >= 1` 应为 `>= 2`，首次采集无历史数据可比较
- [j] 湿度频率有效范围用 `>` `<` 应为 `>=` `<=`，边界值被排除
- [k] 湿度平均值小数位多除一次 10
- [l] Timer0 误开中断 `ET0=1`，计数器模式不需要中断
- [m] DS18B20 上电默认 85°C 未做预热等待
- **解决方案：**
- [a] 写入时十进制→BCD：`bcd = (val/10<<4) | (val%10)`；读取时 BCD→十进制：`val = (bcd>>4)*10 + (bcd&0x0f)`
- [b] `Read_Rtc` / `Set_Rtc` 前后加 `EA=0` / `EA=1` 中断保护
- [c] `rd_temperature` 前后加中断保护，0x44 转换命令后加延时等待
- [d] `iic.c` 中 `SDA` 改为 `sda`，避免与 `ds1302.c` 冲突
- [e] 删除 `Humidity_Get` 中多余的 `rd_temperature()` 调用，统一使用 `Temperature()` 的缓存值
- [f][g] 删除重复的自增和累加，保证一个数据源只在一个地方采集
- [h] 清除数据时补全所有标志位重置
- [i] 判断条件改为 `Touch_Count >= 2`
- [j] 边界条件改为 `freq_local >= 200 && freq_local <= 2000`
- [k] 去掉多余的 `/10`
- [l] 删除 `ET0=1`
- [m] `main()` 中加 `while(rd_temperature()==85);` 预热等待
### （7）seg_jiyi 在定时器 ISR 中被反复覆盖导致永远卡在模式3
- **问题描述：** 电压跌破阈值触发切换到采集界面（模式3）后，3 秒定时结束无法回到原界面，永远卡在模式3
- **问题背景：** AD 电压阈值触发界面切换功能，要求触发后显示采集界面 3 秒再恢复原界面
- **问题分析：** 学员将"记忆旧模式 + 切换新模式"的逻辑写在定时器 ISR 中（每 1ms 执行一次）：
```c
// 定时器 ISR 中（每1ms执行）
if(flag == 1) {
seg_jiyi = seg_mode; // 记忆当前模式
seg_mode = 3; // 切换到模式3
if(++cai_time >= 3000) {
flag = 0;
cai_time = 0;
seg_mode = seg_jiyi; // 恢复原模式
}
}
```
第1毫秒：`seg_jiyi = 0`（正确记忆），`seg_mode = 3`。第2毫秒：`seg_jiyi = seg_mode`，但此时 `seg_mode` 已经是 3，所以 `seg_jiyi` 被覆盖为 3。从第2毫秒起 `seg_jiyi` 就永远是 3，3 秒后 `seg_mode = seg_jiyi = 3`，模式无法恢复
- **解决方案：** 将记忆操作移到触发源（`ad_da` 函数）中，只在 `flag` 从 0 变为 1 的瞬间记忆一次：
```c
// ad_da() 中（触发源）
if((ad_cai1 < 250) && (ad_cai2 > 250)) {
if(flag == 0) { // 只在首次触发时记忆
seg_jiyi = seg_mode; // 记忆旧模式（此时还未被改）
seg_mode = 3; // 切换到模式3
flag = 1; // 开启3秒定时
}
}
// 定时器 ISR 中（只负责计时和恢复）
if(flag == 1) {
if(++cai_time >= 3000) {
flag = 0;
cai_time = 0;
seg_mode = seg_jiyi; // 恢复之前的模式
}
}
```
### （8）Da_Write 函数末尾缺少 I2CStop() 调用
- **问题描述：** 学员发现 `Da_Write` 函数末尾没有调用 `I2CStop()`，疑问为什么不写 Stop 也能正常工作
- **问题背景：** PCF8591 DAC 输出函数编写
- **问题分析：** PCF8591 的 DAC 输出是"写入即生效"的，数据字节发送完并收到 ACK 后，模拟电压就已经输出到 AOUT 引脚。根据 NXP 官方数据手册："In the write mode, a data transfer is terminated by sending either a stop condition or the start condition of the next data transfer." 即 Stop 不是必须的，下一次调用 `Da_Write` 或其他 I2C 函数时的 `I2CStart()`（Repeated Start）会隐式结束上一次通信。但不发 Stop 意味着总线一直被占用（SCL 保持低电平），如果总线上还有其他从设备则无法被访问
- **解决方案：** 在 `Da_Write` 末尾补上 `I2CStop()`，释放总线，养成良好的 I2C 编程习惯：
```c
void Da_Write(unsigned char dat) {
I2CStart();
I2CSendByte(0x90);
I2CWaitAck();
I2CSendByte(0x41);
I2CWaitAck();
I2CSendByte(dat);
I2CWaitAck();
I2CStop(); // 补上，释放总线
}
```
### （10）底层 Seg_Disp 操作 P2 时未保留低位，破坏 I2C 总线导致 AD 读数跳变
- **问题描述：** AD 读数在真实值和 5.00V 之间随机跳变，在 V4T 版本上没有此现象，移植到调度器架构后出现
- **问题背景：** 学员自己敲的 Seg_Disp / Led_Disp / Init 底层代码
- **问题分析：** PCF8591 的 I2C 数据线 SDA 和时钟线 SCL 接在 P2.0 和 P2.1，而数码管/LED 底层操作 P2 高位（选通 74HC573 锁存器）。学员底层代码写成 `temp = temp & 0x1f` 而非 `temp = P2 & 0x1f`，没有先读取 P2 当前值再保留低 5 位，而是对未初始化的局部变量 `temp` 做掩码。这导致写 P2 时低位（P2.0/P2.1）被随机值覆盖，正在进行的 I2C 通信时序被破坏，AD 读取偶发返回 0xFF（5.00V）。在 V4T 版本中由于代码执行时序不同，I2C 通信碰巧未被打断，所以没有暴露此问题
- **解决方案：** 底层操作 P2 高位时，必须先读取 P2 当前值保留低位：
```c
// 错误写法（temp 未初始化，低位被随机值覆盖）
temp = temp & 0x1f;
temp = temp | 0xe0;
P2 = temp;
// 正确写法（先读 P2，保留低5位不动）
temp = P2 & 0x1f;
temp = temp | 0xe0;
P2 = temp;
```
> **附注：** 凡是操作 P2 高位做锁存器选通的地方，都必须用 `P2 & 0x1f` 先保留低位，否则会破坏挂在 P2 低位上的 I2C 总线。
### （11）Keil C51 链接器大小写不敏感导致变量名冲突编译报错
- **问题描述：** 编译报错 `ERROR L104: MULTIPLE PUBLIC DEFINITIONS, SYMBOL: RT_WENDU`，无法生成目标文件
- **问题背景：** 学员在 `main.c` 中定义了全局变量 `rt_wendu`，同时另一个文件中有函数名 `RT_wendu`
- **问题分析：** 虽然 C 语言本身区分大小写，但 Keil C51 的链接器（BL51）在处理 OMF 目标文件格式时，对符号名**不区分大小写**，将 `rt_wendu` 和 `RT_wendu` 视为同一个符号，因此报"重复定义"错误
- **解决方案：** 将变量名或函数名改为完全不同的名称，避免仅靠大小写区分。例如将变量 `rt_wendu` 改为 `temperature_rt` 或其他不会与现有符号冲突的名称
> **附注：** 这是 Keil C51 特有的坑，标准 C 编译器（GCC、MSVC 等）不会有此问题。在 51 单片机开发中，命名时应避免仅靠大小写区分不同的变量/函数。
### （12）频率测量误读 TH1/TL1 导致 Timer1 被清零，按键迟钝、数码管流水
- **问题描述：** 按键响应极慢，数码管出现流水灯效果，界面切换也很慢
- **问题背景：** 频率测量功能，Timer0 做计数器、Timer1 做定时中断
- **问题分析：** 学员在 Timer1 的 ISR 中读频率时写成了 `Freq = (TH1<<8) | TL1`，随后又 `TH1 = TL1 = 0` 清零。但频率计数器用的是 Timer0（TH0/TL0），Timer1 是定时中断的定时器。读 TH1/TL1 读到的是 Timer1 自己的重装值（不是频率），清零 TH1/TL1 后 Timer1 的计数起点被改为 0，需要从 0 计到 65535 才能溢出触发下一次中断，周期从原来的 1ms 变成约 65ms。这导致：
- 数码管扫描频率从 1000Hz 降到约 15Hz，肉眼可见逐位扫描（流水灯效果）
- 按键扫描和调度器全部变慢，响应极其迟钝
- **解决方案：** 将 `TH1`/`TL1` 改为 `TH0`/`TL0`：
```c
// 错误
Freq = (TH1 << 8) | TL1;
TH1 = TL1 = 0;
// 正确
Freq = (TH0 << 8) | TL0;
TH0 = TL0 = 0;
```
### （13）按键编码不统一 + 设置/比赛参数未分离 + 计时残值未清零（3类问题）
- **问题描述：** 设置界面时间参数无法修改（始终显示 U05）、比赛界面显示值错误（应 A07 显示 A04）、倒计时提前结束直接显示 EEE
- **问题背景：** 第十七届模拟题3（按键+状态机+倒计时题目）
- **问题分析：**
- [a] **按键编码不统一：** 底层按键扫描返回值与上层判断使用的编码方式不一致，导致设置界面按键无效、比赛开始条件判断错误、淘汰选手编号判断错误
- [b] **设置参数与比赛参数未分离：** 设置界面修改的参数和比赛界面使用的参数共用同一个变量，比赛倒计时直接递减设置值，导致下一轮比赛初值异常，且设置界面显示被改坏

### （9）AD_Read 缺少 I2CWaitAck() 导致电压读取卡在 5.00V
- **问题描述：** 学员二刷题目时改用官方赛点资源包的 IIC 文件，RB2 电压读取始终卡在 5.00V，但之前用课程提供的 IIC 文件没有问题

```c
I2CSendByte(0x91);
```

### （14）switch-default 无按键时误触发导致 L3 常亮 + EEPROM 保存时机错误（2处修复）
- **问题描述：** 4T 测评中 9 个测试点未通过，其中 7 个为指示灯状态错误（#13、#15、#18、#19、#20、#21、#23），2 个为 EEPROM 存储值错误（#11、#25）。学员反馈"LED 模块的代码检查了几次都没找到问题，底层模块也没问题"
- **问题背景：** 第十一届省赛，涉及电压阈值计数、LED 错误指示、EEPROM 参数保存
- **问题分析：**
- [a] **switch-default 在无按键时被反复触发（核心问题）：** 按键扫描每 10ms 执行一次，无按键按下时 `Key_Down = 0`。`Key_Down = 0` 进入 `switch` 后不匹配 12、13、16、17 中的任何一个 `case`，于是落入 `default` 分支执行 `Error_Key++`。仅 30ms 后 `Error_Key >= 3`，触发 `ucLed[2] = 1`，L3（错误指示灯）几乎永远亮着。学员在 LED 模块中反复查找问题，但根因不在 LED 模块，而在按键处理的 `default` 分支
- [b] **EEPROM 保存时机错误：** 学员在 S12 按键处理中先判断 `Seg_Disp_Mode == 2` 再 `++Seg_Disp_Mode`，意图是"离开计数界面时保存"。但题目要求的保存时机是"离开参数界面、进入计数界面时保存"。正确逻辑应先 `++Seg_Disp_Mode`，再判断切换后 `== 2` 则保存。测试点 #25 期望 EEPROM 值为 25，实际读到 15（上一次的旧值），正是因为保存发生在错误的时机
- **解决方案：**
- [a] 在 `switch` 前加 `if(Key_Down == 0) return;`，排除无按键状态，避免 `default` 被误触发：
```c
if(Key_Down == 0) return; // 没有按键按下，直接返回
switch(Key_Down)
{
case 12: ... break;
case 13: ... break;
case 16: ... break;
case 17: ... break;
default:
Error_Key++;
break;
}
```
- [b] 将模式切换放到 EEPROM 保存判断之前，"先切换，再判断是否需要保存"：
```c
case 12:
Error_Key = 0;
if(++Seg_Disp_Mode == 3) Seg_Disp_Mode = 0; // 先切换模式
if(Seg_Disp_Mode == 2) // 再判断：切换后是否进入计数界面
{
V_Pra_10 = V_Pra * 10;
EEPROM_Write(&V_Pra_10, 0, 1); // 此时保存的是刚修改完的新参数
}
```
> **附注：** 本例中学员反复检查 LED 模块和底层代码均未找到问题，是因为根因不在 LED 本身——LED 模块忠实地执行了 `ucLed[2] = (Error_Key >= 3)` 的逻辑，问题出在上游按键处理中 `Error_Key` 被错误累加。排查"显示/输出异常"时，除了检查显示模块本身，还要沿数据流向上追溯到数据源头。
### （15）超声波忙等循环阻塞主循环导致数码管在 000 和 RB2 值之间跳变
- **问题描述：** 模式1下数码管显示在 `000` 和 RB2 实际电压值之间反复跳变
- **问题背景：** 题目同时使用超声波测距和 AD 电压采集，调度器架构
- **问题分析：** `Get_Distance()` 调用 `Ut_Wave_Data()`，其内部有一个忙等循环 `while((US_RX==1)&&(CF==0));`，等待超声波回波或 PCA 溢出。如果没有回波返回（传感器未接好或超出量程），该循环会阻塞主循环长达 **65ms**（直到 PCA 计数器溢出）。在调度器架构下，超声波测距约每 120ms 调用一次，其中约 65ms 被忙等循环占用，导致：
- AD 采集函数在阻塞期间无法执行，采到的值为 0 或过期数据
- 数码管刷新被中断，显示不连贯
- 交替出现正常值（非阻塞期采到的 RB2 电压）和 `000`（阻塞期未更新的数据）
- **解决方案：** 将超声波测距的忙等循环改为非阻塞方式，利用定时器中断做超时判断，避免长时间占用主循环：
```c
// 原来（阻塞式，最长卡65ms）
void Ut_Wave_Data() {
// ... 发送触发脉冲 ...
while((US_RX == 1) && (CF == 0)); // 死等回波，阻塞主循环
// ... 计算距离 ...
}
// 改进方案：加超时计数保护
void Ut_Wave_Data() {
unsigned int timeout = 0;
// ... 发送触发脉冲 ...
while((US_RX == 1) && (CF == 0)) {
if(++timeout > 10000) break; // 超时跳出，避免长时间阻塞
}
// ... 计算距离 ...
}
```
或者将超声波测距放到独立的定时窗口中执行，与 AD 采集和数码管刷新在时间上错开，确保互不干扰。
> **附注：** 在调度器架构中，任何一个任务函数中的长时间阻塞都会影响所有其他任务的执行。超声波的忙等循环是典型的"一个阻塞拖垮全局"问题。凡是包含 `while` 等待硬件响应的驱动函数，都应加上超时保护或改为非阻塞设计。
### （16）按键底层多写 P3 &= 0xef 导致 T0 计数输入被拉低，频率恒为 0
- **问题描述：** 数码管肉眼观察显示正常，但 4T 测评全部错误：频率界面始终显示 `F 0`，周期界面显示 `N4967295`（溢出值）
- **问题背景：** 频率/周期测量题目，Timer0 做外部计数器
- **问题分析：** 按键扫描底层函数末尾多了一行 `P3 &= 0xef;`。`0xef = 1110 1111b`，该操作将 **P3.4 强制拉低**。P3.4 正是 T0（Timer0 外部计数输入）引脚，每次按键扫描（每 10ms）都会将 T0 输入拉低一次，外部频率信号无法正常进入计数器，Timer0 计数值始终为 0：
- 频率 = 0 → 频率界面显示 `F 0`
- 周期 = 1000000 / 0 → 整数除零溢出，显示 `N4967295`
学员肉眼测试时，因为没有外接信号源或不关注频率值，误认为"显示正常"，直到 4T 自动测评注入频率信号后才暴露问题
- **解决方案：** 删除按键底层中多余的 `P3 &= 0xef;`：
```c
// 修改前（P3.4 被拉低，破坏 T0 计数输入）
P3 &= 0xef; // ← 删除此行
return temp;
}
// 修改后
return temp;
}
```
> **附注：** P3 口在 STC15 上承担多种复用功能（T0、T1、INT0、INT1、RXD、TXD 等），按键扫描操作 P3 时必须只动按键相关的引脚位，不能用掩码误伤其他功能引脚。写底层时建议逐位操作或仔细核对掩码值，避免影响同端口的其他外设。
### （17）徘徊判定用错变量 + 距离参数未生效 + 温度采集间隔过短 + 超声波延时参数错误（6处修复）
- **问题描述：** 4T 测评 57/71，继电器能计次数但记不准，徘徊状态无法触发，修改距离参数后不生效。学员多次自查和使用 AI 辅助排查仍未定位到核心问题
- **问题背景：** 第十六届省赛，涉及超声波测距、运动状态判定（静止/徘徊/跑动）、继电器控制、温度采集
- **问题分析：**

- [a] 将 `Distance < 10` 改为 `D_Distance < 10`：

```c
else if(D_Distance >= 5 && Distance < 10)
Temp_Motion_State = 1;
// 修改后（用差值判定徘徊）
else if(D_Distance >= 5 && D_Distance < 10)
Temp_Motion_State = 1;
```
- [b] 将 `Dist_Para` 改为 `Dist_Para_Set`：
```c
Approached = (Distance < Dist_Para_Set); // 用被按键实际修改的变量
```

### （18）继电器计数用局部变量 + 运动状态编码初始值不匹配 + 高温判定边界错误 + L8 锁定范围错误（7处修复）

### （19）Led_Disp 中 if 语句末尾多敲分号导致 LED 持续闪烁
- **问题描述：** 学员刚开始搭建底层，LED 持续闪烁，且仅在数码管有数显示时闪
- **问题背景：** 底层搭建阶段，验证 LED 基本功能
- **问题分析：** `Led_Disp` 函数第 12 行 `if(temp_1!=temp_1_Old);` 末尾多了一个**分号**。C 语言中 `if(...);` 等价于 `if(...) { }` —— 空语句，条件判断的结果被丢弃，后面的花括号 `{ ... }` 变成了**无条件执行**的代码块。这意味着：
- 无论 `temp_1` 是否变化，每次调用 `Led_Disp` 都会执行 P0 写入和 573 锁存操作
- `Led_Disp` 在定时器 ISR 中与 `Seg_Disp` 交替执行，每次都重新锁存 LED 数据到 P0
- 当数码管正在扫描时，`Seg_Disp` 往 P0 写段码，紧接着 `Led_Disp` 又往 P0 写 LED 数据并锁存，两者交替导致 LED 在段码数据和 LED 数据之间快速切换，表现为闪烁
- 当数码管无数据（全灭）时，P0 段码为 0xFF，LED 恰好也是全灭，闪烁不明显；有数时段码 ≠ LED 数据，闪烁明显
原本 `if(temp_1!=temp_1_Old)` 的优化意图是"LED 状态没变就不操作 573"，加了分号后这个优化完全失效
- **解决方案：** 删除 `if` 语句末尾多余的分号：
```c
// 修改前（分号使 if 判断变成空语句，花括号无条件执行）
if(temp_1!=temp_1_Old);
{
P0=~temp_1;
// ... 573 锁存操作 ...
}
// 修改后（去掉分号，花括号正确受 if 控制）
if(temp_1!=temp_1_Old)
{
P0=~temp_1;
// ... 573 锁存操作 ...
}
```
> **附注：** `if(...);` 是 C 语言中极其隐蔽的错误，编译器不会报错也不会警告，但语义完全改变。建议养成习惯：写完 `if` / `while` / `for` 后立刻写花括号，不要先打分号。

### （23）温度减速变量 unsigned char 装不下 300，条件永真导致温度永不采集
- **问题描述：** 学员照着课程讲解做完第十五届省赛（第二批），温度显示界面下无法显示环境温度，问了多个 AI 均未能解决

```c
idata unsigned int temperture_slow_down;
```

### （25）Freq_Max 更新逻辑放在 Seg_Proc 中且记录时间重读 RTC，导致回显时间锁定在初始值（2处修复）
- **问题描述：** 4T 测评 #34 切换时间回显界面错误，期望显示 HA133445，实际显示 HA235950
- **问题背景：** 第十五届省赛（第一批），涉及频率最大值记录及对应时间回显
- **问题分析：**
- [a] **记录时间时重新调用 Read_Rtc() 读 DS1302（核心问题）：** 学员在检测到新最大值时调用 `Read_Rtc(ucRtc_Max_Para)` 重新读一次 DS1302，而不是直接拷贝已在调度器中定期读好的 `ucRtc` 数组。开机瞬间频率一来（任何非零值）就比 `Freq_Max`（初始值 0）大，立刻触发 `Read_Rtc`，此时 RTC 刚被设置为 23:59:50（初始化值），所以 `ucRtc_Max_Para` 被锁定为 23:59:50。后续频率没有再突破最大值，时间再也没有更新过，回显界面始终显示 HA235950
```c
// 学员写法（重新读 DS1302，读到的是初始化瞬间的值）
if(Freq > Freq_Max && Freq > 0)
{
Read_Rtc(ucRtc_Max_Para); // 重新读一次 DS1302
Freq_Max = Freq;
}
```
- [b] **Freq_Max 更新逻辑放在 Seg_Proc() 中：** 频率最大值判断和时间记录属于数据处理逻辑，不应混在显示函数 `Seg_Proc()` 中。频率计算和最大值判断应该和显示分离，否则数据处理的执行时机受显示刷新节奏影响，增加不可控因素
- **解决方案：**
- [a] 记录时间改为直接拷贝已缓存的 `ucRtc` 数组，不再重新读 DS1302：
```c
// 正确写法（直接拷贝已经读好的 ucRtc 数组）
if(Freq > Freq_Max && Freq > 0)
{
ucRtc_Max_Para[0] = ucRtc[0]; // 直接拷贝已读好的时间
ucRtc_Max_Para[1] = ucRtc[1];
ucRtc_Max_Para[2] = ucRtc[2];
Freq_Max = Freq;
}
```
- [b] 将这段逻辑从 `Seg_Proc()` 中分离出来，单独做一个 `Freq_Proc()` 放进调度器，职责分离——显示函数只负责把现成数据搬到数码管上
> **附注：** 本例暴露了两个设计原则问题：① **职责分离**——数据处理（频率最大值追踪）不应混在显示函数中，应独立为 `Freq_Proc()` 并纳入调度器；② **数据源一致性**——需要时间快照时，应拷贝调度器中已经定期刷新的 `ucRtc` 缓存数组，而非重新调用 `Read_Rtc()` 读硬件。重新读硬件的问题在于读取时刻不可控，尤其在初始化阶段 RTC 刚被写入初始值时，读到的是初始值而非真实运行时间。
### （26）AUXR 操作符写错 + Ctrl_Proc 调度周期过长 + S4 缺少运行模式处理 + idata 溢出，导致界面切换失灵和卡死在运行模式（6处修复）
- **问题描述：** 学员反馈代码逻辑跟满分代码基本一致了，但仍有阻塞现象——要么切换不了界面，要么卡死在运行模式
- **问题背景：** 第十五届国赛，涉及运动控制（坐标计算、速度模拟）、多界面切换、串口接收坐标、参数调节
- **问题分析：**
- [a] **Timer0 初始化 AUXR 操作符写错（致命）：** `AUXR &= 0x80` 用的是与操作，会**清除 bit0~bit6 只保留 bit7**。若 AUXR 默认为 0，`0 & 0x80 = 0`，bit7 也没被置位，Timer0 根本没进入 1T 模式。更严重的是该语句会把 AUXR 其他位全部清零，可能影响串口等后续初始化的外设。正确写法是 `AUXR |= 0x80`（或操作置位 bit7）
- [b] **Ctrl_Proc 调度周期太长（致命）：** 运动控制函数 `Ctrl_Proc` 调度周期设为 20ms，满分代码为 1ms。Dt 值大了 20 倍，位置更新频率过低，浮点运算累积误差更大，运动模拟出现异常，可能无法正确到达目的地
- [c] **S4 按键缺少运行模式的处理——卡死根因（致命）：** S4 只处理了空闲（`Status==0`）→运行 和 等待（`Status==2`）→运行 两种情况，**完全没有处理 `Status==1`（运行中）的情况**。满分代码在运行模式下按 S4 可手动切到等待模式。学员代码一旦进入运行模式，如果速度为 0（没有频率输入且 B=0），设备永远不会到达目的地也不会自动切换状态，就彻底卡死了
- [d] **S9/S12/S13 按键没有限制在参数界面（严重）：** S9、S12、S13 是顶层 `case`，任何界面下按都会生效。在位置或速度界面按 S12/S13 时会悄悄修改参数，切回参数界面时值已变，引起不可预期的行为
- [e] **进入运行模式时没有清除标志位（中等）：** 未清除 `Get_Coordinate_Flag` 和 `Reach_Flag`。不清除 `Get_Coordinate_Flag` 的后果是回到空闲状态后按 S4 会立刻用旧的目的地坐标再次进入运行模式
- [f] **多余的 idata 变量浪费宝贵内存（中等）：** 声明了 `temp_Speed`（float, 4字节）、`Key_Down_data`、`Key_Up_data`、`AD_3_Data_10x` 等从未使用的 `idata` 变量。8051 的 idata 空间只有 256 字节（还要跟寄存器组、栈共享），多余变量挤压栈空间，当 `sqrt()`、`sscanf()` 等库函数调用较深时可能导致**栈溢出**，表现为程序随机卡死、变量值莫名被篡改、界面切换失灵
- **解决方案：**
- [a] `AUXR &= 0x80` 改为 `AUXR |= 0x80`
- [b] `Ctrl_Proc` 调度周期从 20ms 改为 1ms：
```c
// 修改前
{Ctrl_Proc, 20, 0},
// 修改后
{Ctrl_Proc, 1, 0},
```
- [c] S4 按键补充运行模式处理：
```c
case 4:
if(Status == 0 && Get_Coordinate_Flag)
{
Status = 1;
First_Flag = 0;
Reach_Flag = 0;
Get_Coordinate_Flag = 0;
}
else if(Status == 1) // ← 新增：运行中按S4切到等待
{
Status = 2;
}
else if(Status == 2 && Clear_Rubbish_Flag)
{
Status = 1;
First_Flag = 0;
}
break;
```
- [d] S9/S12/S13 加 `Seg_Mode == 2` 前置判断，限制只在参数界面生效
- [e] 进入运行模式时补充清零 `Get_Coordinate_Flag = 0; Reach_Flag = 0;`
- [f] 删除所有未使用的 idata 变量，释放内存空间
> **附注：** 本例的根因结论：**"卡死在运行模式"** = 问题 [c]（S4 没有运行→等待的处理）+ 问题 [b]（Ctrl_Proc 周期太长导致运动异常无法到达目的地）；**"切换不了界面"** = 问题 [f]（idata 内存溢出导致栈破坏，程序行为不可预测）+ 问题 [d]（参数按键不受界面限制，可能污染状态）。学员认为"代码逻辑基本一致"时往往只关注算法流程是否相同，而忽略了操作符（`&=` vs `|=`）、调度参数（20ms vs 1ms）、状态机完整性（缺少一个状态分支）、内存管理（idata 溢出）这些"细节"——但在嵌入式开发中，这些细节决定程序能否正常运行。
