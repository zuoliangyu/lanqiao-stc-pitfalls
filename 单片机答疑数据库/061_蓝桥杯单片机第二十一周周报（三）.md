## 三、本周重点问题详解
### （1）Key_Read() 变量未初始化导致按键失灵
- **问题描述：** 上电后板子在模式0-1之间不自主切换，按键也失灵
- **问题背景：** 第十一届省赛题目
- **问题分析：** `Key_Read()` 函数中 `unsigned char temp;` 未初始化为 0，无按键按下时 `temp` 返回随机值，导致虚假按键事件触发模式切换
- **解决方案：** 将 `unsigned char temp;` 改为 `unsigned char temp = 0;`
### （2）流水灯移位结果未保存导致LED卡住
- **问题描述：** da写入电压值只写入整数
- **问题背景：**da_write()使用
- **问题分析：** `da_write_100x / 100 * 51;` 精度丢失，结果只剩整数
- **解决方案：** 将结果赋值回变量，改为 `da_write_100x * 51 / 100;` 先乘后除
### （3）Seg_Pos 赋值写成比较运算导致LED闪烁
- **问题描述：** Proteus 仿真中 LED 一直在闪烁
- **问题背景：** 数码管动态扫描
- **问题分析：** 定时器中断中 `Seg_Pos == 0;`（双等号，比较运算）不会改变 `Seg_Pos` 的值，导致 `Seg_Pos` 无法归零、数组越界、数码管扫描异常
- **解决方案：** 将 `Seg_Pos == 0;` 改为 `Seg_Pos = 0;`（单等号赋值）
### （4）频率测量 Freq 始终为 0（V4T项目，5处修复）
- **问题描述：** 频率无法输出，Freq 始终为 0
- **问题背景：** V4T 项目（频率测量 + 超声波）
- **问题分析：**
- [a] `Key.c` 中 `P3=P3|0xEF` 导致 P3.4（Timer0 计数输入引脚）被锁死为低电平，计数器失效
- [b] `Key.c` 中 `temp` 未初始化，无按键时返回随机值
- [c] `Key.c` 中 P32 重复检测 / P30 漏检（每行第4个按键都写成了 P32）
- [d] 读频时未停止计数器，读 TH0/TL0 之间可能产生进位竞争
- [e] `Seg_Pos` 未定义行为：`Seg_Pos=(++Seg_Pos)%8` 同一表达式两次修改同一变量
- **解决方案：**
- [a] `0xEF` 改为 `0xFF`，确保 P3.4 锁存器为高阻输入
- [b] `unsigned char temp` 改为 `unsigned char temp = 0`
- [c] 将 `if(P32==0) temp=7/11/15` 改为 `if(P30==0)`
- [d] 读取前 `TR0=0`，读取后 `TR0=1`
- [e] 改为 `Seg_Pos = (Seg_Pos + 1) % 8`
> **附注：** Keil5 Debug 仿真中 Freq/Distance = 0 属正常现象。Timer0 为计数器模式，Keil 软件仿真不产生外部信号；高优化等级下编译器会优化掉只写不读的变量，加 `volatile` 或降低优化等级可解决。必须烧录实物板测试。
### （5）数码管高位熄灭（前导零消隐）无法实现
- **问题描述：** 数码管高位熄灭功能不生效，高位始终显示 0 无法消隐
- **问题背景：** 频率显示数码管处理
- **问题分析：** 高位消隐循环中 `while(Seg_Buf[i]==0)` 将 `Seg_Buf[i]` 赋值为 10（熄灭码）。但循环变量 `i` 从 1 开始，第一次执行时 `Seg_Buf[1]` 被设为 10，下次再调用 `Seg_Task()` 时，`Seg_Buf[1]` 的值已经是 10（不是 0），`while` 条件立即不成立，直接跳出循环，后续高位全部无法消隐
- **解决方案：** 确保高位消隐逻辑在每次重新赋值 `Seg_Buf` 之后再执行，且循环起始索引应从被赋值数码管的最高位开始向低位遍历，保证每次消隐前 `Seg_Buf` 中存放的是本轮最新的数值而非上轮的熄灭码
### （6）EEPROM 校验机制导致自动评测参数读取失败
- **问题描述：** 自动评测中 EEPROM 参数未被正确读取，但手动测试时恰好通过
- **问题背景：** 自动评测系统测试
- **问题分析：** 代码中添加了自定义校验机制：
```c
EEPROM_Read(&EEPROM_Temp, 8, 1); // 读地址8的校验值
if(EEPROM_Temp == EEPROM_D) // EEPROM_D = 5，只有等于5才读参数
EEPROM_Read(&Ad_10x_para, 0, 1);
```
但评测系统只在地址 0 写入了值 10，并未在地址 8 写入校验字节 5，导致 `EEPROM_Temp != 5`，校验失败，`Ad_10x_para` 未从 EEPROM 读取。手动测试能通过是因为代码中初始值恰好也是 10：
```c
unsigned char Ad_10x_para = 10; // 初始值碰巧与EEPROM中一致
// VP = 10 × 0.1 = 1.00V，结果恰好正确
```
- **解决方案：** 去掉自定义校验机制，直接按评测系统约定的地址读取 EEPROM 参数，不要添加评测系统未预期的额外校验逻辑
