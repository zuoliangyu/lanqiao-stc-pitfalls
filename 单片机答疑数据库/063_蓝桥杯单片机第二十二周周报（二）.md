## 三、本周重点问题详解
### （1）unsigned char 数据类型溢出（多届重复出现）
- **问题描述：** 电压最多显示2.4几 / DAC显示2.44而非5.00 / 数码管显示比设置值多0.04
- **问题背景：** 第10届（电压）、第12届（DAC值、温度显示）
- **问题分析：** `unsigned char` 最大存储255，当计算结果超过255时发生溢出截断：
```c
// 第10届：Voltage_100x 为 unsigned char
Voltage_100x = temp * 100 / 51; // temp=255时结果500，溢出
// 第12届：DAC_Out_100x 为 unsigned char
DAC_Out_100x = 500; // 500 % 256 = 244，显示2.44
// 第12届：(unsigned char)强转
Seg_Buf[7] = (unsigned char)(DA_Value * 100) % 10; // 310截断为54，%10=4
```
- **解决方案：** 将变量类型从 `unsigned char` 改为 `unsigned int`，或将强转类型改为 `(unsigned int)`
> **规律总结：** 凡是涉及乘法运算（×100、×51等）的中间结果，必须检查是否超过变量类型上限。这是本周出现频率最高的Bug类型（4次）。
### （2）中断时序冲突导致通信失败
- **问题描述：** 温度读取失效 / DS1302数据错误
- **问题背景：** 第13届省赛（调度器顺序问题）、第14届省赛（ISR浮点运算）
- **问题分析：**
- DS18B20 OneWire协议的位读取只有4μs采样窗口，1ms定时器中断（ISR执行50-100μs）落入该窗口时采样报废
- DS1302通信通过P2.3(SDA)进行，ISR中数码管扫描的 `P2 = temp | 0xe0` 会篡改P2.3状态
- ISR中执行浮点运算（`0.0444*Freq+1.1111`）耗时800-1300μs，超过1ms定时器周期，主循环几乎得不到CPU时间
```c
// ISR中浮点运算——致命！
shidu_j[chufa_num] = (unsigned int)((0.0444 * Freq + 1.1111) * 10 + 0.5);
// 8051无FPU，此运算耗时 > 1ms定时器周期
```
- **解决方案：**
- OneWire/DS1302通信前关中断：`EA = 0;` ... 通信 ... `EA = 1;`
- 将浮点运算移出ISR，改用整数运算：`44UL * Freq + 1111) / 100` 或 `(Freq - 200) * 4 / 9 + 100`
> **规律总结：** 8051单片机ISR中绝对不要做浮点运算；OneWire、DS1302等时序敏感通信必须加关中断保护。
### （3）BCD码与十进制混淆
- **问题描述：** EEPROM存入的时钟时和分数据错误（16时存成22，47分存成71）
- **问题背景：** 第15届省赛
- **问题分析：** DS1302寄存器返回BCD格式数据（0x16表示16时，字节值=22），代码未做转换直接存入EEPROM，评测器按十进制解释读到22和71
```
DS1302寄存器(BCD) → ucRtc[](BCD) → Timer_Record[](BCD) → EEPROM(BCD)
评测器期望：16、47（十进制）
实际存入：22、71（BCD原始字节值）
```
- **解决方案：** 写入EEPROM时做BCD→十进制转换：
```c
EEPROM_Write_Dat[0] = Timer_Record[0] / 16 * 10 + Timer_Record[0] % 16;
EEPROM_Write_Dat[1] = Timer_Record[1] / 16 * 10 + Timer_Record[1] % 16;
```
> **附注：** 题目中"HEX编码写入"仅针对输入数据（地址2和3），时间数据（地址0和1）存实际十进制数值即可。
### （4）有符号/无符号类型隐式提升
- **问题描述：** 频率校准值为负数时，DAC输出5V而非0V
- **问题背景：** 第15届省赛（频率校准功能）
- **问题分析：** `Freq`(unsigned int) + `Correct_Para`(int) 运算时，int被隐式提升为unsigned int，负数变为巨大正数：
```c
Freq_Last = Freq + Correct_Para;
// Freq=100, Correct_Para=-200
// 预期：-100（应输出0V）
// 实际：-200 → unsigned int = 65336，100+65336 = 65436（巨大正数→输出5V）
```
- **解决方案：** 强制转为long再运算：`Freq_Last = (long)Freq + Correct_Para;`
### （5）P3.4引脚复用冲突
- **问题描述：** 频率测量始终显示约91，与实际输入频率无关
- **问题背景：** 第15届省赛
- **问题分析：** P3.4同时被用于Timer0计数器输入（接NE555频率信号）和矩阵键盘第四行行线。按键扫描每10ms执行一次将P3.4拉低再恢复，Timer0将这些跳变当作外部脉冲计数
- **解决方案：** 注释掉Key.c中第四行扫描代码（S16-S19），本题只用S4/S5/S8/S9，不在第四行
> **补充提醒：** 学员在使用最后一列按键（S4/S8/S12/S16）时，经常忘记将跳线帽从P3.4上拔掉。P3.4既是矩阵键盘第四行行线，也是Timer0计数器的外部输入引脚（T0）。如果题目涉及频率测量（Timer0计数器模式），**必须拔掉P3.4跳线帽**并注释第四行扫描代码，否则按键扫描产生的电平跳变会被计数器误计。这是一个硬件层面的易忘操作，建议作为赛前检查清单的固定项。
### （6）竞态条件导致长按检测失败
- **问题描述：** S7长按有时没反应
- **问题背景：** 第12届省赛
- **问题分析：** 松键时先清 `key_press_long = 0`，ISR看到标志为0立即清 `time_1000 = 0`，随后检查 `time_1000 >= 1000` 已经是0
- **解决方案：** 清标志前先保存计时值到局部变量：
```c
unsigned int t = time_1000; // 先保存
key_press_long = 0; // 再清标志
if(t >= 1000) // 用保存的值判断
led_off_flag ^= 1;
```
### （7）组合按键判断位置错误
- **问题描述：** S8和S9组合按键清空继电器计数不成功
- **问题背景：** 第16届省赛
- **问题分析：** 在 `Key_Read()` 矩阵键盘扫描函数中，S8+S9的组合按键判断 `if(P30 == 0 && P32 == 0) temp = 89;` 被放在了该列单键扫描的**中间位置**。扫描时代码从上往下执行，S8（`if(P30 == 0) temp = 9;`）先被匹配到并赋值 `temp = 9`，后续的组合判断虽然条件也满足，但已经晚了一步（或者根据具体代码结构，组合判断可能被跳过）
```c
// 错误写法：组合判断夹在单键扫描中间
if(P30 == 0) temp = 9;
if(P30 == 0 && P32 == 0) temp = 89; // ← 放在这里
if(P30 == 0) temp = 11;
// ...
// 正确写法：组合判断放在该列扫描的最后
if(P30 == 0) temp = 9;
if(P30 == 0) temp = 11;
// ...其他单键扫描
if(P30 == 0 && P32 == 0) temp = 89; // ← 放最后，覆盖单键值
```
- **解决方案：** 将组合按键判断移到该列所有单键扫描的**最后面**，利用后赋值覆盖前赋值的特性，确保两键同时按下时 `temp` 最终被赋为组合键值89
> **规律总结：** 矩阵键盘中实现组合按键时，组合判断必须放在单键判断之后，利用"后覆盖"机制保证组合键优先级高于单键。

## 五、经验教训与注意事项
1. **数据类型选择：** 涉及乘法运算的变量，先计算最大可能结果再选类型。`unsigned char`(0-255)、`unsigned int`(0-65535)、`unsigned long`(0-4294967295)
2. **ISR纪律：** ISR中禁止浮点运算、禁止长耗时操作；OneWire/DS1302等时序敏感通信必须加 `EA=0/EA=1` 保护
3. **BCD转换意识：** DS1302返回BCD码，存入EEPROM或参与十进制运算前必须转换（`/16*10 + %16`）
4. **类型提升警惕：** `unsigned` 和 `signed` 混合运算时，signed会被提升为unsigned，负数变巨大正数。使用 `(long)` 强转避免
5. **引脚复用检查：** 使用Timer0计数器模式（P3.4输入）时，检查矩阵键盘是否占用同一引脚
6. **组合按键顺序：** 矩阵键盘扫描中，组合按键判断必须放在单键判断之后（利用后覆盖机制），否则单键先匹配导致组合键失效
