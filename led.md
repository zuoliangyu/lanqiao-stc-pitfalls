## 坑 1：Timer2 中断中使用 32 位取模导致软件 I2C 时序崩溃

> 第十三届国赛 · 发现者：左岚

### 现象

与 Timer2 中断共用软件 I2C 总线的传感器（如 AHT10）读数异常：输出为 0，或数值随机乱跳，但代码逻辑看起来没有问题。

### 原因

Timer2 每 100μs 中断一次用于 PWM 计数，中断里用 `unsigned long int` 做 32 位取模，8051 需调用库函数，耗时是 `unsigned char` 自增比较的 **10 倍以上**。中断频繁打断软件模拟 I2C 的 SDA/SCL 操作，破坏位时序，导致从机误判：

```c
// 错误写法：32 位取模在 8051 上调用库函数，中断耗时大幅增加
idata unsigned long int pwm;

// ISR 内
pwm = (++pwm) % 10;
```

### 解决方法

换成 `unsigned char`，用条件归零代替取模：

```c
// 正确写法：2~3 条指令，中断耗时 < 1μs
idata unsigned char PWM_Count = 0;

// ISR 内
if (++PWM_Count == 10) PWM_Count = 0;
```

### 举一反三

- ISR 应尽可能短，避免 32 位运算、除法、浮点运算、库函数调用
- 软件模拟 I2C / SPI 对中断延迟极敏感，与中断共用总线时要特别评估 ISR 执行时间
- PWM 计数变量用 `unsigned char` 足够（0～255），无需更大类型

## 坑 7：按键无法控制流水灯的停止

> 来源：[007_蓝桥杯单片机第三周周报（二）.md](./单片机答疑数据库/007_蓝桥杯单片机第三周周报（二）.md)
> 周期：2025年09月22日 — 2025年09月28日
> 关联模块：综合赛题 / 系统 / 按键

### 整理记录

* **问题描述**：按键可以控制流水灯的启动但无法控制流水灯的停止
* **问题背景**：[b站] [零基础入门二] [LED灯]，采用proteus仿真图
* **问题分析**：

  + 可能是因为用阻塞性延时导致按键无法实时检测
  + 这个程序除了delay都进行的太快了，点按键的那一步是在delay这个阶段时间内点的，无法奏效
* **解决方案**：

  + 用定时器实现非阻塞流水，主循环负责扫描按键控制启停，不用delay：

```
void key_proc()
{
    if(key_slow<=10) return;
    key_slow=0;
	Key_Val = Key_Read();
    Key_Down = Key_Val & (Key_Val ^ Key_Old);
   	Key_Up = ~Key_Val & (Key_Val ^ Key_Old);
    Key_Old = Key_Val;
	switch(Key_Down)
    {
         //流水灯启动
		case 1:System_Flag=1;break;
         //流水灯暂停
		case 2:System_Flag=0;break;
	}
}

void Timer0Server() interrupt 1
{
	TL0 = 0x18;
	TH0 = 0xFC;
	key_slow++;
    if(System_Flag==0) time_200ms=0;
    if(System_Flag==0) return;
    if(++time_200ms>=200)
    {
		time_200ms=0;
		usLed=_crol_(usLed,1);
		P1=usLed;
	}
}
```

* **问题背景**:无具体课程背景。
* **问题分析**:

  + 连续读取光敏和电位器的数值时,ADC通道切换不当
  + 会导致数据互换或采样错误
  + 判断逻辑基于错误的数据,导致LED状态不稳定
* **解决方案**:

  + 读取完一个传感器数据后,立即进行判断和处理,再读取下一个传感器数据。避免连续读取多个ADC通道导致数据混乱。每次读取后要有足够的时间间隔,确保ADC转换完成。

    }

## 坑 23：15省赛2的LED测评问题

> 来源：[054_蓝桥杯单片机第十八周周报（二）.md](./单片机答疑数据库/054_蓝桥杯单片机第十八周周报（二）.md)
> 周期：2026年2月3日 — 2026年2月8日

### 整理记录

* **问题描述**：15省赛2的LED处理程序在实际板子上现象正常，但总是无法通过在线测评系统。
* **问题背景**：蓝桥杯单片机竞赛STC15F2K60S2平台
* **问题分析**：
经过代码审查，发现存在两大类问题；
**问题1：bit类型取反错误（3处）**
代码中存在3处对bit类型变量使用`~`运算符进行取反的错误：
- **line 70**: `Control_Mode=~Control_Mode;`
- **line 76**: `Block_Flag=~Block_Flag;`
- **line 217**: `Start_Flag=~Start_Flag;`
**错误原因**：在C51中，对bit类型（单bit变量）使用按位取反运算符`~`时，会先将bit提升为int类型（0或1），然后进行按位取反。例如：
- bit值为0时，提升为int的0，取反得到0xFFFF（-1）
- bit值为1时，提升为int的1，取反得到0xFFFE（-2）
- 再赋值回bit时，由于非零即真，结果始终为1
这导致变量无法正确在0和1之间切换。
**问题2：竞态条件**
代码中存在多个函数同时访问和修改共享数组`ucLed[]`：
- `Get_Temperature()`在主循环中修改`ucLed[1]`和`ucLed[2]`
- `Timer1_Isr()`定时器中断中也修改`ucLed[1]`和`ucLed[2]`
- `Led_Proc()`每1ms读取`ucLed`数组进行显示
**问题根源**：没有原子保护机制，主循环和中断可能同时修改同一内存位置，导致：
- 主循环写入一半时被中断打断
- 中断修改覆盖主循环的写入
- LED显示状态不一致或闪烁
* **解决方案**：
**方案1：修复bit类型取反**
将所有bit类型取反操作改为逻辑取反或异或运算：
```c
// 错误写法
Control_Mode = ~Control_Mode; // line 70
Block_Flag = ~Block_Flag; // line 76
Start_Flag = ~Start_Flag; // line 217
// 正确写法1：使用逻辑取反
Control_Mode = !Control_Mode;
Block_Flag = !Block_Flag;
Start_Flag = !Start_Flag;
// 正确写法2：使用异或运算
Control_Mode ^= 1;
Block_Flag ^= 1;
Start_Flag ^= 1;
```
**方案2：解决竞态条件**
```c
// 在主循环修改ucLed数组前后关闭/开启中断
void Get_Temperature(void)
{
EA = 0; // 关闭全局中断
ucLed[1] = temp_value / 10;
ucLed[2] = temp_value % 10;
EA = 1; // 恢复全局中断
}
// 或者使用临时变量缓存后一次性更新
void Get_Temperature(void)
{
unsigned char temp1, temp2;
temp1 = temp_value / 10;
temp2 = temp_value % 10;
EA = 0;
ucLed[1] = temp1;
ucLed[2] = temp2;
EA = 1;
}
```

## 坑 25：流水灯一次性跑完无动画效果（for 循环误用）

> 来源：[062_蓝桥杯单片机第二十二周周报.md](./单片机答疑数据库/062_蓝桥杯单片机第二十二周周报.md)
> 周期：蓝桥杯单片机第二十二周周报

### 整理记录

- **问题描述：** 流水灯模式下 LED 没有逐个移动的效果，直接跳到最终状态后立即切换模式
- **问题背景：** 流水灯程序设计
- **问题分析：** `LED_Proc()` 中用 `for(i=1;i<8;i++)` 在单次函数调用中将所有 LED 状态一次性移完。循环体内 `Led_Buf[i]=1; Led_Buf[i-1]=0;` 连续执行 7 轮，中间没有延时，所有赋值在同一帧完成，视觉上只能看到最终状态（仅 `Led_Buf[7]=1`）。此外，循环结束后 `i` 必然等于 8，`if(i==8) Led_Mode=1;` 每次都成立，模式被立即切换，流水灯动画完全无法呈现
```c
// 错误写法：一次调用中 for 循环跑完所有步
for(i=1; i<8; i++)
{
Led_Buf[i] = 1;
Led_Buf[i-1] = 0; // 7 轮连续执行，只看到最终态
}
if(i == 8) Led_Mode = 1; // i 必然等于 8，模式立即切换
```
- **解决方案：** 引入 `Step_Count` 变量记录当前移动步数，每次定时到达时只执行一次数组左移操作（`Led_Buf[i]=Led_Buf[i+1]`），步数累计到 8 时才切换模式
```c
// 正确写法：每次调用只移一格
if(Timer >= Run_Switch_Time[Led_Mode])
{
Timer = 0;
for(i = 0; i < 7; i++)
Led_Buf[i] = Led_Buf[i+1]; // 数组整体左移一格
Led_Buf[7] = 0;
Step_Count++;
if(Step_Count >= 8)
{
Step_Count = 0;
Led_Mode = 1;
}
}
```
**核心理解：** 流水灯每一帧只移动一格，帧与帧之间靠定时器间隔产生动画效果，不能在一次调用中用循环完成所有帧

## 坑 26：Led_Disp 锁存器冲突导致电压显示错误及 LED 状态异常（4处修复）

> 来源：[066_蓝桥杯单片机第二十三周周报.md](./单片机答疑数据库/066_蓝桥杯单片机第二十三周周报.md)
> 周期：蓝桥杯单片机第二十三周周报

### 整理记录

- **问题描述：** 4T 测评中通道3电压显示 2.50V（应为 3.50V）、LED 状态错误、LED 功能开关失效，共 4 个测试点未通过
- **问题背景：** 第十二届省赛（第二批）
- **问题分析：**
- [a] **Led_Disp 锁存器冲突（核心问题）：** `Led_Disp` 在主循环中调用，`Seg_Disp` 在定时器 ISR 中调用，两者共用 74HC573 锁存器。ISR 随时可能打断主循环的 `Led_Disp`，导致锁存器状态混乱，数码管段码串到 LED 上，电压显示偶发错误
- [b] **S7 长按逻辑错误：** `Key_Down==7 && Key_Up==7` 同时判断，但 `Key_Down`（按下沿）和 `Key_Up`（松开沿）是互斥事件，不可能同时为真，导致 LED 开关功能完全失效
- [c] **AD 地址多加 0x40 DAC 使能位：** `Ad_Read(0x43)` 中 0x40 为 DAC 使能位，纯 AD 采集不需要，属多余操作
- [d] **进入电压界面未重置通道：** 从周期界面按 S4 进入电压界面时未重置 `Seg_AD_Mode = 0`，不符合题目"每次进入电压界面均显示通道1"的要求
- **解决方案：**
- [a] 将 `Led_Disp` 移到定时器 ISR 中紧跟 `Seg_Disp` 之后调用，所有 573 锁存器操作集中在 ISR 中避免竞争
- [b] 长按检测改为"按下启动计时，松开时判断时长"：`Key_Down` 时启动计时标志，`Key_Up` 时判断 `Time_1_1s >= 1000` 则为长按
- [c] `0x43` 改为 `0x03`，`0x41` 改为 `0x01`，去掉 DAC 使能位；`51.0` 改为 `51`，避免浮点运算
- [d] 切入电压界面时添加 `Seg_AD_Mode = 0`

## 坑 28：Led_Disp 中 if 语句末尾多敲分号导致 LED 持续闪烁

> 来源：[066_蓝桥杯单片机第二十三周周报.md](./单片机答疑数据库/066_蓝桥杯单片机第二十三周周报.md)
> 周期：蓝桥杯单片机第二十三周周报

### 整理记录

- **问题描述：** 学员刚开始搭建底层，LED 持续闪烁，且仅在数码管有数显示时闪
- **问题背景：** 底层搭建阶段，验证 LED 基本功能
- **问题分析：** `Led_Disp` 函数第 12 行 `if(temp_1!=temp_1_Old);` 末尾多了一个**分号**。C 语言中 `if(...);` 等价于 `if(...) { }` —— 空语句，条件判断的结果被丢弃，后面的花括号 `{ ... }` 变成了**无条件执行**的代码块。这意味着：
- 无论 `temp_1` 是否变化，每次调用 `Led_Disp` 都会执行 P0 写入和 573 锁存操作
- `Led_Disp` 在定时器 ISR 中与 `Seg_Disp` 交替执行，每次都重新锁存 LED 数据到 P0
- 当数码管正在扫描时，`Seg_Disp` 往 P0 写段码，紧接着 `Led_Disp` 又往 P0 写 LED 数据并锁存，两者交替导致 LED 在段码数据和 LED 数据之间快速切换，表现为闪烁
- 当数码管无数据（全灭）时，P0 段码为 0xFF，LED 恰好也是全灭，闪烁不明显；有数时段码 ≠ LED 数据，闪烁明显
原本 `if(temp_1!=temp_1_Old)` 的优化意图是"LED 状态没变就不操作 573"，加了分号后这个优化完全失效
- **解决方案：** 删除 `if` 语句末尾多余的分号：
```c
// 修改前（分号使 if 判断变成空语句，花括号无条件执行）
if(temp_1!=temp_1_Old);
{
P0=~temp_1;
// ... 573 锁存操作 ...
}
// 修改后（去掉分号，花括号正确受 if 控制）
if(temp_1!=temp_1_Old)
{
P0=~temp_1;
// ... 573 锁存操作 ...
}
```
> **附注：** `if(...);` 是 C 语言中极其隐蔽的错误，编译器不会报错也不会警告，但语义完全改变。建议养成习惯：写完 `if` / `while` / `for` 后立刻写花括号，不要先打分号。

## 坑 31：LED 底层驱动冗余疑问

> 来源：[071_蓝桥杯单片机第二十五周周报（二）.md](./单片机答疑数据库/071_蓝桥杯单片机第二十五周周报（二）.md)
