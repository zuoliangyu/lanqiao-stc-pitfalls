## 坑 14：时钟设置界面切换失败（赋值写成比较）

> 来源：[044_蓝桥杯单片机第十二周周报（二）.md](./单片机答疑数据库/044_蓝桥杯单片机第十二周周报（二）.md)
> 周期：2025年12月9日 — 2025年12月14日
> 关联模块：温度 / 时钟 / 传感器

### 整理记录

* **问题描述**：学员在基于定时器的时钟项目中，从显示界面切换到设置界面时，时钟没有停止计时，模式切换逻辑失败。
* **问题背景**：基于定时器的时钟项目实战。
* **问题分析**：

  + 按键扫描函数中使用了**赋值运算符** `=` 而非**比较运算符** `==`
  + 错误代码：`if(temp=0)`
  + 这导致每次判断时都把temp赋值为0，条件永远为假，无法正确识别按键状态
  + temp变量用于记录按键值，应该用`==`判断其是否为0
* **解决方案**：

  1. 将按键扫描函数中的 `if(temp=0)` 修改为 `if(temp==0)`
  2. 修改后按键扫描逻辑正常，模式切换成功，时钟能够正确暂停
  3. 强调C语言基础：`=` 是赋值，`==` 是比较
  4. 建议学员使用编译器警告功能，Keil会对这类可疑代码发出警告
* **知识点总结**：

  + C语言中赋值运算符和比较运算符的区别
  + `if(temp=0)`会先执行赋值，再判断赋值结果（0为假），导致条件永远不成立
  + 按键扫描中temp变量的作用：记录按下的按键值，0表示无按键

## 坑 16：main函数缺少循环导致程序跑飞

> 来源：[047_蓝桥杯单片机第十三周周报（二）.md](./单片机答疑数据库/047_蓝桥杯单片机第十三周周报（二）.md)
> 周期：2024年12月14日 — 2024年12月21日

### 整理记录

* **问题描述**：程序运行后立即结束，无法持续执行，表现为程序”跑飞”。
* **问题背景**：51单片机程序开发，main函数结构不完整。
* **问题分析**：

  + main函数中缺少`while(1)`无限循环
  + 单片机程序应该持续运行，而不是执行一次就结束
  + 没有循环会导致程序执行完main函数后进入未定义状态
* **解决方案**：

  + 在main函数中添加无限循环：

```
void main(void)
{
    // 初始化代码
    Init_System();

    // 主循环
    while(1)
    {
        // 持续执行的任务
        Key_Scan();
        Display_Refresh();
    }
}
```

> 来源：[062_蓝桥杯单片机第二十二周周报.md](./单片机答疑数据库/062_蓝桥杯单片机第二十二周周报.md)
> 周期：蓝桥杯单片机第二十二周周报

### 整理记录

- **问题描述：** 跑动状态下数码管应显示 L3，实际显示 L1（静止）；进入参数设置界面后 L8 应保持之前的闪烁状态，但实际停止闪烁
- **问题背景：** 第十六届省赛真题。学员使用 DeepSeek、豆包等 AI 工具反复调试仍无法解决
- **问题分析：** 存在两处设计缺陷：
- [a] **运动状态被立刻覆盖（测评28、49、61）：** `Get_Distance` 调度周期为 1000ms，运动状态锁定时间为 3000ms。1000 整除 3000，导致第 4 次采样恰好卡在锁定到期瞬间，锁刚解除就立刻重新判定状态，此时连续两次距离值相近（物体已停下），状态被立刻覆盖为"静止"（L1），跑动状态（L3）无法维持
- [b] **L8 闪烁缺少状态冻结（测评22、63）：** 中断里闪烁条件只判断 `Motion_State==3` 才驱动闪烁计时器，但进入参数界面后 `Motion_State` 可能被新的距离采样覆盖为 1（静止），闪烁随之停止。代码缺少状态冻结机制——题目要求进入参数界面时 LED 状态"锁定"，但没有保存进入前的运动状态快照
- **解决方案：**
- [a] 采样周期改为 999ms（错开整除关系），锁定时间改为 3050ms（增加 50ms 余量防止边界竞争）。**教训：** 定时器设计中要避免周期之间存在整除关系，否则会产生同步问题
- [b] 新增 `Run_State`、`Stay_State`、`Paralysis_State` 三个冻结变量，在非锁定时实时更新，锁定时保持不变；中断中闪烁条件改为 `Motion_State==3 || (Led_Lock && Run_State)`；`Led_Proc` 锁定分支中用冻结变量控制 L8。**教训：** 题目说"状态锁定"时，需用额外变量快照当前状态，不能直接依赖实时变化的原始变量

## 坑 20：无效按键计数未按界面区分导致错误指示灯不亮（4T 测评 #21/#23）

> 来源：[062_蓝桥杯单片机第二十二周周报.md](./单片机答疑数据库/062_蓝桥杯单片机第二十二周周报.md)
> 周期：蓝桥杯单片机第二十二周周报

### 整理记录

- **问题描述：** 4T 测评得分 67，测评 #21（无效按键触发 3 次后 L3 应亮）和 #23（功能按键触发后 L1/L2/L3 应熄灭）始终未通过，错误状态指示灯逻辑异常
- **问题背景：** 第十一届省赛题目
- **问题分析：** `Key_Proc()` 中用一个 `switch(Key_Down)` 直接判断按键编号，S4/S5/S8/S9 不管在哪个界面按下都会清零 `Key_Error`。但题目要求的是：**按键在当前界面没有功能时，才算无效按键**。例如 S5 的功能是清零计数，仅在计数界面（界面3）有效；在数据监测界面（界面1）按 S5 应算一次无效按键，`Key_Error` 应 +1。但代码直接把 `Key_Error` 清零了，导致错误计数永远涨不到 3，L3 永远不会亮
```c
// 错误写法：不区分界面，所有按键都清零错误计数
switch(Key_Down)
{
case 4: Key_Error = 0; /* ... */ break;
case 5: Key_Error = 0; /* ... */ break;
// S4/S5/S8/S9 在任何界面按下都清零，无效计数永远累加不起来
}
```
- **解决方案：** 改为**先判断当前界面，再判断按键**的双层 switch 结构。每个界面只有该界面的有效按键才清零错误计数，其余按键走 `default` 分支累加 `Key_Error`
```c
// 正确写法：外层按界面分，内层按按键分
switch(Seg_Mode) // 先判断当前界面
{
case 0: // 界面0
switch(Key_Down)
{
case 4: Key_Error = 0; /* 界面0的有效操作 */ break;
// ...
default: Key_Error++; break; // 该界面无效的按键，累加错误
}
break;
case 2: // 计数界面
switch(Key_Down)
{
case 5: Key_Error = 0; /* 清零计数 */ break;
// ...
default: Key_Error++; break;
}
break;
}
```

## 坑 21：继电器计数用局部变量 + 运动状态编码初始值不匹配 + 高温判定边界错误 + L8 锁定范围错误（7处修复）

> 来源：[066_蓝桥杯单片机第二十三周周报.md](./单片机答疑数据库/066_蓝桥杯单片机第二十三周周报.md)
> 周期：蓝桥杯单片机第二十三周周报

### 整理记录

- **问题描述：** 4T 测评 59/71（68.8/85 分），12 个测试点未通过。学员反馈"自己挑不出来错了"，曾多次使用 AI 辅助排查仍未修到点子上。主要表现为继电器能计次但记不准、运动状态指示灯行为异常、参数界面锁定后继电器/LED 状态错误
- **问题背景：** 第十六届省赛，涉及超声波测距运动检测、继电器吸合计数、温度阈值判定、参数界面锁定
- **问题分析：**
- [a] **继电器计数使用局部变量（致命）：** `Led_Proc()` 中用局部变量 `temp` 记录继电器上一次状态，意图在继电器从关→开瞬间才计数一次。但局部变量每次函数调用都是未初始化的随机值，函数返回后赋值即丢失，导致 `Relay_Count` 计数完全不可控。影响 #39、#41、#54、#70 共 4 个测试点
- [b] **运动状态编码初始值不匹配（严重）：** 学员编码为 1=静止、2=徘徊、3=跑动，但 `Motion_State` 全局变量初始值为 0（C 语言默认）。上电后第一次检测到静止时 `temp_motion_state=1`，与初始值 0 不同，系统误判为"状态变化"触发 3 秒锁定，打乱后续所有状态检测时序。满分代码用 0=静止，初始值天然匹配，不会伪触发
- [c] **L8 运动指示灯被参数界面错误锁定（严重）：** L8 的更新逻辑写在了 `Relay_Led_Lock` 的 `if` 块内部，参数界面时 Lock=1，L8 完全冻结。但题目要求只锁定 L1-L4 和继电器，L8 应始终实时反映运动状态
- [d] **L8 闪烁计时器退出跑动后未清零（中等）：** 退出跑动状态后 `Time_100ms` 和 `Led_Blink` 保持最后的值，再次进入跑动时第一个闪烁周期不是 0.1s，测评判定闪烁间隔错误
- [e] **高温判定 `>=` 应为 `>`（中等）：** 题目原文"温度**高于**参数时触发高温"，"高于"是严格大于，不包含等于。温度恰好等于参数时被误判为高温，继电器不该吸合时吸合。影响 #67、#68、#69、#71
- [f] **运动状态输出位未做锁定隔离（中等）：** 参数界面期间 L1-L4 和继电器应冻结在进入前的状态，但 `Motion_State` 持续更新，直接用它控制 LED/继电器无法实现冻结。需要用独立的输出控制位，只在未锁定时同步
- [g] **统计界面继电器次数只用 3 位显示（轻微）：** 题目要求 4 位显示，高位不足时熄灭
- **解决方案：**
- [a] 将 `temp` 改为全局变量 `Relay_Old`（或加 `static`），确保跨函数调用保持值：
```c
bit Relay_Old; // 全局变量
// Led_Proc() 中：
if(Relay_Old == 0) Relay_Count++;
Relay_Old = 1;
```
- [b] 将状态编码改为 0=静止、1=徘徊、2=跑动，与全局变量初始值 0 匹配；显示时 `Seg_Buf[1] = Motion_State + 1`
- [c] 将 L8 更新逻辑移到 `Relay_Led_Lock` 的 `if` 块**外面**，始终更新
- [d] 退出跑动状态时清零 `Time_100ms = 0; Led_Blink = 0;`
- [e] `Temperature_10x >= T_Temp*10` 改为 `Temperature_10x > T_Temp*10`
- [f] 用 `Stay_State`、`Wander_State`、`Run_State` 作为输出控制位，只在 `Relay_Led_Lock==0` 时更新
- [g] 继电器计数显示扩展为 4 位，`Relay_Count` 类型改为 `unsigned int`
> **附注：** 本例中学员多次使用 AI 辅助排查但始终未修到核心问题，原因是 AI 倾向于修改表层逻辑而忽略底层设计缺陷（如局部变量生命周期、状态编码初始值一致性）。这类问题需要理解 C 语言基础概念和硬件交互机制才能定位。关键经验：① 需要跨调用保持的状态**必须用全局或 static**；② 状态编码时默认状态应编码为 0；③ "高于/低于"是严格比较，"不低于/不高于"才包含等于。

## 坑 22：L1 指示灯条件写反 + 进入回显界面未重置子页面 + 上升趋势判断变量覆盖（4处修复）

> 来源：[066_蓝桥杯单片机第二十三周周报.md](./单片机答疑数据库/066_蓝桥杯单片机第二十三周周报.md)

- **问题背景：** 第十四届省赛（第一批），涉及时间显示、回显界面（温度/湿度/触发子页面）、上升趋势判断

```c
if(Key_Down == 4)
{
Seg_Mode = (++Seg_Mode) % 3;
if(Seg_Mode == 1)
HuiXian_Mode = 0; // 进入回显界面时重置为温度子页面
}
```

## 坑 23：AUXR 操作符写错 + Ctrl_Proc 调度周期过长 + S4 缺少运行模式处理 + idata 溢出，导致界面切换失灵和卡死在运行模式（6处修复）

> 来源：[066_蓝桥杯单片机第二十三周周报.md](./单片机答疑数据库/066_蓝桥杯单片机第二十三周周报.md)
> 周期：蓝桥杯单片机第二十三周周报

### 整理记录

- **问题描述：** 学员反馈代码逻辑跟满分代码基本一致了，但仍有阻塞现象——要么切换不了界面，要么卡死在运行模式
- **问题背景：** 第十五届国赛，涉及运动控制（坐标计算、速度模拟）、多界面切换、串口接收坐标、参数调节
- **问题分析：**
- [a] **Timer0 初始化 AUXR 操作符写错（致命）：** `AUXR &= 0x80` 用的是与操作，会**清除 bit0~bit6 只保留 bit7**。若 AUXR 默认为 0，`0 & 0x80 = 0`，bit7 也没被置位，Timer0 根本没进入 1T 模式。更严重的是该语句会把 AUXR 其他位全部清零，可能影响串口等后续初始化的外设。正确写法是 `AUXR |= 0x80`（或操作置位 bit7）
- [b] **Ctrl_Proc 调度周期太长（致命）：** 运动控制函数 `Ctrl_Proc` 调度周期设为 20ms，满分代码为 1ms。Dt 值大了 20 倍，位置更新频率过低，浮点运算累积误差更大，运动模拟出现异常，可能无法正确到达目的地
- [c] **S4 按键缺少运行模式的处理——卡死根因（致命）：** S4 只处理了空闲（`Status==0`）→运行 和 等待（`Status==2`）→运行 两种情况，**完全没有处理 `Status==1`（运行中）的情况**。满分代码在运行模式下按 S4 可手动切到等待模式。学员代码一旦进入运行模式，如果速度为 0（没有频率输入且 B=0），设备永远不会到达目的地也不会自动切换状态，就彻底卡死了
- [d] **S9/S12/S13 按键没有限制在参数界面（严重）：** S9、S12、S13 是顶层 `case`，任何界面下按都会生效。在位置或速度界面按 S12/S13 时会悄悄修改参数，切回参数界面时值已变，引起不可预期的行为
- [e] **进入运行模式时没有清除标志位（中等）：** 未清除 `Get_Coordinate_Flag` 和 `Reach_Flag`。不清除 `Get_Coordinate_Flag` 的后果是回到空闲状态后按 S4 会立刻用旧的目的地坐标再次进入运行模式
- [f] **多余的 idata 变量浪费宝贵内存（中等）：** 声明了 `temp_Speed`（float, 4字节）、`Key_Down_data`、`Key_Up_data`、`AD_3_Data_10x` 等从未使用的 `idata` 变量。8051 的 idata 空间只有 256 字节（还要跟寄存器组、栈共享），多余变量挤压栈空间，当 `sqrt()`、`sscanf()` 等库函数调用较深时可能导致**栈溢出**，表现为程序随机卡死、变量值莫名被篡改、界面切换失灵
- **解决方案：**
- [a] `AUXR &= 0x80` 改为 `AUXR |= 0x80`
- [b] `Ctrl_Proc` 调度周期从 20ms 改为 1ms：
```c
// 修改前
{Ctrl_Proc, 20, 0},
// 修改后
{Ctrl_Proc, 1, 0},
```
- [c] S4 按键补充运行模式处理：
```c
case 4:
if(Status == 0 && Get_Coordinate_Flag)
{
Status = 1;
First_Flag = 0;
Reach_Flag = 0;
Get_Coordinate_Flag = 0;
}
else if(Status == 1) // ← 新增：运行中按S4切到等待
{
Status = 2;
}
else if(Status == 2 && Clear_Rubbish_Flag)
{
Status = 1;
First_Flag = 0;
}
break;
```
- [d] S9/S12/S13 加 `Seg_Mode == 2` 前置判断，限制只在参数界面生效
- [e] 进入运行模式时补充清零 `Get_Coordinate_Flag = 0; Reach_Flag = 0;`
- [f] 删除所有未使用的 idata 变量，释放内存空间
> **附注：** 本例的根因结论：**"卡死在运行模式"** = 问题 [c]（S4 没有运行→等待的处理）+ 问题 [b]（Ctrl_Proc 周期太长导致运动异常无法到达目的地）；**"切换不了界面"** = 问题 [f]（idata 内存溢出导致栈破坏，程序行为不可预测）+ 问题 [d]（参数按键不受界面限制，可能污染状态）。学员认为"代码逻辑基本一致"时往往只关注算法流程是否相同，而忽略了操作符（`&=` vs `|=`）、调度参数（20ms vs 1ms）、状态机完整性（缺少一个状态分支）、内存管理（idata 溢出）这些"细节"——但在嵌入式开发中，这些细节决定程序能否正常运行。

## 坑 24：BTN 独立按键模式误用矩阵扫描

> 来源：[067_蓝桥杯单片机第二十三周周报（二）.md](./单片机答疑数据库/067_蓝桥杯单片机第二十三周周报（二）.md)
> 周期：蓝桥杯单片机第二十三周周报（二）

### 整理记录

- **问题描述：** 所有按键功能完全失效，但初始界面显示正常（得了19.1分纯靠不需要按键的测试点）
- **问题背景：** 第13届国赛
- **问题分析：** 题目要求 J5 配置为 **BTN 独立按键模式**，但代码使用了矩阵键盘的列扫描逻辑。在BTN模式下，S4-S7 直接连接 P3.3-P3.0 与 GND，按下时 P3.x 恒为低电平，与列线无关。矩阵扫描时三次列遍历都能检测到，后赋值覆盖前赋值：
```
按下S7 (P3.0=0):
第1次列扫描 P44=0: if(P30==0) temp=7; ← 正确
第2次列扫描 P42=0: if(P30==0) temp=11; ← 覆盖！
第3次列扫描 P35=0: if(P30==0) temp=15; ← 再次覆盖！
→ 最终返回15，代码检查 Key_Down==7 永远不匹配
```
所有按键的实际返回值都偏移了8：S4→12, S5→13, S6→14, S7→15
- **解决方案：** BTN模式下直接读P3端口，不做列扫描：
```c
unsigned char Key_Read()
{
unsigned char temp = 0;
if(P33 == 0) temp = 4;
if(P32 == 0) temp = 5;
if(P31 == 0) temp = 6;
if(P30 == 0) temp = 7;
return temp;
}
```
> **规律总结：** 赛前必须确认题目要求的J5跳线模式。BTN模式用独立读取，KBD模式用矩阵扫描，两套代码不能混用。建议准备两个版本的 `Key_Read()` 模板。

> 来源：[069_蓝桥杯单片机第二十四周周报（二）.md](./单片机答疑数据库/069_蓝桥杯单片机第二十四周周报（二）.md)

### 整理记录

- **问题描述：** 4T 测评得分 49.7，学员反馈三个主要困惑：①继电器吸合次数错误 ②"运动状态改变"的题意不理解 ③L8 闪烁间隔不对。测评中指示灯状态错误（#2/#3/#15/#25/#26/#44/#46/#59）、L8 闪烁间隔 error（#4/#22/#48）、继电器误动作（#16）、运动状态/继电器计数错误（#23/#27/#38/#39/#41/#42/#49/#50/#54/#55/#57/#58）等多项未通过
- **问题背景：** 第十六届省赛第一次题目（温度+超声测距+光强+运动状态综合题）
- **问题分析：** 非单一错误，存在多个系统性问题叠加：
- [a] **光强 ADC 通道读取错误：** ADC 读取通道写错，导致光强等级 `N1/N2/N3/N4` 经常被误判为 `N4`，环境界面显示错误，L1~L4 光强等级点亮逻辑全部受影响

- [d] **L8 闪烁间隔错误：** 跑动状态下 L8 应以 0.1s 间隔闪烁，实际闪烁周期不正确
- [e] **参数界面锁定逻辑不完整：** 进入参数界面后锁住了输出，但退出后恢复逻辑不完整，LED 状态"卡住"、继电器不再随实际条件更新
- [f] **启动阶段状态灯初始化时序问题：** 上电后前几次超声波采样期间，运动状态相关灯被提前带出，测评机启动阶段检查严格导致前几项始终报错
- **解决方案：**
- [a] 修正光强 ADC 读取通道
- [b] 高温/接近标志位改为满足条件置 1、不满足条件清 0
- [c] 参考满分代码拆分运动状态和输出状态，分别定义静止/徘徊/跑动输出状态，L8 显示逻辑与"当前输出状态"关联而非直接绑定原始检测变量
- [d] L8 闪烁改为：静止时熄灭、徘徊时常亮、跑动时按 0.1s 周期闪烁
- [e] 修正参数界面进入/退出时的锁定与恢复逻辑
- [f] 建议上电后在前两次有效距离数据拿到之前，强制将运动状态固定为"静止"，L8 始终熄灭，避免启动阶段状态灯误亮

## 坑 27：第十五届国赛 程序阻塞卡死——多处致命错误叠加

> 来源：[069_蓝桥杯单片机第二十四周周报（二）.md](./单片机答疑数据库/069_蓝桥杯单片机第二十四周周报（二）.md)
> 周期：蓝桥杯单片机第二十四周周报（二）

### 整理记录

- **问题描述：** 学员反馈代码逻辑与参考代码基本一致，但存在阻塞现象——要么切换不了界面，要么卡死在运行模式
- **问题背景：** 第十五届国赛题目（运动控制综合题）
- **问题分析：** 共发现 6 个问题，其中 3 个致命：
- [a] **Timer0 初始化 AUXR 操作符写错（致命）：** 将 `AUXR |= 0x80` 误写为 `AUXR &= 0x80`，`&=` 会清除 bit0~bit6，Timer0 没有进入 1T 模式，还可能影响串口等其他外设初始化
- [b] **Ctrl_Proc 调度周期太长（致命）：** 运动控制函数设为 20ms 执行一次（应为 1ms），位置更新频率太低，Dt 值大 20 倍导致浮点累积误差大，运动模拟异常无法正确到达目的地
- [c] **S4 按键缺少运行模式处理（致命/卡死根因）：** S4 只处理了空闲→运行和等待→运行，完全没有处理运行中按 S4 切换到等待模式的情况。一旦进入运行模式且速度为 0，设备永远不会到达目的地也不会自动切换状态，彻底卡死
- [d] **S9/S12/S13 按键没有限制在参数界面（严重）：** 这些按键在任何界面下都会生效，在位置/速度界面按下时会悄悄修改参数，引起不可预期的行为
- [e] **进入运行模式时没有清除标志位（中等）：** 未清除 `Get_Coordinate_Flag` 和 `Reach_Flag`，回到空闲状态后按 S4 会用旧坐标再次进入运行模式
- [f] **多余的 idata 变量浪费内存（中等）：** 声明了多个未使用的 `idata` 变量（共约 7 字节），8051 的 idata 仅 256 字节且与栈共享，挤压栈空间可能导致 `sqrt()`、`sscanf()` 等库函数调用时栈溢出，表现为程序随机卡死、界面切换失灵
- **解决方案：**
- [a] `AUXR &= 0x80` 改为 `AUXR |= 0x80`
- [b] `Ctrl_Proc` 调度周期从 20ms 改为 1ms
- [c] S4 按键处理中补充 `Status == 1` 时切换到 `Status = 2`（等待模式）
- [d] S9/S12/S13 处理加 `Seg_Mode == 2` 前置判断，仅在参数界面有效
- [e] 进入运行模式时同时清除 `Get_Coordinate_Flag = 0` 和 `Reach_Flag = 0`
- [f] 删除所有未使用的 idata 变量，释放栈空间

## 坑 29：第十五届模拟二 Key_Read() 变量未初始化导致模式误切换

> 来源：[070_蓝桥杯单片机第二十五周周报.md](./单片机答疑数据库/070_蓝桥杯单片机第二十五周周报.md)
> 周期：蓝桥杯单片机第二十五周周报

### 整理记录

- **问题描述：** Output_mode 为模式二时按下 S5，没有直接切换界面到 Output_mode=3（输出界面），而是先切回模式一，再次按下才切换到输出界面
- **问题背景：** 第十五届模拟二，矩阵键盘模式切换与界面切换
- **问题分析：**
- [a] **局部变量 `temp` 未初始化：** `Key_Read()` 中 `unsigned char temp;` 没有初始化为 0，无按键按下时返回随机值
- [b] **矩阵键盘多行扫描中随机值残留：** `Key_Read()` 依次扫描四行矩阵键盘，无按键按下时 `temp` 保持随机值。随机值可能恰好等于某按键的键值，导致主程序误判为按键事件
- [c] **模式误切换：** 当 Output_mode=2 时按下 S5（键值 5），由于 `temp` 随机值干扰，第一次 `Key_Read()` 可能返回错误键值（键值 4），主程序优先匹配了"切回模式一"的逻辑，输出模式被意外改回 1；第二次按下时随机值碰巧不冲突，才正确切到输出界面
- **解决方案：**
- [a] 将 `unsigned char temp;` 改为 `unsigned char temp = 0;`，确保无按键时返回 0
- **经验教训：** `Key_Read()` 中 `temp` 未初始化是蓝桥杯单片机最常见的 Bug 之一（本周第 1 题和第 7 题均为此类），每次写按键驱动时必须检查

## 坑 30：第十六届省赛 4T 测评组合键覆盖与参数界面锁定偏差

> 来源：[070_蓝桥杯单片机第二十五周周报.md](./单片机答疑数据库/070_蓝桥杯单片机第二十五周周报.md)
> 周期：蓝桥杯单片机第二十五周周报

### 整理记录

- **问题描述：** 第十六届省赛 4T 测评中四项失败：统计界面 S8+S9 长按 2 秒清零失败（第 56 条）、清零后统计值继续沿用旧值（第 57、66 条）、参数界面下跑动状态时 L8 停止按 0.1s 闪烁（第 63 条）
- **问题背景：** 第十六届省赛题目，涉及超声波测距、运动状态判定、温度+距离双条件继电器控制、S8+S9 组合键清零、参数界面锁定等
- **问题分析：** 学员代码存在 2 个根因，导致 4 个测评项失败：
- [a] **组合键被单键判断覆盖：** 按键驱动第二列扫描时，先判断了 `if((P32==0)&&(P33==0)) temp=89;`，但后续又用 `if(P32==0) temp=9;` 和 `if(P33==0) temp=8;` 覆盖了返回值，导致 `Key_Old==89` 无法稳定检测，长按清零功能失效
- [b] **参数界面将整个 LED 刷新停掉：** 参数界面锁定状态下用 `if(Led_Mode == 0) return;` 提前返回 `Led_Proc()`，虽然 L1-L4 正确保持状态，但 L8 的运动状态闪烁也被一起停止，不符合题目"锁定状态下 L8 仍需按跑动节奏闪烁"的要求
- [c] **清零失败的连锁反应：** 第 57、66 条本质上是第 56 条的连带结果——统计值未被清零，继电器再次吸合后计数继续累加，显示值比期望值大
- **解决方案：**
- [a] 将组合键判断改为 `else if` 链，确保 `89` 优先级最高不被覆盖：
```c
if((P32==0)&&(P33==0))
temp=89;
else if(P32==0)
temp=9;
else if(P33==0)
temp=8;
```
- [b] 拆分 `Led_Proc()` 为两部分：平时完整刷新 L1-L4；进入参数界面后锁定 L1-L4（不更新），但 L8 的运动状态闪烁逻辑继续执行
- [c] 温度采样顺序建议改为先采样再比较，避免判断慢一拍；统计次数变量建议改为 `unsigned int`，防止回卷

## 坑 31：第十五届国赛状态机与运动时间基准未闭环

> 来源：[070_蓝桥杯单片机第二十五周周报.md](./单片机答疑数据库/070_蓝桥杯单片机第二十五周周报.md)

### 整理记录

- **问题描述：** 设备进入等待态后串口查询 `?` 返回 `Idle`、速度界面显示 `E2 -----`（应为 `E3 xx` 或 `E1 xx.x`）、运行 1 秒后坐标明显偏小、非空闲态下收到串口坐标指令未返回 `Wait` 或 `Busy`
- **问题背景：** 第十五届国赛题目，涉及串口通信、运动状态机、超声波避障、光敏昼夜判断、位移计算等
- **问题分析：** 此问题有两份不同学员代码的分析报告：
- **第一份（2026-03-29）——上电后数码管、LED、按键全部不正常：**
- [a] **控制任务关键状态变量写成局部变量：** `Ctrl_Last_Tick` 和 `Ctrl_First_Flag` 在 `Ctrl_Proc()` 内部局部定义，每次进入重新清零，运动控制逻辑每次都"重新开始"，`Dt` 时间差不对
- [b] **串口接收缓冲提前清空：** 接收未超时就清缓冲区和索引，数据包没收到就被丢，`#`、`?`、`(x,y)` 命令无法完整解析

- [e] **超声波返回 `0` 未做无效值保护：** 测距失败返回 `0` 被当作"前方 0cm 有障碍物"，运行态立刻切等待态
- [f] **变量存储位置不合理：** 过多变量压在 `idata` 里，配合 `float`、`sqrt`、`printf`、`sscanf` 等重型库函数，内部 RAM 和栈空间压力大
- **第二份（2026-03-30）——等待态 S4 无法恢复 + E3 显示错误：**
- [g] **超声波返回值未做异常处理（主因）：** `Ut_Wave_Data()` 超时/无回波时返回 `0`，主程序直接当真实距离使用。导致两个后果：(1) E3 页面显示 `0`，看起来像"障碍物贴脸"；(2) 等待恢复判断 `Obstacle_Distance >= 30` 永远不成立，S4 无法恢复运动
- [h] **等待恢复运行时未重置运动基准：** 位移计算基于"最近一次运行时间差 × 速度"，但等待态恢复运行时只改了状态变量，未刷新基准时间戳（`First_Flag` 未清零），导致等待期间的时间被算入位移，一步跨到终点后状态立即回到空闲态
- [i] **启动时未立即刷新当前速度：** 速度值只在 1 秒定时中更新，刚启动时拿旧速度（甚至 0）计算位移，导致 1 秒后的坐标明显偏小
- [j] **S4 状态机不完整：** 只写了"空闲启动"和"等待恢复"，缺少"运行态切换到等待态"这一分支，状态机不闭环
- [k] **串口坐标指令分支逻辑错误：** 空闲态判断被重复嵌套，导致等待态和运行态收到坐标时没有被正确区分处理，状态返回不完整
- [l] **参数生效时机不合理：** 临时参数直接在显示界面里同步到正式参数，生效时机不明确
- [m] **次级风险：** 光敏通道可能写成 `0x01` 而非 `0x41`；串口缓冲区先写数据再判断越界，未保留字符串结束符

## 坑 32：第十六届省赛运动状态首帧误判与统计清零不稳

> 来源：[070_蓝桥杯单片机第二十五周周报.md](./单片机答疑数据库/070_蓝桥杯单片机第二十五周周报.md)
> 周期：蓝桥杯单片机第二十五周周报

### 整理记录

- **问题描述：** L8 状态指示灯错误（`L3 046` 显示成 `L1 045`）、统计清零失败（`NC 6` 应为 `NC 0`/`NC 1`）、统计界面显示边界不对
- **问题背景：** 第十六届省赛题目，超声波运动检测、接近判定、统计界面、光强分级、组合键清零等
- **问题分析：** 学员代码存在 4 个核心缺陷：
- [a] **第一次距离采样就参与运动判定：** 运动状态判定基于"连续两次采集的距离差值"，但第一次采样时距离初值为 0，直接和当前距离做差值容易误判成某运动状态并锁定 3 秒。后续真实状态变化反而进不来，导致 `L8` 状态灯和运动界面整体出错
- [b] **统计清零依赖"同时松手"：** `S8+S9` 长按清零逻辑是检测 `Key_Down==89` 开始计时、等 `Key_Up==89` 满 2 秒才清零，但两个键不一定在同一次扫描里同时松开，导致长按够了时间但程序未识别到有效清零，后续计数沿用了旧值
- [c] **统计界面显示边界判断用 `>` 而非 `>=`：** `(Relay_Count>1000)` 等判断使用了严格大于，导致 `Relay_Count==10` 时十位不显示、`==100` 时百位不显示、`==1000` 时千位不显示
- [d] **光强分级使用浮点转整型：** ADC 结果先做浮点除法 `Ad_Read()*5.0/255` 再存整型，`0.5V~1.0V` 区间会被截断成 `0`，误判为最低等级
- **解决方案：**
- [a] 增加 `First_Distance_flag` 首帧标志，第一次采到距离时只保存 `Distance_last`，不参与运动判定；`Sport_Mode` 初值设为 1，避免运动界面出现 `L0`
- [b] 统计清零改为"持续按压计时、松手即取消"模式：只要 `Key_Val==89` 就持续计时，满 2 秒直接清零；松开则立即重置计时。不再依赖"同时松手"
- [c] 显示判断改为 `>=`：`Relay_Count >= 1000`、`>= 100`、`>= 10`
- [d] 光强分级直接使用原始 AD 值分档（如 `AD>=3*51`、`>=2*51`、`>=26`），去掉浮点转换

## 坑 34：第十六届省赛运动状态赋值错写为比较与统计锁定逻辑未闭环

> 来源：[070_蓝桥杯单片机第二十五周周报.md](./单片机答疑数据库/070_蓝桥杯单片机第二十五周周报.md)
> 周期：蓝桥杯单片机第二十五周周报

### 整理记录

- **问题描述：** L1/L2/L3 运动状态指示灯错误（全部停在初始值）、L8 指示状态跟着错、统计界面 NC 后面数字显示 0 或 1 而非累计值、参数界面 LED/继电器未锁定、S8+S9 长按清零无效
- **问题背景：** 第十六届省赛题目，超声波运动检测、接近判定、统计界面、光强分级、组合键清零、参数界面输出锁定
- **问题分析：** 不是零散错误，而是几处核心逻辑缺陷导致连锁失败：
- [a] **赋值写成比较（最核心）：** `Distance_Old==Distance;` 和 `Distance_Count==1;` 用 `==` 替代了 `=`，不会改变量值。紧接着又执行 `Distance_Count=0` 重置，导致 `Distance_Count` 永远回 0，`else if(Distance_Count==1)` 分支永远进不去，`Exercise_Mode` 基本停在初始值，所有运动相关测试项连锁失败
- [b] **统计界面显示错变量：** 显示的是 `Relay_enable`（当前是否吸合的 0/1 标志），而非 `Relay_Count`（累计吸合次数），NC 数值完全错误
- [c] **继电器次数按循环累加而非上升沿：** 只要继电器条件成立就 `Relay_Count++`，统计的是"吸合期间任务执行次数"而非"吸合次数"
- [d] **S8+S9 组合键判断不可能成立：** `if(Key_Val==89 && Key_Down==8 && Key_Down==9)` 中 `Key_Down` 不可能同时等于 8 和 9
- [e] **参数界面未做输出锁定：** 进入参数界面后 L1-L4 仍跟着光强/距离实时变化，继电器也实时动作，L8 跟着运动状态实时变，不符合题目"参数界面锁定"要求
- [f] **运动状态采样周期不对：** `Get_Distance` 每 50ms 调一次，题目要求间隔 1 秒采样一次，且缺少 3 秒状态锁定逻辑
- **解决方案：**
- [a] 修正赋值：`Distance_Old=Distance;` 和 `Distance_Count=1;`；补全运动状态锁定逻辑：新增 `Exercise_Lock_Flag` 和 `Time_3s`，状态变化后锁定 3 秒不重新判定
- [b] 统计界面显示改为 `Relay_Count`
- [c] 继电器计数改为上升沿统计：仅在 `Relay_enable==0` 时吸合才 `Relay_Count++`
- [d] 组合键清零改为 `if(Key_Val==89)` 持续计时、满 2 秒清零、松手重置计时
- [e] `Led_Proc()` 中拆分参数界面内外逻辑：参数界面内锁定 L1-L4（保持进入时的状态），退出后恢复正常实时控制
- [f] `Get_Distance` 调度周期改为 1000ms
- **经验教训：** `==` 和 `=` 是最难肉眼发现的 Bug 之一，调试困难时应开启编译器警告（`-Wparentheses` 或 `-Wall`）；参数界面锁定是本题的必考点，进入时需冻结所有输出状态

## 坑 35：第十一届省赛EEPROM 提前写入（未实现退出界面存入）

> 来源：[071_蓝桥杯单片机第二十五周周报（二）.md](./单片机答疑数据库/071_蓝桥杯单片机第二十五周周报（二）.md)
> 周期：蓝桥杯单片机第二十五周周报（二）

### 整理记录

- **问题描述：** 在参数设置界面下修改电压阈值时，系统未按题目要求“退出界面存入”，而是实时保存了数据，导致评测系统在验证中间步骤时读取到错误的新值（预期读旧值 10，实际读到新值 20）。
- **问题背景：** 嵌入式系统的参数修改通常采用“草稿机制”，即在界面中调整暂存变量（`Volt_Para_Sketch`），等到用户按下确认键或切换界面离开时，才将最终结果同步给真实控制变量（`Volt_Para`）并写入非易失性存储器（EEPROM），以防止误修改及保护芯片寿命。
- **问题分析：** 该问题主要是由 EEPROM 写入指令的位置摆放错误，导致触发条件由“动作触发”变成了“状态触发”：
- [a] **存入时机游离于动作之外：** 赋值语句 `Volt_Para=Volt_Para_Sketch;` 和 `EEPROM_Write` 直接暴露在了 `if(Seg_Slow_Mode==1)` 分支下，未能与具体的“退出”按键动作绑定。
- [b] **循环扫描导致高频擦写：** 程序只要停留在参数界面，按键扫描函数就会以每 20 毫秒（`Key_Slow_Down < 20`）的频率执行一次这一块代码，导致系统疯狂向 EEPROM 写入数据。
- [c] **逻辑违规与硬件隐患并存：** 这不仅导致了实时更新覆盖了旧数据（判题报错），更严重的是，EEPROM 的擦写寿命一般在 10 万次左右，这种 1 秒钟写 50 次的逻辑漏洞，会在半小时内将真实芯片彻底烧毁。
- **解决方案：**
- [a] **界面内仅限草稿操作：** 清理 `if(Seg_Slow_Mode==1)` 中的越界逻辑，在该分支内只保留按键 S16 和 S17 对草稿变量 `Volt_Para_Sketch` 的加减运算。
- [b] **重构退出瞬间的触发逻辑：** 捕捉按下界面切换键（`S12`）的那一瞬间。在 `if(Key_Down==12)` 的判定体内加入条件拦截：如果当前处于参数界面（`Seg_Slow_Mode == 1`），则利用这唯一的一次下降沿，执行草稿转正并单次写入 EEPROM，随后再执行界面自增切换。

## 坑 36：界面进入与退出逻辑的认知误区

> 来源：[071_蓝桥杯单片机第二十五周周报（二）.md](./单片机答疑数据库/071_蓝桥杯单片机第二十五周周报（二）.md)
> 周期：蓝桥杯单片机第二十五周周报（二）

### 整理记录

- **问题描述：** 学员在实现“退出参数界面时保存数据”的功能时，错误地将保存动作写在了当前界面扫描分支的物理最末尾，导致程序停留在此界面时一直在不断执行保存操作。
**问题背景：** 蓝桥杯多界面切换系统（如第十一届省赛），题目要求在发生“退出界面”这一动作时，单次执行某项操作（如写入 EEPROM 存盘）。
**问题分析：** 这是一个典型的由缺乏“状态机事件驱动”思维导致的架构误区：
- [a] **主观脑补代码执行流：** 学员的主观逻辑是“单片机代码是从上往下执行的，界面的顺序是 0、1、2 循环。既然要退出 1 进入 2，那我把退出时的存盘代码写在界面 1 那个 `if` 块的最后面，就等同于退出了”。
- [b] **混淆了“状态停留”与“切换瞬间”：** 单片机的 `while(1)` 是一直在高速循环的。只要没有按下切换按键，界面状态（`Seg_Slow_Mode`）就会一直等于 1。程序每次循环都会扫过这个 `if` 块的末尾，把原本该只执行一次的“退出动作”，变成了每隔几十毫秒执行一次的“驻留动作”。
**解决方案：**
- [a] **纠正认知思维：** 明确向学员强调：“代码写在括号的最后面”绝对不等于“退出界面”。界面的退出不是一个**物理位置**，而是一个由按键触发的**瞬间动作（事件）**。
- [b] **重构代码位置：** 存盘动作必须跟“引发切换的按键事件”绑定。在处理按键（如按下 S12 切换界面）的代码逻辑中，先判断当前是否身处界面 1，如果是，则执行存盘，**然后**再将界面变量 `Seg_Slow_Mode` 加 1。这才是真正的“退出界面并保存”。

## 坑 37：按键切换模式跳跃（连续使用 if 导致的逻辑穿透）

> 来源：[071_蓝桥杯单片机第二十五周周报（二）.md](./单片机答疑数据库/071_蓝桥杯单片机第二十五周周报（二）.md)
> 周期：蓝桥杯单片机第二十五周周报（二）

### 整理记录

- **问题描述：** 学员反馈在使用按键切换系统界面时，发生“跳步”现象，即只能在其中两个模式之间来回切换，中间的某个界面（如时间参数界面）根本进不去。
**问题背景：** 单片机多界面菜单系统，通常利用一个状态变量（如 `Seg_Mode`）记录当前所处的界面，并在按键按下时修改该变量，以实现菜单的轮询切换（如 0 -> 1 -> 2 -> 0）。
**问题分析：** 结合聊天记录，这是一个非常典型且高频的 C 语言基础控制流漏洞（条件穿透）：
- [a] **连续的独立判断：** 学员在按键状态机的处理逻辑中，使用了多个平级的 `if` 语句来切换状态（例如：`if(mode==0) mode=1; if(mode==1) mode=2;`）。
- [b] **状态被瞬间二次篡改：** 当 `mode` 初始为 0 时，第一个 `if` 条件成立，`mode` 被赋值为 1。程序紧接着往下执行，遇到第二个 `if(mode==1)`。由于此时 `mode` 刚刚被改成了 1，这个条件自然也成立了，于是 `mode` 瞬间又被改成了 2。
- [c] **视觉表现为“跳步”：** 导致用户虽然只按了一次按键，但状态变量在单次循环的微秒级时间内连续跨越了多个值，中间状态被彻底跳过，肉眼看来就像是那个界面“坏掉进不去”了一样。
**解决方案：**
- [a] **使用互斥分支：** 引导学员将连续的平级 `if` 改为互斥的 `if ... else if ... else ...` 结构，或者使用 `switch ... case`。这样一旦上方的条件命中并执行完毕，就会直接跳出整个判断块，避免下方条件被连锁触发。
