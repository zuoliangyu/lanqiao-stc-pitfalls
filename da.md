## 坑 1：整数除法导致小数精度丢失

> 第十四届国赛 · 发现者：Water

### 现象

DA 输出值全部偏低或异常，明显与预期不符。

### 原因

调用 `da_mix_10x(uchar)` 时直接用整数除以 10，C 语言整数除法会截断小数部分：

```c
// 错误写法：结果为整数，小数全部丢失
da_mix_10x(value / 10);
```

### 解决方法

强制转换为浮点数再做除法：

```c
// 正确写法：保留小数部分
da_mix_10x(value / 10.0f);
```

### 举一反三

只要参与除法的两个操作数都是整型，结果就是整型。需要浮点结果时，至少让一侧是浮点数（写 `10.0f` 或显式强转 `(float)value`）。

## 坑 2：uint 类型算术下溢导致 DA 值异常

> 第十四届国赛 · 发现者：Water

### 现象

DA 输出值在某些测评点严重偏大（如本应输出 -50 相关结果，实际输出接近 65486 对应的错误值）。

![5ead2a2992da6622f785c43a5dbaef71](https://raw.githubusercontent.com/zuoliangyu/TuChuang/master/20260317192753645.png)

### 原因

一次函数取点时选了 `(90, 5)`，公式为 `y = k * (x - 90) + 5`。

测评时 x 的实际值往往小于 90，而 x 是 `unsigned int` 类型，`x - 90` 发生算术下溢：

```c
unsigned int x = 40;
// 错误：40 - 90 在 uint 下不是 -50，而是 65486
y = k * (x - 90) + 5;
```

### 解决方法

**方法一（推荐）**：取点时选 x 轴最小值那个点，保证 `x - x0` 永远为正：

```c
// 取 (10, Vol)，测评时 x >= 10，不会下溢
y = k * (x - 10) + Vol;
```

**方法二**：对 x 强制转换为有符号整型：

```c
y = k * ((int)x - 90) + 5;
```

### 举一反三

`uint` / `uchar` 类型做减法时，只要被减数可能小于减数，就会下溢。常见于：

- 一次函数插值取点
- 传感器量程映射
- 时间差计算（如 TH/TL 拼合后做差）

**建议：一次函数取点时，永远取 x 轴最小的那个已知点。**

## 坑 3：ADC 满量程时缺少钳位保护导致 DA 输出边界异常

> 第十三届国赛 · 发现者：左岚

### 现象

传感器 ADC 读数接近满量程时，DA 输出值不符合题目要求的最大值钳位，或输出行为与预期不一致。

### 原因

ADC 原始值 ≥ 255 时，直接用浮点公式计算湿度，数学上不会超过 100，但缺少对满量程的明确钳位，边界语义不清晰，实测时存在隐患：

```c
// 缺少钳位保护：依赖浮点公式隐式限幅，满量程行为未明确定义
Shidu = temp * 20 / 51.0f;
```

### 解决方法

在 ADC 读值后加显式钳位判断：

```c
// 正确写法：ADC 满量程时强制钳位为 100
if (AD_3_Data >= 255)
    Shidu = 100;
else
    Shidu = AD_3_Data * 20 / 51.0f;
```

### 举一反三

- ADC 做量程映射时，务必同时处理满量程（255）和零点（0）两个边界
- 钳位保护不仅防止数值越界，也让代码对边界行为的意图一目了然
- 题目若规定某阈值以上输出固定值，应优先用 `if` 显式处理，而非依赖数学公式隐式限幅

## 坑 7：DAC 界面显示与按键逻辑误绑 ad_mode 导致测评失败

> 来源：[062_蓝桥杯单片机第二十二周周报.md](./单片机答疑数据库/062_蓝桥杯单片机第二十二周周报.md)
> 周期：蓝桥杯单片机第二十二周周报
> 关联模块：综合赛题 / 系统 / 按键 / C51 语法 / 基础逻辑

### 整理记录

- **问题描述：** 板子上手动测试现象正确，但 4T 自动测评 DAC 相关步骤全部报错——数码管始终显示 A 102，按键加减无效
- **问题背景：** 第十五届省赛题目
- **问题分析：** 存在两处逻辑错误：
- [a] **显示内容搞混：** DAC 界面（`seg_mode == 1`）按 `ad_mode` 区分了两种显示——自动模式显示 `da_show`，手动模式显示 `ad`。但题目要求 DAC 界面始终显示 DAC 数字量（即 `ad`），与输出模式无关。测评时处于自动模式，界面显示的是温度自动计算出的 102，而非初始值 100
- [b] **按键加了多余判断：** S16/S17 调整 DAC 数字量时加了 `if(ad_mode)` 判断，导致只有手动模式下才能修改 `ad` 值。题目要求只要在 DAC 界面，按键就能调整数字量，与输出模式无关。测评时按加减键值根本不变
- **解决方案：** DAC 界面的"显示"和"按键调整"均不应受 `ad_mode` 限制。`ad_mode` 仅决定实际 DAC 输出走自动还是手动，不影响界面显示内容和按键操作。去掉显示和按键中对 `ad_mode` 的条件判断即可

## 坑 8：DAC输出电压测量不准与if...else if逻辑问题(十五届第一次省赛题)

> 来源：[065_蓝桥杯单片机第二十二周周报（四）.md](./单片机答疑数据库/065_蓝桥杯单片机第二十二周周报（四）.md)
> 周期：蓝桥杯单片机第二十二周周报（四）
> 关联模块：C51 语法 / 基础逻辑

### 整理记录

* **问题描述**:DAC输出电压测量不准确,且if...else if语句顺序互换导致结果不同。
* **问题背景**:小鹅通课程十五届第一次省赛题,学员在处理数据判断逻辑时遇到条件覆盖问题。
* **问题分析**:
* C语言中if...else if结构存在逻辑覆盖问题
* 当多个条件可能同时满足时,只会执行第一个匹配的分支
* 数据判断逻辑设计不合理
* **解决方案**:
* 改用单if语句进行独立判断,避免条件覆盖
* 确保每个判断条件互斥,或使用独立if语句
```c
// 修改前(错误):
if (voltage > 3.0) {
// 处理高电压
} else if (voltage > 2.0) {
// 当voltage=3.5时,永远不会执行到这里
}
// 修改后(正确):
if (voltage > 3.0) {
// 处理高电压
}
if (voltage > 2.0 && voltage <= 3.0) {
// 明确条件范围
}
```

## 坑 9：整数除法精度丢失导致 DA 输出电压卡在 1V

> 来源：[066_蓝桥杯单片机第二十三周周报.md](./单片机答疑数据库/066_蓝桥杯单片机第二十三周周报.md)
> 周期：蓝桥杯单片机第二十三周周报

### 整理记录

- **问题描述：** 模式1下温度≥20°C 时，DA 输出电压始终卡在 1V，不随温度线性变化
- **问题背景：** 第十二届省赛（第一批），DA 线性输出模式
- **问题分析：** `dat = (Temperature_10x * 3 / 200 - 2) * 51;` 中 `Temperature_10x * 3 / 200` 是整数除法，会截断小数。手动验算：20.0°C → `200*3/200=3`，21.0°C → `210*3/200=3`，25.0°C → `250*3/200=3`，26.6°C → `266*3/200=3`，结果在 20~26.6°C 范围内 `/200` 截断后始终为 3，`3-2=1`，`1*51=51`，电压恒为 1V。直到 26.7°C（`267*3/200=4`）才跳变到 2V
- **解决方案：** 调整运算顺序，先乘大再除，将 `*51` 提前合并：
```c
// 原来（精度丢失严重）
dat = (Temperature_10x * 3 / 200 - 2) * 51;
// 修改后（先乘大再除，保留精度）
dat = (long)Temperature_10x * 153 / 200 - 102;
// 等价于 (T*3*51)/200 - 2*51，加 (long) 防溢出
```

## 坑 10：DAC 反算截断误差导致电压显示差 0.01V

> 来源：[066_蓝桥杯单片机第二十三周周报.md](./单片机答疑数据库/066_蓝桥杯单片机第二十三周周报.md)
> 周期：蓝桥杯单片机第二十三周周报

### 整理记录

- **问题描述：** 4T 测评 68.6/70 分，35 个测试点通过 34 个，#32 未通过：期望 A 3.10，实际显示 A 3.09
- **问题背景：** 第十二届省赛（第一批）
- **问题分析：** 学员先算 DAC 寄存器值 `dat`，再用 `DA_100x = dat * 100 / 51` 反算电压用于数码管显示。`dat * 100 / 51` 是整数除法反算，存在截断误差，DAC 寄存器的量化误差直接传递到了显示值上。正确做法应该"先算显示值，再量化输出"，显示值从源头直接计算，不经过反算
- **解决方案：** 改为先算电压值 `DA_100x`（直接精确），再正向量化成寄存器值：
```c
// 原来（先量化再反算 → 截断误差影响显示）
dat = 255;
Da_Write(dat);
DA_100x = dat * 100 / 51;
// 改后（先算电压再量化 → 显示精确）
DA_100x = 500;
dat = DA_100x * 51 / 100;
Da_Write(dat);
```
同时将温度精度从 ×10 改为 ×100 以匹配两位小数显示要求

## 坑 11：Da_Write 函数末尾缺少 I2CStop() 调用

> 来源：[066_蓝桥杯单片机第二十三周周报.md](./单片机答疑数据库/066_蓝桥杯单片机第二十三周周报.md)
> 周期：蓝桥杯单片机第二十三周周报

### 整理记录

- **问题描述：** 学员发现 `Da_Write` 函数末尾没有调用 `I2CStop()`，疑问为什么不写 Stop 也能正常工作
- **问题背景：** PCF8591 DAC 输出函数编写
- **问题分析：** PCF8591 的 DAC 输出是"写入即生效"的，数据字节发送完并收到 ACK 后，模拟电压就已经输出到 AOUT 引脚。根据 NXP 官方数据手册："In the write mode, a data transfer is terminated by sending either a stop condition or the start condition of the next data transfer." 即 Stop 不是必须的，下一次调用 `Da_Write` 或其他 I2C 函数时的 `I2CStart()`（Repeated Start）会隐式结束上一次通信。但不发 Stop 意味着总线一直被占用（SCL 保持低电平），如果总线上还有其他从设备则无法被访问
- **解决方案：** 在 `Da_Write` 末尾补上 `I2CStop()`，释放总线，养成良好的 I2C 编程习惯：
```c
void Da_Write(unsigned char dat) {
I2CStart();
I2CSendByte(0x90);
I2CWaitAck();
I2CSendByte(0x41);
I2CWaitAck();
I2CSendByte(dat);
I2CWaitAck();
I2CStop(); // 补上，释放总线
}
```

## 坑 12：第十三届省赛（第二次）页面状态机错误+DAC 线性公式整数除法

> 来源：[069_蓝桥杯单片机第二十四周周报（二）.md](./单片机答疑数据库/069_蓝桥杯单片机第二十四周周报（二）.md)
> 周期：蓝桥杯单片机第二十四周周报（二）
> 关联模块：综合赛题 / 系统

### 整理记录

- **问题描述：** 4T 测评多项失分，界面切换后显示内容与预期不符，DAC 输出在中间段异常
- **问题背景：** 第十三届省赛第二次题目（电压测量 + 超声测距 + 参数设置综合题）
- **问题分析：** 该问题涉及多处逻辑缺陷：
- [a] **界面切换顺序错误：** 实际为 `U→P→L→U`，题目要求 `U→L→P→U`，导致切换后显示和 LED 全部错位
- [b] **参数界面和测距界面显示代码写反：** `case 1` 写了参数界面、`case 2` 写了测距界面，与页面定义不统一
- [c] **S5/S6/S7 有效界面判断错误：** 按键有效条件绑定了错误的 Mode 值，页面改正后按键也需同步修改
- [d] **DAC 中间段线性公式整数除法为 0：** `1/15*(Wave-20)` 中 `1/15` 为整数除法结果为 0，导致 20~80cm 区间 DAC 输出始终约为 1V
- [e] **测距未启动时 DAC 未清零：** 题目要求未启动测距时输出 0V，程序缺少 `Da_Write(0)`
- [f] **参数提交逻辑多加了题目未要求的限制：** 只有 `AD_up > AD_down` 才保存，但题目要求退出参数界面即生效
- **解决方案：**
- [a] 页面定义统一为 `0=U, 1=L, 2=P`，S4 切换顺序改为 `U→L→P→U`
- [b] `Seg_proc()` 中 `case 1` 对应测距界面 L、`case 2` 对应参数界面 P
- [c] S5/S6/S7 有效条件同步改为仅在参数界面（`Mode_show == 2`）有效
- [d] 线性公式避免整数除法，改为先乘后除：如 `(Wave - 20) * 4 / 60 + 1`（单位 V）
- [e] 测距未启动时显式执行 `Da_Write(0)`
- [f] 去掉额外的上下限比较限制，退出参数界面时直接生效

## 坑 13：第十二届省赛（第一次）DAC 输出全部为 1.0V——超声波距离单位不匹配

> 来源：[069_蓝桥杯单片机第二十四周周报（二）.md](./单片机答疑数据库/069_蓝桥杯单片机第二十四周周报（二）.md)
> 周期：蓝桥杯单片机第二十四周周报（二）
> 关联模块：超声波

### 整理记录

- **问题描述：** 4T 测评 61 项通过、5 项未通过，全部集中在 DAC 输出，实际输出均为 1.0V，期望值分别为 2.3/5.0/2.0/5.0/4.0V
- **问题背景：** 第十二届省赛第一次题目（超声波测距+DAC 输出）
- **问题分析：** 超声波底层函数返回的距离单位是厘米（如 20、80），但 DAC 逻辑判断中使用的阈值是毫米（2000、8000）。实际距离值永远远小于判断阈值，始终落入第一个分支 `Distance >= 0 && Distance <= 2000`，DAC 输出固定为 `DA_100x = 100`（即 1.0V）：
```c
// 错误：阈值用毫米，实际值是厘米
if((Distance >= 0) && (Distance <= 2000))
DA_100x = 100;
else if((Distance > 2000) && (Distance <= 8000))
DA_100x = (unsigned int)(Distance * 100 - 2000) * 10 / 150 + 100;
else if(Distance > 8000)
DA_100x = 500;
```
- **解决方案：** 将判断阈值统一为厘米，与超声波底层返回值一致：
```c
if((Distance >= 0) && (Distance <= 20))
DA_100x = 100;
else if((Distance > 20) && (Distance <= 80))
DA_100x = (unsigned int)(Distance - 20) * 400 / 60 + 100;
else if(Distance > 80)
DA_100x = 500;
```

## 坑 14：第十四届国赛 继电器判定错误+DAC 输出始终 1.0V

- **问题描述：** 4T 测评中继电器控制结果与标准答案不一致（应断开却闭合，如 #28/#32），DAC 输出始终为 1.0V（期望 2.5/3.5V，如 #66/#67/#68）
- **问题背景：** 第十四届国赛题目（超声测距+温度+记录回放+DAC 输出综合题）
- **问题分析：** 存在两类核心问题：
- [a] **继电器温度判断边界条件错误：** 将"严格小于"写成了"小于等于"，温度等于参数值时继电器行为与题意相反
- [b] **继电器温度比较量纲不一致：** 放大 10 倍后的温度值 `Temperature_10x` 与未放大的温度参数 `CanShu_Buf[1]` 直接比较，应写为 `Temperature_10x < CanShu_Buf[1] * 10`
- [c] **DAC 记录对象错误：** 没有按题意记录 6 秒内的距离序列（每 200ms 采一次，共 30 个点），而是只保存了一次起始距离
- [d] **DAC 输出模型错误：** 输出时计算的是前后距离差值，而非对记录数组逐点回放并做分段映射
- [e] **DAC 电压变量使用整型导致截断：** 用整型变量保存 DAC 电压值，小数部分被截断，无法输出 2.5V、3.5V 等非整数电压
- **解决方案：**
- [a] 继电器判定条件中温度比较改为严格小于 `<`
- [b] 统一温度比较单位：`Temperature_10x < CanShu_Buf[1] * 10`
- [c] 改为定时采集距离数组：每 200ms 记录一次距离值到 `Record_Buf[]`，6 秒内最多记录 30 个点
- [d] 输出阶段改为按记录数组逐点回放，每 200ms 取出一个点，按分段规则映射 DAC 电压（<10cm 输出下限、>90cm 输出 5V、中间线性映射）
- [e] DAC 中间计算变量改为浮点类型 `float`，避免截断误差

## 坑 15：DA 录制回放逻辑不完整与组合键恢复出厂识别不稳

> 来源：[070_蓝桥杯单片机第二十五周周报.md](./单片机答疑数据库/070_蓝桥杯单片机第二十五周周报.md)
> 周期：蓝桥杯单片机第二十五周周报
> 关联模块：综合赛题 / 系统 / 按键 / C51 语法 / 基础逻辑

### 整理记录

- **问题描述：** 4T 测评中多项失败：第 54~55 项 DA 输出变为 `0.0`；第 57~65 项恢复出厂失败（`P1 50` 变成 `P1 60`），后续界面连锁跑偏
- **问题背景：** 该题涉及组合键长按恢复出厂设置、距离采样录制为 30 组数据、DA 按规则回放输出
- **问题分析：** 学员代码存在两大类问题，共涉及两个模块的主逻辑未对齐参考实现。
- **DA 录制回放模块（4 个缺陷）：**
- [a] 记录数组仅开 12 个数据点，不够参考实现要求的 30 组
- [b] 测距任务周期过慢，采样节拍不符参考实现
- [c] DA 回放步进时间为 500ms，应为 200ms
- [d] 按下输出键后直接开始输出，未判断"记录是否已完成"
- **组合键恢复出厂模块（3 个缺陷）：**
- [e] **组合键扫描不稳定：** 单键扫描与组合键扫描在同一循环中，两个键同时按下时程序先识别单键（当作加 10 按键），再识别组合键，导致长按恢复默认还没触发参数值已被改掉
- [f] **恢复默认复位不完整：** 只改了部分主参数，界面状态、控制副本、模式变量未一起恢复，导致参数看似恢复了但界面仍停留在恢复前状态
- [g] **连锁效应：** 第 57 项恢复出厂失败后，后续按键全部建立在错误界面上运行，第 57~65 项连锁出错
- **解决方案：**
- [a] 将组合键扫描改为 `else if` 链结构，确保组合键 99 优先于单键 8 和 9：
```c
if((P33==0) && (P32==0)) temp = 99;
else if(P33==0) temp = 8;
else if(P32==0) temp = 9;
```
- [b] 封装统一的 `Sys_Default()` 恢复函数，同时恢复界面状态（`Seg_Disp_Mode`、`Distance_Mode`、`Pare_Mode`、`Fa_Mode`）、主参数、控制副本（`_Ctrl` 系列变量）
- [c] 长按恢复改为 "持续按压计时、松手取消" 模式：`Key_Down==99` 计时开始，满 2 秒执行 `Sys_Default()`；`Key_Val!=99` 时重置所有标志
- [d] 重做 DA 数据流：记录数组扩容至 30 个；新增 `Record_Finish_Flag` 判断记录完成；输出启动条件改为 `Key_Down==9 && Record_Finish_Flag==1`
- [e] DA 回放步进计时改为 200ms 周期（`Timer_200ms == 200`），不再使用 500ms
- [f] 任务调度节拍调整：Key_Proc 15ms、Get_Temperature 100ms、Get_Distance 200ms、AD_DA 90ms、Seg_Proc 50ms
- **经验教训：** 遇到多项连续失败时，先找"第一项先跑偏"的根因，不要逐项单独修；组合键扫描中组合值优先级必须高于单键

## 坑 16：第十五届省赛温度题DAC数字量修改限制与LED事件状态保持错误

> 来源：[070_蓝桥杯单片机第二十五周周报.md](./单片机答疑数据库/070_蓝桥杯单片机第二十五周周报.md)
> 周期：蓝桥杯单片机第二十五周周报
> 关联模块：温度 / 时钟 / 传感器 / LED

### 整理记录

- **问题描述：** 4T 测评主要失分：DAC 数字量界面加减操作受限、L4 闪烁节奏和持续时间不符、L2/L3 指示灯未按"触发后保持 2s"工作
- **问题背景：** 第十五届省赛题目，涉及温度采集+校准、DAC 自动/手动输出、温度变化事件（升温/降温/突变）LED 指示
- **问题分析：** 核心是 3 类问题：
- [a] **DAC 数字量加减被多余条件限制：** 原程序在 DAC 数字量界面下，用 `if(DA_Mode==1)` 限制了加减操作仅在手动模式下生效，4T 测评在 DAC 界面测试时不接受这个额外限制，数值加减测试直接失败
- [b] **L4 突变闪烁状态被提前清除：** 温差超过 1℃ 时置位了 `Temperature_change_Flag`，但后续温差不再超过 1℃ 时 `else` 分支立刻将其清零，导致 L4 根本无法稳定闪烁满 3 秒，属于"条件成立时亮"和"事件触发后保持"两种逻辑混淆
- [c] **L2/L3 升降温保持逻辑两重错误：** 计时变量使用 `unsigned char`（最大值 255），无法完成 2000ms 计时；温度不再继续上升或下降时立刻把标志清零，L2/L3 只亮很短时间就熄灭
- **解决方案：**
- [a] 去掉 DAC 数字量加减上的 `DA_Mode==1` 限制，界面下按键加减直接生效：
```c
// 修改前：if(DA_Mode==1) { dat_2=(dat_2>=250)?255:(dat_2+5); }
// 修改后：dat_2=(dat_2>=250)?255:(dat_2+5);
```
- [b] L4 改为触发型结构：温差超过 1℃ 时触发 L4 闪烁并重置闪烁计时和总时长计时，后续不再在普通采样中提前清除闪烁状态，由定时过程完成 3s 后自动熄灭
- [c] L2/L3 计时变量改为 `unsigned int`；温度上升/下降触发时只设置标志和重置计时，不再在 `else` 分支中清除标志，由计时结束后自动熄灭
- **经验教训：** "当前条件是否成立"≠"状态是否应该保持"；遇到"亮 2s""闪烁 3s"类需求必须用"触发标志 + 定时清除"结构；计时变量位宽必须和计时时长匹配

## 坑 17：温度题DAC温控模式默认错误、温度单位错用与短按长按复用风险

> 来源：[070_蓝桥杯单片机第二十五周周报.md](./单片机答疑数据库/070_蓝桥杯单片机第二十五周周报.md)
> 周期：蓝桥杯单片机第二十五周周报
> 关联模块：综合赛题 / 系统 / 温度 / 时钟 / 传感器 / 按键

### 整理记录

- **问题描述：** 4T 测评中 DAC 温度控制项目失败：温度 30℃、校准值 -5 时期望输出 3.5V，实际为 0.0；温度 13℃、校准值 -5 时期望输出 2V，实际为 0.0。温度显示和校准加减均正常
- **问题背景：** 第十五届省赛题目，DAC 自动温控输出与手动控制切换，温度→电压公式：10℃ 对应 2V，40℃ 对应 5V，区间线性插值
- **问题分析：** 失分不在显示，而在 DAC 控制逻辑有两层错误：
- [a] **DAC 工作模式默认状态不合理：** 默认是手动模式而非温控模式，测评流程未先切换到温控模式时，温控公式代码根本不会被执行，DAC 输出始终为 0.0V
- [b] **温控公式使用错误的温度单位：** 温度保存为"10 倍值"（25℃→250），但 DAC 计算时直接拿 250 与 10/40 比较。25℃ 实际判断的是 250，8℃（80）被误判为 ">=40" 输出 5V。公式完全失真
- [c] **短按/长按功能复用风险：** 同一按键同时用于 DAC 模式切换、长按计时、继电器控制，处理位置不统一，测评脚本时序与代码状态机不一致时容易模式切换失败
- **解决方案：**
- [a] DAC 默认模式改为温控模式（`Dac_Work_Mode = 0`），符合测评预期
- [b] 温度计算时统一使用"实际温度值"而非 10 倍值：`rd_temperature()` 加校准后得到真实摄氏温度，直接与 10℃/40℃ 比较：
```c
if (temp <= 10.0f)
Dac_Digit_Temperature = 2 * 51; // 2V
else if (temp >= 40.0f)
Dac_Digit_Temperature = 5 * 51; // 5V
else
Dac_Digit_Temperature = (unsigned char)((temp / 10.0f + 1.0f) * 51.0f); // 线性插值
```
- [c] 统一 DAC 输出函数：分别算出温控值和手动值，再根据模式选择写入，确保任何模式下都执行 `Da_Write()`
- [d] 短按/长按按"按下计时、松开判断"结构统一处理，避免功能互相干扰
- **经验教训：** 变量存储格式（10 倍值/真实值）在整个计算链中必须保持一致，不要"存的时候放大、算的时候忘记还原"；DAC/PWM 等输出类功能不管模式如何都应周期性调用 `Da_Write()`，用变量决定输出值而非决定要不要输出

## 坑 18：十五届省赛真题长按短按混淆，DAC错误

> 来源：[071_蓝桥杯单片机第二十五周周报（二）.md](./单片机答疑数据库/071_蓝桥杯单片机第二十五周周报（二）.md)
> 周期：蓝桥杯单片机第二十五周周报（二）
> 关联模块：按键

### 整理记录

- **问题描述：** 测评用例中多项功能错误，包括：继电器锁定/解锁状态完全颠倒、继电器无法正常控制；DAC 在自动模式下输出电压错误（本应为 4.2V 却输出 1.7V）；L1 指示灯逻辑异常。
- **问题背景：** 蓝桥杯单片机设计与开发第十五届省赛真题练习（涉及 DS18B20 温度采集、PCF8591 DAC 输出、继电器及按键长短按复合控制）。
- **问题分析：** 该问题主要是由底层按键状态机设计缺陷导致的功能耦合，非单一错误：
- [a] **按键长短按逻辑冲突（核心死锁）：** S13 既负责短按（切换 DAC 模式）又负责长按（切换继电器锁定）。原代码将短按写在 `Key_Down`（按下瞬间触发），导致每次长按必然先触发一次短按，DAC 模式与继电器锁定状态相互干扰，系统逻辑完全乱套。
- [b] **长按清零机制存在死锁漏洞：** 原代码中定时器内的长按清零条件苛刻（必须按满 1.5 秒且松开才清零）。若未按满 1.5 秒松手，标志位 `Long_Press` 无法清零，导致时间变量在后台持续累加，后续随意触碰都会被误判为长按。
- [c] **依赖边沿打标签的稳定性差：** 依赖 `Key_Down` 置位和 `Key_Up` 复位的写法，一旦主循环受其他程序阻断（错过了松手瞬间），状态机就会彻底卡死。
- [d] **指示灯逻辑绑定错误：** L1 本应反映 DAC 的控制模式，代码中却错误地绑定了数码管的界面标志位。
- **解决方案：**
- [a] **长短按在时序上彻底剥离：** 将 S13 的短按执行逻辑移至按键松开时（`Key_Up == 13`）判断；长按则在按下期间由定时器独立计时。
- [b] **重构为“状态驱动”模型：** 定时器中放弃标签判断，直接判断按键实时状态（`Key_Old == 13`）。只要判定手松开（走 `else` 分支），无条件强制清零长按计时器和标志位。
- [c] **修正短按触发条件：** 在 `Key_Up` 触发时，增加对长按标志位 `Long_Press` 的判断。只有当没有触发过长按时（`Long_Press == 0`），松手才执行短按逻辑，随后立刻清零该标志位。
- [d] **修正指示灯绑定源：** 修改 `Led_Proc`，将 `ucLed[0]` 直接绑定为 `(Da_Mode == 0)`，确保其准确反映温度控制模式。
