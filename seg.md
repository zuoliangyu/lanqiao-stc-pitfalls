## 坑 1：数码管十位分解缺少取模保护导致显示空白

> 第十三届国赛 · 发现者：左岚

### 现象

被显示的值为 100 时，数码管十位熄灭或显示异常，但个位和百位正常。

### 原因

十位只用了除法，没有对结果取模，当值为 100 时 `100 / 10 = 10`，而解码表中索引 10 对应空白字符：

```c
// 错误写法：缺少 %10，值为 100 时十位索引为 10（空白）
Seg_Buf[6] = Shidu / 10;
Seg_Buf[7] = Shidu % 10;
```

### 解决方法

十位及以上的位分解一律加 `% 10`：

```c
// 正确写法
Seg_Buf[6] = Shidu / 10 % 10;
Seg_Buf[7] = Shidu % 10;
```

### 举一反三

- 只要被显示的值可能达到 10 的整次幂（如 100、1000），高位就必须取模
- 三位数标准拆法：`百位 = n / 100`，`十位 = n / 10 % 10`，`个位 = n % 10`
- 显示前加钳位或断言，确保值在预期范围内，能提前暴露此类问题

## 坑 41：新模板数码管小数点编码技巧不理解

> 来源：[044_蓝桥杯单片机第十二周周报（二）.md](./单片机答疑数据库/044_蓝桥杯单片机第十二周周报（二）.md)
> 周期：2025年12月9日 — 2025年12月14日

### 整理记录

* **问题描述**：

  + 学员不理解新模板中 `Seg_Buf[Seg_Pos] > 20` 的判断逻辑
  + 不确定if-else语句的作用
  + 不确定这个机制是和数码管硬件原理有关还是seg_disp函数设计有关
  + 询问按键扫描中防串口冲突的机制（问题未完整描述）
* **问题背景**：学习新模板代码，研究数码管显示和按键扫描底层实现。
* **问题分析**：

  + 这是一种**小数点编码技巧**，与硬件原理无关，是软件层面的处理方式
  + 通过给字符ASCII值加上偏移量来标识是否需要显示小数点
  + 利用 `> 20` 判断来区分普通字符和带小数点的字符
* **解决方案**：

  1. **小数点编码原理详解**：

  ```
   // 核心思路：给需要显示小数点的字符加上偏移量
   if(Seg_Buf[Seg_Pos] > 20) {
       // ASCII值大于20，说明加了偏移量，需要显示小数点
       // 1. 显示小数点（点亮DP段）
       // 2. 减去偏移量还原成原字符
       // 3. 查表得到对应的数码管段码
   } else {
       // ASCII值 <= 20，正常显示，不带小数点
   }
  ```

  1. **编码技巧说明**：

     + **正常字符**：ASCII值范围 0-19（数字0-9，字母A-F等）
     + **需要小数点的字符**：给ASCII值加上偏移量（如加44），使其 &gt; 20
     + **识别机制**：通过判断 &gt; 20 来识别是否需要显示小数点
     + **还原字符**：减去偏移量得到原字符，再查段码表
  2. **使用示例**：

  ```
   // 显示 "12.34"
   Seg_Buf[0] = 1;      // 显示 '1'
   Seg_Buf[1] = 2 + 44; // 显示 '2.'（2的值 + 偏移量44，> 20）
   Seg_Buf[2] = 3;      // 显示 '3'
   Seg_Buf[3] = 4;      // 显示 '4'
  ```

  1. **按键防串口冲突机制（简述）**：

     + 蓝桥杯板子的矩阵键盘使用P3和P4口，P3口同时复用为串口引脚
     + 扫描前将所有列线拉高（`P44=1; P42=1; P35=1; P34=1;`），确保初始状态
     + 快速检测行线，避免串口通信时的电平干扰被误认为按键
  2. **学习建议**：

     + 这个技巧在左岚的考点精讲课程中有详细展示
     + 需要先理解数码管硬件原理（共阴/共阳，段码表）
     + 这是一种巧妙的软件处理方式，避免单独维护小数点标志位数组
* **知识点总结**：

  + **数码管小数点控制**：通过ASCII偏移量巧妙实现，比单独维护标志位更优雅
  + **矩阵键盘扫描**：列线拉高 → 逐行扫描 → 识别按键位置
  + **引脚复用冲突**：多个外设共用GPIO时需要防止相互干扰
  + **代码可读性**：选择20作为阈值是因为常用字符编码都小于20，逻辑清晰

> 来源：[046_蓝桥杯单片机第十三周周报.md](./单片机答疑数据库/046_蓝桥杯单片机第十三周周报.md)
> 周期：2025年12月9日 — 2025年12月14日

### 整理记录

* **问题描述**：

  + 学员不理解新模板中 `Seg_Buf[Seg_Pos] > 20` 的判断逻辑
  + 不确定if-else语句的作用
  + 不确定这个机制是和数码管硬件原理有关还是seg_disp函数设计有关
  + 询问按键扫描中防串口冲突的机制（问题未完整描述）
* **问题背景**：学习新模板代码，研究数码管显示和按键扫描底层实现。
* **问题分析**：

  + 这是一种**小数点编码技巧**，与硬件原理无关，是软件层面的处理方式
  + 通过给字符ASCII值加上偏移量来标识是否需要显示小数点
  + 利用 `> 20` 判断来区分普通字符和带小数点的字符
* **解决方案**：

  1. **小数点编码原理详解**：

  ```
   // 核心思路：给需要显示小数点的字符加上偏移量
   if(Seg_Buf[Seg_Pos] > 20) {
       // ASCII值大于20，说明加了偏移量，需要显示小数点
       // 1. 显示小数点（点亮DP段）
       // 2. 减去偏移量还原成原字符
       // 3. 查表得到对应的数码管段码
   } else {
       // ASCII值 <= 20，正常显示，不带小数点
   }
  ```

  1. **编码技巧说明**：

     + **正常字符**：ASCII值范围 0-19（数字0-9，字母A-F等）
     + **需要小数点的字符**：给ASCII值加上偏移量（如加44），使其 &gt; 20
     + **识别机制**：通过判断 &gt; 20 来识别是否需要显示小数点
     + **还原字符**：减去偏移量得到原字符，再查段码表
  2. **使用示例**：

  ```
   // 显示 "12.34"
   Seg_Buf[0] = 1;      // 显示 '1'
   Seg_Buf[1] = 2 + 44; // 显示 '2.'（2的值 + 偏移量44，> 20）
   Seg_Buf[2] = 3;      // 显示 '3'
   Seg_Buf[3] = 4;      // 显示 '4'
  ```

  1. **按键防串口冲突机制（简述）**：

     + 蓝桥杯板子的矩阵键盘使用P3和P4口，P3口同时复用为串口引脚
     + 扫描前将所有列线拉高（`P44=1; P42=1; P35=1; P34=1;`），确保初始状态
     + 快速检测行线，避免串口通信时的电平干扰被误认为按键
  2. **学习建议**：

     + 这个技巧在左岚的考点精讲课程中有详细展示
     + 需要先理解数码管硬件原理（共阴/共阳，段码表）
     + 这是一种巧妙的软件处理方式，避免单独维护小数点标志位数组
* **知识点总结**：

  + **数码管小数点控制**：通过ASCII偏移量巧妙实现，比单独维护标志位更优雅
  + **矩阵键盘扫描**：列线拉高 → 逐行扫描 → 识别按键位置
  + **引脚复用冲突**：多个外设共用GPIO时需要防止相互干扰
  + **代码可读性**：选择20作为阈值是因为常用字符编码都小于20，逻辑清晰

## 坑 49：数码管高位熄灭（前导零消隐）无法实现

> 来源：[061_蓝桥杯单片机第二十一周周报（三）.md](./单片机答疑数据库/061_蓝桥杯单片机第二十一周周报（三）.md)
> 周期：蓝桥杯单片机第二十一周周报（三）

### 整理记录

- **问题描述：** 数码管高位熄灭功能不生效，高位始终显示 0 无法消隐
- **问题背景：** 频率显示数码管处理
- **问题分析：** 高位消隐循环中 `while(Seg_Buf[i]==0)` 将 `Seg_Buf[i]` 赋值为 10（熄灭码）。但循环变量 `i` 从 1 开始，第一次执行时 `Seg_Buf[1]` 被设为 10，下次再调用 `Seg_Task()` 时，`Seg_Buf[1]` 的值已经是 10（不是 0），`while` 条件立即不成立，直接跳出循环，后续高位全部无法消隐
- **解决方案：** 确保高位消隐逻辑在每次重新赋值 `Seg_Buf` 之后再执行，且循环起始索引应从被赋值数码管的最高位开始向低位遍历，保证每次消隐前 `Seg_Buf` 中存放的是本轮最新的数值而非上轮的熄灭码

## 坑 50：数码管闪烁无法实现（Seg_Slow 刷新周期过长）

> 来源：[062_蓝桥杯单片机第二十二周周报.md](./单片机答疑数据库/062_蓝桥杯单片机第二十二周周报.md)
> 周期：蓝桥杯单片机第二十二周周报

### 整理记录

- **问题描述：** 数码管应实现闪烁效果，但实际始终常亮，看不到闪烁
- **问题背景：** 日常课程练习
- **问题分析：** 定时器中断中 `Seg_Slow` 的归零阈值设为了 500，即 `if(++Seg_Slow == 500) Seg_Slow = 0;`。定时器 1ms 中断，`Seg_Slow` 每 500ms 才归零一次，`Seg_Proc()` 开头 `if(Seg_Slow) return;` 导致数码管内容每 500ms 才刷新一次。而闪烁依赖 `LED_Flag`（250ms 翻转），数码管刷新周期远大于闪烁周期，`Seg_Buf` 中的闪烁值（在显示值和熄灭码 10 之间切换）无法被及时刷新到数码管上，视觉上看不到闪烁效果
- **解决方案：** 将 `Seg_Slow` 的归零阈值从 500 改为 10，即 `if(++Seg_Slow == 10) Seg_Slow = 0;`，使数码管每 10ms 刷新一次，远快于 250ms 的闪烁周期，闪烁效果即可正常显示

## 坑 51：Key_Val/Key_Old 来回抖动（temp 未初始化 + Seg_Slow_Down 类型溢出）

> 来源：[062_蓝桥杯单片机第二十二周周报.md](./单片机答疑数据库/062_蓝桥杯单片机第二十二周周报.md)
> 周期：蓝桥杯单片机第二十二周周报

### 整理记录

- **问题描述：** 按下按键后 `Key_Val` 和 `Key_Old` 一直在 0 和按键序号之间来回跳变（如按1键，值在 0 和 1 之间反复抖动）
- **问题背景：** 日常课程练习（普中 A2 单片机）
- **问题分析：** 存在两处错误：
- [a] **temp 未初始化：** `Key_read()` 中 `unsigned char temp;` 没有赋初值 0。当没有按键按下时，四个 `if` 条件都不成立，`temp` 保持未初始化的随机值并返回，导致 `Key_Val` 不稳定，与 `Key_Old` 异或后产生虚假的 `Key_Down` 事件，表现为反复抖动
- [b] **Seg_Slow_Down 类型溢出：** `unsigned char Seg_Slow_Down;` 声明为 `unsigned char`（最大值 255），但中断中归零阈值设为 500。计数器永远到不了 500，`Seg_Slow_Down` 永远非零，`Seg_Proc()` 开头 `if(Seg_Slow_Down) return;` 始终成立，数码管刷新函数永远被跳过
- **解决方案：**
- [a] 将 `unsigned char temp;` 改为 `unsigned char temp = 0;`
- [b] 将 `Seg_Slow_Down` 类型改为 `unsigned int`，或将归零阈值改为 255 以内的合理值（如 10）

## 坑 52：seg_jiyi 在定时器 ISR 中被反复覆盖导致永远卡在模式3

> 来源：[066_蓝桥杯单片机第二十三周周报.md](./单片机答疑数据库/066_蓝桥杯单片机第二十三周周报.md)
> 周期：蓝桥杯单片机第二十三周周报

### 整理记录

- **问题描述：** 电压跌破阈值触发切换到采集界面（模式3）后，3 秒定时结束无法回到原界面，永远卡在模式3
- **问题背景：** AD 电压阈值触发界面切换功能，要求触发后显示采集界面 3 秒再恢复原界面
- **问题分析：** 学员将"记忆旧模式 + 切换新模式"的逻辑写在定时器 ISR 中（每 1ms 执行一次）：
```c
// 定时器 ISR 中（每1ms执行）
if(flag == 1) {
seg_jiyi = seg_mode; // 记忆当前模式
seg_mode = 3; // 切换到模式3
if(++cai_time >= 3000) {
flag = 0;
cai_time = 0;
seg_mode = seg_jiyi; // 恢复原模式
}
}
```
第1毫秒：`seg_jiyi = 0`（正确记忆），`seg_mode = 3`。第2毫秒：`seg_jiyi = seg_mode`，但此时 `seg_mode` 已经是 3，所以 `seg_jiyi` 被覆盖为 3。从第2毫秒起 `seg_jiyi` 就永远是 3，3 秒后 `seg_mode = seg_jiyi = 3`，模式无法恢复
- **解决方案：** 将记忆操作移到触发源（`ad_da` 函数）中，只在 `flag` 从 0 变为 1 的瞬间记忆一次：
```c
// ad_da() 中（触发源）
if((ad_cai1 < 250) && (ad_cai2 > 250)) {
if(flag == 0) { // 只在首次触发时记忆
seg_jiyi = seg_mode; // 记忆旧模式（此时还未被改）
seg_mode = 3; // 切换到模式3
flag = 1; // 开启3秒定时
}
}
// 定时器 ISR 中（只负责计时和恢复）
if(flag == 1) {
if(++cai_time >= 3000) {
flag = 0;
cai_time = 0;
seg_mode = seg_jiyi; // 恢复之前的模式
}
}
```

## 坑 54：函数声明与定义参数类型不匹配（bit vs unsigned char）

> 来源：[067_蓝桥杯单片机第二十三周周报（二）.md](./单片机答疑数据库/067_蓝桥杯单片机第二十三周周报（二）.md)
> 周期：蓝桥杯单片机第二十三周周报（二）

### 整理记录

- **问题描述：** 数码管所有位的小数点全亮，无法单独控制
- **问题背景：** 第11届省赛
- **问题分析：** `seg.h` 中函数声明使用 `bit` 类型参数，而 `seg.c` 中定义使用 `unsigned char`：
```c
// seg.h 声明
void Seg_Disp(unsigned char wela, unsigned char dula, bit point);
// seg.c 定义
void Seg_Disp(unsigned char wela, unsigned char dula, unsigned char point);
```
在 8051 架构中，`bit` 类型通过进位标志（CY）传递，`unsigned char` 通过寄存器（R5/R7等）传递。调用者（main.c）包含 `seg.h`，按 `bit` 方式将 0/1 设入CY；但 `seg.c` 编译时按 `unsigned char` 从寄存器读取，读到的是寄存器中的**残留值**（通常非零），导致 `if(point) P0 &= 0x7f;` 几乎总是成立。
**注意：** Keil C51 对此不会报编译错误或警告，因为 `.h` 和 `.c` 分别编译，链接器不检查参数类型一致性。
- **解决方案：** 统一参数类型，将 `seg.c` 中的 `unsigned char point` 改为 `bit point`
> **规律总结：** 8051 的 `bit` 类型有独特的传参机制（通过CY而非寄存器），这与标准C的 `char/int` 传参完全不同。`.h` 和 `.c` 之间的类型不匹配在普通平台上可能只是警告，但在8051上会导致完全错误的数据传递。**写完函数后务必逐参数核对声明和定义。**

## 坑 55：DS18B20 时序期间数码管闪烁

> 来源：[070_蓝桥杯单片机第二十五周周报.md](./单片机答疑数据库/070_蓝桥杯单片机第二十五周周报.md)
> 周期：蓝桥杯单片机第二十五周周报

### 整理记录

- **问题描述：** 数码管第 5、6 位会不时闪烁一下
- **问题背景：** 第十二届省赛题目，DS18B20 温度读取
- **问题分析：** `rd_temperature()` 函数执行 DS18B20 单总线通信（`init_ds18b20()`、`Write_DS18B20()`、`Read_DS18B20()`），这些函数内部包含精确时序延迟（微妙级），而主循环或定时器中开启了定时器中断。定时器中断用于数码管动态扫描，在 DS18B20 通信期间中断被打断，数码管扫描时序错乱，导致部分位闪烁
- **解决方案：** 在进入 DS18B20 时序前关闭中断（`EA = 0`），时序结束后恢复中断（`EA = 1`），确保单总线通信不被中断打断，同时保证数码管扫描时序完整
```c
EA = 0;
init_ds18b20();
Write_DS18B20(0xcc);
Write_DS18B20(0x44);
// ... 其余时序
EA = 1;
```
