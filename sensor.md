## 坑 8：蜂鸣器和继电器不能同时使能（底层状态未做累加）

> 来源：[014_蓝桥杯单片机第六周周报.md](./单片机答疑数据库/014_蓝桥杯单片机第六周周报.md)
> 周期：2025年10月19-2025年10月25

### 整理记录

* **问题描述**：蜂鸣器和继电器无法同时工作，只能单独控制其中一个。
* **问题背景**：无具体背景
* **问题分析**：

  + 两个外设控制函数各自使用独立的临时变量
  + 缺乏全局状态管理
  + 后执行的函数会覆盖前一个的设置
* **解决方案**：

  + 使用全局变量保存外设状态，实现状态累加控制：

```
unsigned char temp;
void Beep(unsigned char flag) // 蜂鸣器  
{  
    if(flag)  
        temp |= 0x40;  
    else  
        temp &= ~0x40;  
    if(temp != temp_old)  
    {  
        P0 = ~temp;  
        P2 = P2 & 0x1f | 0xa0;  
        P2 &= 0x1f;  
        temp_old = temp;          
    }  
}  

void Relay(unsigned char flag) // 继电器  
{  
    if(flag)  
        temp |= 0x10;  
    else  
        temp &= ~0x10;  
    if(temp != temp_old)  
    {  
        P0 = ~temp;  
        P2 = P2 & 0x1f | 0xa0;  
        P2 &= 0x1f;  
        temp_old = temp;          
    }  
}
```

## 坑 19：蜂鸣器和继电器异常启动（底层状态变量名写错）

> 来源：[054_蓝桥杯单片机第十八周周报（二）.md](./单片机答疑数据库/054_蓝桥杯单片机第十八周周报（二）.md)
> 周期：2026年2月3日 — 2026年2月8日

### 整理记录

* **问题描述**：代码烧录到板子后，蜂鸣器自动启动，继电器对应的LED灯也异常点亮。
* **问题背景**：蓝桥杯单片机竞赛STC15F2K60S2平台
* **问题分析**：
在底层led.c文件中，蜂鸣器（beep）和继电器（relay）函数的状态更新判断存在变量名笔误：
```c
// 错误代码
if (temp_2_old != temp_) // 变量名不完整
{
// 更新蜂鸣器/继电器状态
}
```
**问题根源**：
- 应该比较`temp_2_old`和`temp_2`是否相等
- 实际写成了`temp_2_old != temp_`（变量名不完整或未定义）
- 导致条件判断异常，状态更新逻辑失效
- 外设在初始化后立即被触发
* **解决方案**：
修正变量名拼写错误：
```c
// 正确代码
if (temp_2_old != temp_2)
{
temp_2_old = temp_2;
// 更新蜂鸣器状态
BEEP = temp_2;
}
if (temp_2_old != temp_2)
{
temp_2_old = temp_2;
// 更新继电器状态
RELAY = temp_2;
}
```

> 来源：[056_蓝桥杯单片机第十九周周报.md](./单片机答疑数据库/056_蓝桥杯单片机第十九周周报.md)
> 周期：2026年02月8日 — 2026年02月13日

### 整理记录

* **问题描述**：程序烧录到单片机后，蜂鸣器自动响起，继电器指示灯亮起，但按照逻辑不应该触发。
* **问题背景**：蓝桥杯单片机竞赛项目，使用STC15F2K60S2单片机。
* **问题分析**：
* 检查LED底层驱动函数
* 发现beep和relay的if判断条件错误
* 变量名使用错误：将`temp_2`误写成了`temp`
* **解决方案**：
* 修正LED底层函数中的变量名
* 将beep和relay的if判断条件中的`temp`改为`temp_2`
* 确保条件判断使用正确的变量
```c
// 错误代码示例
if(temp & 0x40) // ❌ 使用了错误的变量temp
beep_flag = 1;
// 正确代码
if(temp_2 & 0x40) // ✓ 使用正确的变量temp_2
beep_flag = 1;
```
* **经验总结**：
* 底层驱动函数的变量命名要清晰，避免混淆
* 类似功能的变量（如temp、temp_2）容易写错，需要仔细检查
* 烧录后出现异常行为，优先检查底层驱动和初始化代码

- **问题分析：** `Delay_OneWire()` 函数中关键的延时校准语句 `t *= 12;` 被注释掉了。DS18B20 采用单总线（OneWire）协议，对时序要求极其严格——复位脉冲、写时隙、读时隙都依赖精确的微秒级延时。`t *= 12` 是针对当前单片机主频的延时校准系数，注释掉后实际延时远小于协议要求，导致 DS18B20 初始化失败、数据读写全部异常，温度值始终为 0
```c
void Delay_OneWire(unsigned int t)
{
// t *= 12; // ← 被注释掉，延时不足，OneWire 时序全部失败
while(t--);
}
```
- **解决方案：** 取消 `t *= 12;` 的注释即可恢复正常温度读取
> **附注：** 该校准系数与单片机主频相关。蓝桥杯竞赛板使用 STC15F2K60S2（11.0592MHz），底层驱动代码中的延时系数需要匹配实际主频，不可随意注释或修改

## 坑 26：底层驱动被修改 + 主逻辑多处 Bug 导致大面积丢分（13处修复）

> 来源：[066_蓝桥杯单片机第二十三周周报.md](./单片机答疑数据库/066_蓝桥杯单片机第二十三周周报.md)
> 周期：蓝桥杯单片机第二十三周周报

### 整理记录

- **问题描述：** 4T 测评原始得分约 30/62，时钟显示完全错乱、温湿度数据异常、LED 状态残留，多个测试点未通过
- **问题背景：** 第十四届省赛（第一批）
- **问题分析：**
- **底层驱动问题（致命）：**
- [a] DS1302 缺少 BCD 转换：直接写十进制数，芯片要求 BCD 编码，导致时钟显示错乱（如应显示 `13-46-50` 却显示 `19-70-85`）
- [b] DS1302 缺少中断保护：SDA 接 P2^3，ISR 中 `Seg_Disp` 也操作 P2，冲突导致 RTC 通信失败
- [c] DS18B20 缺少中断保护和转换等待：ISR 打断单总线时序导致读取失败，发送 0x44 后未等待就读取得到脏数据
- [d] IIC 与 DS1302 的 `sbit SDA` 命名冲突：两个文件定义同名 `sbit` 指向不同引脚（P2^1 vs P2^3），链接器随机选择导致通信失效
- **主逻辑问题：**
- [e] `Humidity_Get` 中多余的 `rd_temperature()` 调用，两次读取间隔不足导致脏数据
- [f] `Touch_Count` 在两个函数中重复自增，每次触发计数加2
- [g] `Tempera_Sum` 在两个函数中重复累加，平均温度翻倍
- [h] 清除数据时遗漏 `L5_Flag`、`Humi_Dat_Flag`、`L4_Blink_Flag` 标志位重置，LED 残留点亮
- [i] 温湿度上升判断 `Touch_Count >= 1` 应为 `>= 2`，首次采集无历史数据可比较
- [j] 湿度频率有效范围用 `>` `<` 应为 `>=` `<=`，边界值被排除
- [k] 湿度平均值小数位多除一次 10
- [l] Timer0 误开中断 `ET0=1`，计数器模式不需要中断
- [m] DS18B20 上电默认 85°C 未做预热等待
- **解决方案：**
- [a] 写入时十进制→BCD：`bcd = (val/10<<4) | (val%10)`；读取时 BCD→十进制：`val = (bcd>>4)*10 + (bcd&0x0f)`
- [b] `Read_Rtc` / `Set_Rtc` 前后加 `EA=0` / `EA=1` 中断保护
- [c] `rd_temperature` 前后加中断保护，0x44 转换命令后加延时等待
- [d] `iic.c` 中 `SDA` 改为 `sda`，避免与 `ds1302.c` 冲突
- [e] 删除 `Humidity_Get` 中多余的 `rd_temperature()` 调用，统一使用 `Temperature()` 的缓存值
- [f][g] 删除重复的自增和累加，保证一个数据源只在一个地方采集
- [h] 清除数据时补全所有标志位重置
- [i] 判断条件改为 `Touch_Count >= 2`
- [j] 边界条件改为 `freq_local >= 200 && freq_local <= 2000`
- [k] 去掉多余的 `/10`
- [l] 删除 `ET0=1`
- [m] `main()` 中加 `while(rd_temperature()==85);` 预热等待

## 坑 27：整点继电器吸合 5 秒用事件触发式写法导致状态残留 + L1 无默认清零分支（2处修复）

> 来源：[066_蓝桥杯单片机第二十三周周报.md](./单片机答疑数据库/066_蓝桥杯单片机第二十三周周报.md)
> 周期：蓝桥杯单片机第二十三周周报

### 整理记录

- **问题描述：** 温度控制模式下存在不该亮的指示灯（L1 状态残留）；整点时继电器吸合后本应 5 秒断开，实际仍保持吸合
- **问题背景：** 第十三届省赛（第一批），涉及定时模式整点继电器吸合 5 秒、温度控制模式 LED 指示
- **问题分析：**
- [a] **整点继电器"持续 5 秒"用事件触发式写法（核心问题）：** 学员在 `Get_Temperature()` 中写成"分==0 且 秒==0 时吸合，分==0 且秒个位==5 时断开"。这是**两个时间点改状态**，不是**持续计时 5 秒**。问题在于：① 程序是否能精确在那个节拍完成状态切换依赖扫描时机，可能错过；② `ucRtc[2] % 10 == 5` 意味着 05、15、25、35、45、55 秒都会触发断开，逻辑不精确；③ 一旦错过断开时刻，继电器就一直保持吸合
- [b] **L1 状态灯无默认清零分支：** `Led_Light_Flag` 只在整点时刻置 1、在秒个位为 5 时置 0，除此之外不会被重新赋值。一旦被点亮后如果没有命中清零条件，L1 就一直保持亮着。切换到温控模式后，L1 应当熄灭，但由于没有默认清零逻辑，状态残留导致测评判错
- **解决方案：**
- [a] 整点吸合改为"标志位 + 定时计数器"方案：
```c
// 整点时只做一件事：置启动标志
if(ucRtc[1] == 0 && ucRtc[2] == 0)
Start_Flag = 1;
// 继电器由标志位控制
Relay(Start_Flag);
// 中断中累计 5000ms，到时自动清零
if(Start_Flag) {
if(++Time_5000ms >= 5000) {
Start_Flag = 0;
Time_5000ms = 0;
}
} else {
Time_5000ms = 0;
}
```
- [b] 所有状态灯必须有完整的"亮/灭/默认"逻辑，切换模式后应将不相关的状态标志位清零
> **附注：** 本例的根本原因是**不完整的事件触发式状态控制**。"持续 N 秒"类需求必须用计时思维实现（标志位 + 定时器累计），不能靠判断两个时间点来开关状态。状态机设计的三要素：**触发条件（何时开始）、保持条件（如何维持）、退出条件（何时结束）**，缺一不可。

## 坑 28：温度减速变量 unsigned char 装不下 300，条件永真导致温度永不采集

> 来源：[066_蓝桥杯单片机第二十三周周报.md](./单片机答疑数据库/066_蓝桥杯单片机第二十三周周报.md)
> 周期：蓝桥杯单片机第二十三周周报

### 整理记录

- **问题描述：** 学员照着课程讲解做完第十五届省赛（第二批），温度显示界面下无法显示环境温度，问了多个 AI 均未能解决
- **问题背景：** 第十五届省赛（第二批），DS18B20 温度采集减速控制
- **问题分析：** `Get_temperture()` 函数开头用 `if(temperture_slow_down < 300) return;` 做减速，每 300ms 才执行一次温度采集。但 `temperture_slow_down` 定义为 `unsigned char`（第 39 行），最大值只有 255。在定时器 ISR 中 `temperture_slow_down` 每 1ms 自增，增到 255 后溢出回到 0，**永远不可能达到 300**。条件 `< 300` 永远为真，函数每次都直接 `return`，温度采集永远不会执行，数码管上温度值始终为初始值 0 或不显示
- **解决方案：** 将 `temperture_slow_down` 的类型从 `unsigned char` 改为 `unsigned int`：
```c
// 修改前（最大255，装不下300）
idata unsigned char temperture_slow_down;
// 修改后（最大65535，可以装300）
idata unsigned int temperture_slow_down;
```
> **附注：** 这是本周第二次出现 `unsigned char` 装不下减速阈值的问题（另见第 17 题 [d]）。规律：凡是用于毫秒级计时的变量，阈值超过 255ms 时**必须**用 `unsigned int`。建议养成习惯——减速计时变量统一用 `unsigned int`，避免此类隐蔽溢出。

## 坑 29：Freq_Max 更新逻辑放在 Seg_Proc 中且记录时间重读 RTC，导致回显时间锁定在初始值（2处修复）

> 来源：[066_蓝桥杯单片机第二十三周周报.md](./单片机答疑数据库/066_蓝桥杯单片机第二十三周周报.md)
> 周期：蓝桥杯单片机第二十三周周报

### 整理记录

- **问题描述：** 4T 测评 #34 切换时间回显界面错误，期望显示 HA133445，实际显示 HA235950
- **问题背景：** 第十五届省赛（第一批），涉及频率最大值记录及对应时间回显
- **问题分析：**
- [a] **记录时间时重新调用 Read_Rtc() 读 DS1302（核心问题）：** 学员在检测到新最大值时调用 `Read_Rtc(ucRtc_Max_Para)` 重新读一次 DS1302，而不是直接拷贝已在调度器中定期读好的 `ucRtc` 数组。开机瞬间频率一来（任何非零值）就比 `Freq_Max`（初始值 0）大，立刻触发 `Read_Rtc`，此时 RTC 刚被设置为 23:59:50（初始化值），所以 `ucRtc_Max_Para` 被锁定为 23:59:50。后续频率没有再突破最大值，时间再也没有更新过，回显界面始终显示 HA235950
```c
// 学员写法（重新读 DS1302，读到的是初始化瞬间的值）
if(Freq > Freq_Max && Freq > 0)
{
Read_Rtc(ucRtc_Max_Para); // 重新读一次 DS1302
Freq_Max = Freq;
}
```
- [b] **Freq_Max 更新逻辑放在 Seg_Proc() 中：** 频率最大值判断和时间记录属于数据处理逻辑，不应混在显示函数 `Seg_Proc()` 中。频率计算和最大值判断应该和显示分离，否则数据处理的执行时机受显示刷新节奏影响，增加不可控因素
- **解决方案：**
- [a] 记录时间改为直接拷贝已缓存的 `ucRtc` 数组，不再重新读 DS1302：
```c
// 正确写法（直接拷贝已经读好的 ucRtc 数组）
if(Freq > Freq_Max && Freq > 0)
{
ucRtc_Max_Para[0] = ucRtc[0]; // 直接拷贝已读好的时间
ucRtc_Max_Para[1] = ucRtc[1];
ucRtc_Max_Para[2] = ucRtc[2];
Freq_Max = Freq;
}
```
- [b] 将这段逻辑从 `Seg_Proc()` 中分离出来，单独做一个 `Freq_Proc()` 放进调度器，职责分离——显示函数只负责把现成数据搬到数码管上
> **附注：** 本例暴露了两个设计原则问题：① **职责分离**——数据处理（频率最大值追踪）不应混在显示函数中，应独立为 `Freq_Proc()` 并纳入调度器；② **数据源一致性**——需要时间快照时，应拷贝调度器中已经定期刷新的 `ucRtc` 缓存数组，而非重新调用 `Read_Rtc()` 读硬件。重新读硬件的问题在于读取时刻不可控，尤其在初始化阶段 RTC 刚被写入初始值时，读到的是初始值而非真实运行时间。

## 坑 30：DS18B20 通信失败的复合原因

> 来源：[067_蓝桥杯单片机第二十三周周报（二）.md](./单片机答疑数据库/067_蓝桥杯单片机第二十三周周报（二）.md)
> 周期：蓝桥杯单片机第二十三周周报（二）

### 整理记录

- **问题描述：** 上电后温度始终显示 55.9 或 59.9，与实际温度无关
- **问题背景：** 第11届省赛（多名学生）
- **问题分析：** 这个问题需要**同时修复两个bug**，只改一个仍然不行：
**Bug A — 1T单片机延时未补偿：**
```c
void Delay_OneWire(unsigned int t)
{
//t *= 12; // ← 被注释掉了！
while(t--);
}
```
STC15F2K60S2 是 1T 单片机，每时钟周期执行一条指令，比传统 12T 的 8051 快 12 倍。所有 OneWire 时序（复位脉冲、读写位）都快了约12倍，DS18B20 根本无法正确响应。
**Bug B — 发完转换命令后立即复位读取：**
```c
float rd_temperature()
{
init_ds18b20();
Write_DS18B20(0xcc);
Write_DS18B20(0x44); // 发起温度转换（需要750ms）
init_ds18b20(); // ← 立即复位！中断了转换！
Write_DS18B20(0xcc);
Write_DS18B20(0xbe); // 读取的是转换被中断后的垃圾值
// ...
}
```
**为什么显示55.9/59.9？** 通信完全失败时，总线保持高电平，读回全1（0xFFFF），0xFFFF / 16.0 ≈ 4095.9，对于 DS18B20 的 12 位有效数据，实际截取低12位后为 0x0FFF = 4095，但受符号位影响，0x07FF / 16 = 127.9 或特定截取方式得到 55.9/59.9。
- **解决方案：** 两个bug必须同时修复：
```c
// 1. 取消注释 t *= 12
void Delay_OneWire(unsigned int t) { t *= 12; while(t--); }
// 2. 改为"先读上次结果，再发起下次转换"
float rd_temperature()
{
unsigned char low, high;
init_ds18b20();
Write_DS18B20(0xcc);
Write_DS18B20(0xbe); // 先读上次结果
low = Read_DS18B20();
high = Read_DS18B20();
init_ds18b20();
Write_DS18B20(0xcc);
Write_DS18B20(0x44); // 再发起下次转换
return ((high << 8) | low) / 16.0;
}
```
同时确保调用间隔 ≥ 750ms（调度器周期设为1000ms）。
> **规律总结：** DS18B20 问题几乎每周都有，但本周的特点是**两个bug叠加**，修一个不够。赛前模板中应预置正确的"先读后转"模式和 `t *= 12` 补偿。

- [b] 第一次采样只记录温度不比较，从第二次开始做升降判断（`temperature_sample_count` 保护）
- [c] L2/L3 改为"事件保持灯"模式：检测到升/降温时启动 2s 计时器，显示时根据计时值决定亮灭
- [d] L4 独立做成"闪烁窗口"：仅由"本次温差绝对值 > 1℃"直接触发，0.2s 翻转，3s 后强制熄灭

- [e] 删除时间界面中的参数修改代码，S17 修改参数仅在参数界面有效

- [g] RTC 初始时间改为非整点值如 `{23,59,55}`

## 坑 39：第十二届省赛 温度始终显示 00.30——OneWire 底层时序被破坏

> 来源：[069_蓝桥杯单片机第二十四周周报（二）.md](./单片机答疑数据库/069_蓝桥杯单片机第二十四周周报（二）.md)
> 周期：蓝桥杯单片机第二十四周周报（二）

### 整理记录

- **问题描述：** 学员按照大模板写的温度显示，但数码管始终显示 `00.30`，无法读取实际温度
- **问题背景：** 第十二届省赛题目
- **问题分析：** 存在两个问题：
- [a] **底层驱动 `Delay_OneWire` 时序被破坏（主要原因）：** 学员可能为适配 12MHz 主频，在延时函数中加了 `t *= 12`，导致 OneWire 协议中所有时序（复位脉冲、读写时隙）放大为原来的 12 倍。DS18B20 对时序要求非常严格，时序一乱通信就失败，读出的数据自然是错的：
```c
void Delay_OneWire(unsigned int t) {
unsigned char i;
t *= 12; // ← 导致所有延时放大 12 倍
while (t--) {
for (i = 0; i < 12; i++);
}
}
```
- [b] **温度采集放在了 `Seg_Proc()` 显示函数中（结构问题）：** `Seg_Proc()` 执行周期为 500ms，而 DS18B20 在 12 位精度下一次转换需要 750ms，采集间隔不满足转换时间要求，且采集与显示逻辑耦合
- **解决方案：**
- [a] 删除 `Delay_OneWire` 中的 `t *= 12`，恢复为官方原版底层驱动。底层驱动不要随意修改，如需适配不同主频应调整循环内部迭代次数，而非对参数做倍乘
- [b] 将温度采集抽离为独立函数 `Temp_Proc()`，使用独立减速计数器，间隔设为 800ms，`Seg_Proc()` 只负责显示

## 坑 40：第十四届省赛 温度界面值跳变导致最大值记录错误——中断打断单总线时序

> 来源：[069_蓝桥杯单片机第二十四周周报（二）.md](./单片机答疑数据库/069_蓝桥杯单片机第二十四周周报（二）.md)
> 周期：蓝桥杯单片机第二十四周周报（二）

### 整理记录

- **问题描述：** 温度界面的值会偶发跳变，导致最大值记录被异常值污染
- **问题背景：** 第十四届省赛题目
- **问题分析：** DS18B20 单总线通信对时序要求非常严格，读取过程中如果被定时器中断打断，时序被破坏会导致偶发读到异常值，表现为温度显示跳变，异常高值被记录为最大值
- **解决方案：** 在 `rd_temperature()` 函数中，单总线通信前后加入开关总中断保护（`EA = 0` / `EA = 1`），确保读取过程不被中断打断：
```c
float rd_temperature() {
unsigned char low, high;
EA = 0;
init_ds18b20();
Write_DS18B20(0xcc);
Write_DS18B20(0x44);
Delay_OneWire(200);
init_ds18b20();
Write_DS18B20(0xcc);
Write_DS18B20(0xbe);
low = Read_DS18B20();
high = Read_DS18B20();
EA = 1;
return (float)(high << 8 | low) * 0.0625;
}
```

- [d] 将数码管刷新周期调快至 100ms，将温度采样从显示函数中分离（新增 `Temp_Proc()` 独立采样），确保界面切换时显示能即时刷新

## 坑 42：第十四届 4T 测评采集状态流未走顺

> 来源：[070_蓝桥杯单片机第二十五周周报.md](./单片机答疑数据库/070_蓝桥杯单片机第二十五周周报.md)
> 周期：蓝桥杯单片机第二十五周周报

### 整理记录

- **问题描述：** 第十四届省赛 4T 测评大量失分：历史界面尚未采集就出现数据、触发采集后未进入 `E xx-yy` 页面、温度/湿度最大值一直不更新、湿度非法时 `AA` 无法显示、时间回显界面显示当前时钟而非最后一次采集时间
- **问题背景：** 第十四届省赛题目，暗光触发采集 → 温湿度采样 → 历史记录入库，涉及多步状态流转
- **问题分析：** 学员代码的采集数据链存在 7 处缺陷：
- [a] **暗光触发旧值未保存：** 触发暗光采集时保存了上一次光照值，但每次采样后没有更新旧值，导致触发条件实际很难按预期成立
- [b] **未采集就提前写入历史数据：** 历史记录更新条件太宽（仅 `error_flag==0 && entry_data==0`），在温湿度实际完成采集前就已写入假数据
- [c] **湿度采样标志写反：** 判断条件写成了"已经采完了才去采"，导致湿度采样根本不会执行，湿度始终为 0
- [d] **非法频率判断逻辑错误：** `freq<200 && freq>2000` 使用了 `&&` 而非 `||`，条件永远不成立，`AA` 无法显示
- [e] **最大值更新写反：** 把"当前值赋给当前值"而非"当前值赋给最大值变量"，导致最大值长期保持 0
- [f] **L4 判断使用旧温度值：** 比较温度参数时使用的是上一轮的旧值，导致 L4 闪烁状态慢一拍
- [g] **LED 状态未完整刷新：** 部分 LED 只在满足条件时赋值，不满足时不主动清零，遗留上一轮状态
- **解决方案：**
- [a] 每次光照采样后更新旧值：`voltage_value = temp;`，触发条件改为 `temp<80 && voltage_value>80`（由亮变暗才触发）
- [b] 修改入库条件为"温度和湿度都完成触发后才允许写入一次"：`if(temper_trigger_over==1 && hum_trigger_over==1 && entry_data==0)`
- [c] 湿度采样标志判断纠正为"未完成时去采"：`if(hum_trigger_over==0)` 进入采集
- [d] 非法频率判断改为 `||`：`if(freq<200 || freq>2000)` 判非法，显示 `AA`
- [e] 最大值更新修正为 `temper_max = temper_val;` / `hum_max = hum_val;`
- [f] L4 比较使用最新采样值：`l4_flag = (temper_val > temper_para);`
- [g] LED 处理函数中每次完整赋值所有 LED 状态，不满足条件时显式清零

## 坑 43：第十七届模拟一记录时间错位与蜂鸣器分支错误

> 来源：[070_蓝桥杯单片机第二十五周周报.md](./单片机答疑数据库/070_蓝桥杯单片机第二十五周周报.md)
> 周期：蓝桥杯单片机第二十五周周报

### 整理记录

- **问题描述：** 处理数据显示异常（`45cm` 时显示成 `L000`）、记录界面时间整体错位（应显示 `A1091803` 却显示 `A1180300`）、蜂鸣器状态判断不稳定
- **问题背景：** 第十七届模拟题一，涉及超声波测距、DS1302 时钟记录、多界面切换
- **问题分析：** 学员代码存在 4 个核心缺陷：
- [a] **记录时间数组下标用错：** `ucRtc[0]` 为时、`ucRtc[1]` 为分、`ucRtc[2]` 为秒，但学员在记录时循环从 `i=1` 开始并写了 `take[0][i]=ucRtc[i]`，导致小时没写进去、分钟被写到小时位置、秒被写到分钟位置，还出现数组越界访问。以 `09:18:03` 为例，最终写入变成 `18 03 00`，显示为 `A1180300`
- [b] **清空记录时数组越界：** 记录数组只有 3 行，但清空时循环写到了第 4 行，破坏记录数据的同时可能误改相邻变量（如超声波采样值 `Wave`、蜂鸣器标志等），引发连锁显示异常
- [c] **蜂鸣器 `else if` 缺少花括号：** `wave_flag=0;` 受 `else if` 约束，但 `Beep_flag=0;` 变成了每次都会执行，导致蜂鸣器状态"刚置位就被清零"，状态机不完整
- [d] **距离值重复读取：** 在不同处理函数中多次调用超声波读取函数，显示用一次采样，记录和报警另一次。若其中一次超时返回 `0`，就会导致显示 `L000`、误判距离、三者互不同步
- **解决方案：**
- [a] 时间记录改为分别赋值：`take[0][1]=ucRtc[0]; take[0][2]=ucRtc[1]; take[0][3]=ucRtc[2];`，不再依赖循环错位
- [b] 清空记录时限制循环到 3 行内（`i<3, j<4`），避免越界误改
- [c] 蜂鸣器控制的 `else if` 分支补全花括号，确保 `wave_flag` 和 `Beep_flag` 在同一条件下同步清零
- [d] 距离值统一在 `Data_Proc()` 中只读取一次到 `Wave` 变量，显示、记录、蜂鸣器共用同一份数据；当采样返回 `0` 时不覆盖上一次有效距离，并增加 `Data_Ready` 标志确保拿到第一份有效数据前不触发记录逻辑

## 坑 44：第十四届省赛温湿度旧值保存时机与比较位置错位

> 来源：[070_蓝桥杯单片机第二十五周周报.md](./单片机答疑数据库/070_蓝桥杯单片机第二十五周周报.md)
> 周期：蓝桥杯单片机第二十五周周报

### 整理记录

- **问题描述：** 光照由亮变暗触发后，温湿度"是否都比上一次触发时更高"的判断结果不稳定，甚至触发后比较失真
- **问题背景：** 第十四届省赛题目，光敏下降沿触发采集 → 温湿度锁存 → 与上一次触发值比较判断上升标志位，涉及三种不同类型旧值变量的区分
- **问题分析：**
- [a] **温湿度 old 值保存时机错误：** 学员在本次触发采集完成后，立刻执行了 `Temperature_Value_old = Temperature_Value; Humidity_Value_old = Humidity_Value;`，导致本次触发值刚算出来，`old` 就被覆盖成这一次的值。后续比较时 `current` 和 `old` 已经一样，"当前值大于上一次值"这个条件失真
- [b] **比较位置放错：** 温湿度比较逻辑写在了 `Get_Temperature()` 温度采集函数里，但采集函数只是周期读取当前温度，不是"触发事件真正成立的时刻"。比较应该在"亮到暗触发成立且本次温湿度已锁存完成"后进行
- [c] **三种旧值变量类型混淆：** `AD_old` 表示"上一轮轮询光照值"（每轮更新是对的），但 `Temperature_Value_old` 和 `Humidity_Value_old` 表示"上一次有效触发时锁存的值"，不能像 `AD_old` 一样每轮直接覆盖
- **解决方案：**
- [a] 温湿度 old 值更新放到触发处理函数（`AD_DA()`）末尾，顺序必须是：**先比较当前值与 old 值 → 更新上升标志位 → 最后再更新 old 值**，即 "先比较，再更新 old"
- [b] 将温湿度比较从 `Get_Temperature()` 移出，收回到触发处理函数中的正确位置：亮到暗触发成立 → 锁存温湿度 → 与上一次触发值比较 → 得出 `Temperature_Humidity_Rise_Flag` → 最后更新 old
- [c] `Get_Temperature()` 只负责更新当前温度值：`Temperature_now = rd_temperature();`，不再做旧值比较
- 修改原则：`old` 是"上一轮扫描值"则每轮循环后更新；`old` 是"上一次有效事件值"则在"事件处理结束后"再更新

## 坑 47：温度突变检测错位

> 来源：[071_蓝桥杯单片机第二十五周周报（二）.md](./单片机答疑数据库/071_蓝桥杯单片机第二十五周周报（二）.md)
> 周期：蓝桥杯单片机第二十五周周报（二）

### 整理记录

**问题描述：** 在进行温度报警与指示灯控制逻辑测评时，系统出现多处报错，具体表现为：温度突变（如 23°C 瞬间跳变至 35°C）时未能触发报警、以及温度下降后相关指示灯未能按时（2s 后）自动熄灭。
**问题分析：** 此次测评大规模报错的核心在于“时序耦合”与“状态机冗余”，具体原因如下：
- [a] **冗余的“两拍”状态机导致跳变漏检：** 温度突变检测逻辑错误地引入了前置状态变量（`SOS_Long` 等）。温度跳变的第一拍仅记录了状态而未报警，导致第二拍采集时因温度已稳定（`T_new - T_Old == 0`），完美错过了 `>1` 的报警触发条件。
- [b] **变量越界引用：** 在处理温度降低（`Low_T_Flag`）逻辑时，代码错误地引用了用于判断温度升高的计时变量 `f2000ms_L`。由于降温时该变量恒为 0，导致关闭指示灯的阈值条件永远成立，对应指示灯（L3）长亮不灭。
**解决方案：**
- [a] **简化状态，实施直接“单拍”比对：** 彻底删除 `SOS_Long` 和 `SOS_Low` 等无用中间态。在每次读取温度后，直接将当前值 `T_new` 与上一次值 `T_Old` 进行差值比对，一旦温差大于 1，立即触发报警标志（`SOS_Flag = 1`），并在函数最末尾执行 `T_Old = T_new` 更新历史值。
- [b] **修正逻辑变量映射：** 排查各个状态标志位的从属关系，将降温逻辑分支中的倒计时判断变量由 `f2000ms_L` 修正为专属此分支的 `f2000ms_R`，恢复独立计时。
