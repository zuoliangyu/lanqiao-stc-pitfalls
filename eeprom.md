## 坑 1：AT24C02 写入后等待延时不足导致数据丢失

> 第十三届国赛 · 发现者：左岚

### 现象

参数掉电后丢失，或连续写入时第二次写操作被静默忽略，但 I2C 时序和代码逻辑看起来正常。

### 原因

AT24C02 页写时序要求 Stop 信号后至少等待 **5ms** 才能发起下一次 I2C 操作。等待延时被削减后，芯片仍处于内部擦写周期中，后续操作会被忽略：

```c
// 错误写法：只有 4 次 Delay，总等待约 1~2ms，不足 5ms
I2CStop();
I2C_Delay(255);
I2C_Delay(255);
I2C_Delay(255);
I2C_Delay(255);
// 后续 6 次被注释掉了
```

### 解决方法

恢复足够的等待次数，确保 Stop 后总延时 ≥ 5ms：

```c
// 正确写法：共 10 次，保证写周期等待充足
I2CStop();
I2C_Delay(255); I2C_Delay(255); I2C_Delay(255); I2C_Delay(255); I2C_Delay(255);
I2C_Delay(255); I2C_Delay(255); I2C_Delay(255); I2C_Delay(255); I2C_Delay(255);
```

### 举一反三

- AT24C02 写周期（tWR）典型值 5ms，最长 10ms，必须严格保证
- 调试时不要随意注释"看起来多余"的延时，写周期等待是协议必须项
- 进阶做法：用 ACK 轮询代替固定延时（收到 ACK 即说明写周期完成，更可靠）

## 坑 2：I2C 接收函数局部变量未初始化导致数据最高位污染

> 第十三届国赛 · 发现者：左岚

### 现象

I2C 数据接收偶发错误，接收到的字节最高位可能被随机污染，复位后偶尔恢复正常。

### 原因

`I2CReceiveByte` 中局部变量 `da` 未初始化，第一轮循环执行 `da <<= 1` 前 `da` 含栈上垃圾值，最终结果的高位可能错误：

```c
// 错误写法：da 未初始化，高位可能含垃圾值
unsigned char da;
```

### 解决方法

声明时显式初始化为 0：

```c
// 正确写法
unsigned char da = 0;
```

### 举一反三

- C51 局部变量存放于寄存器或栈，不自动清零，必须显式初始化
- 所有参与移位、位或运算的变量，初值必须确定，否则结果不可预测
- 养成声明时同步初始化的习惯，可彻底避免此类隐患

## 坑 5：EEPROM读写16位数据

> 来源：[054_蓝桥杯单片机第十八周周报（二）.md](./单片机答疑数据库/054_蓝桥杯单片机第十八周周报（二）.md)
> 周期：2026年2月3日 — 2026年2月8日

### 整理记录

* **问题描述**：不知道如何在EEPROM中读写16位（2字节）的数据，只掌握了8位数据的读写方法。
* **问题背景**：蓝桥杯单片机竞赛STC15F2K60S2平台
* **问题分析**：
STC15F2K60S2的EEPROM读写函数通常只提供字节级别的操作接口（unsigned char类型）。要存储16位数据（如int、unsigned int），需要将其拆分为高字节和低字节分别存储。
**关键概念**：
- 16位数据 = 高8位 + 低8位
- 需要两个连续的EEPROM地址存储
- 读写时注意字节序（高低位顺序）
* **解决方案**：
**写入16位数据**：
```c
// 将16位数据拆分成两个8位数据写入EEPROM
void EEPROM_Write16(unsigned int addr, unsigned int data16)
{
unsigned char data_array[2];
// 拆分16位数据
data_array[0] = (unsigned char)(data16 >> 8); // 高字节
data_array[1] = (unsigned char)(data16 & 0xFF); // 低字节
// 分别写入两个地址
EEPROM_WriteByte(addr, data_array[0]);
EEPROM_WriteByte(addr + 1, data_array[1]);
}
```
**读取16位数据**：
```c
// 从EEPROM读取两个8位数据并拼接成16位
unsigned int EEPROM_Read16(unsigned int addr)
{
unsigned char data_array[2];
unsigned int result;
// 分别读取两个字节
data_array[0] = EEPROM_ReadByte(addr); // 高字节
data_array[1] = EEPROM_ReadByte(addr + 1); // 低字节
// 拼接成16位数据
result = ((unsigned int)data_array[0] << 8) | data_array[1];
return result;
}
```
**使用示例**：
```c
unsigned int temp_value = 1234;
// 写入16位数据到地址0x0000
EEPROM_Write16(0x0000, temp_value);
// 从地址0x0000读取16位数据
unsigned int read_value = EEPROM_Read16(0x0000);
```

## 坑 6：EEPROM多字节连续写入丢失数据

> 来源：[059_蓝桥杯单片机第二十一周周报.md](./单片机答疑数据库/059_蓝桥杯单片机第二十一周周报.md)
> 周期：蓝桥杯单片机第二十一周周报

### 整理记录

-
- **问题描述**：使用数组连续写入10个字节到EEPROM后，导致数据错乱，且紧接着写入的“密码锁”标志位无法成功保存。
- **问题背景**：第八届国赛及需要掉电保存大容量数据的赛题。
- **问题分析**：AT24C02每次收到写入指令后需要约5ms的内部刻录时间，期间拒收新指令（导致标志位写入失败）。且内部按“页(8字节)”管理，连续块写超过8字节边界会折返覆盖当前页的起始数据。
- **解决方案**：弃用多字节连续写入API，改为单字节循环写入（for循环），并且在**每一次**EEPROM_Write操作之后，强行加入 Delay5ms() 等待芯片内部烧录完成。

## 坑 8：EEPROM 校验机制导致自动评测参数读取失败

> 来源：[061_蓝桥杯单片机第二十一周周报（三）.md](./单片机答疑数据库/061_蓝桥杯单片机第二十一周周报（三）.md)
> 周期：蓝桥杯单片机第二十一周周报（三）

### 整理记录

- **问题描述：** 自动评测中 EEPROM 参数未被正确读取，但手动测试时恰好通过
- **问题背景：** 自动评测系统测试
- **问题分析：** 代码中添加了自定义校验机制：
```c
EEPROM_Read(&EEPROM_Temp, 8, 1); // 读地址8的校验值
if(EEPROM_Temp == EEPROM_D) // EEPROM_D = 5，只有等于5才读参数
EEPROM_Read(&Ad_10x_para, 0, 1);
```
但评测系统只在地址 0 写入了值 10，并未在地址 8 写入校验字节 5，导致 `EEPROM_Temp != 5`，校验失败，`Ad_10x_para` 未从 EEPROM 读取。手动测试能通过是因为代码中初始值恰好也是 10：
```c
unsigned char Ad_10x_para = 10; // 初始值碰巧与EEPROM中一致
// VP = 10 × 0.1 = 1.00V，结果恰好正确
```
- **解决方案：** 去掉自定义校验机制，直接按评测系统约定的地址读取 EEPROM 参数，不要添加评测系统未预期的额外校验逻辑

## 坑 9：EEPROM 加锁机制导致 4T 测评不通过

> 来源：[062_蓝桥杯单片机第二十二周周报.md](./单片机答疑数据库/062_蓝桥杯单片机第二十二周周报.md)
> 周期：蓝桥杯单片机第二十二周周报

### 整理记录

- **问题描述：** 4T 测评中其他功能全部正确，唯独 EEPROM 存储项全部报错——期望读出 10，实际读出 20/15 等错误值。加锁后得了 20 多分，去掉锁后只剩 EEPROM 不通过
- **问题背景：** 第十一届省赛题目
- **问题分析：** 学员在 EEPROM 读写中添加了自定义的校验/加锁机制。4T 自动测评系统按照题目约定的地址直接读写 EEPROM，不会配合用户自定义的锁逻辑，导致测评系统写入的数据被锁机制干扰，读出值与预期不符。这是 4T 测评的已知兼容性问题，米醋论坛上已有相关反馈
- **解决方案：** 4T 测评时不要在 EEPROM 中加锁/校验机制，直接按题目约定的地址进行读写即可。自定义校验逻辑仅适用于实际产品场景，不适用于自动评测环境

## 坑 10：EEPROM 掉电数据丢失（unsigned int 只写了 1 字节 + 局部变量未初始化）

> 来源：[062_蓝桥杯单片机第二十二周周报.md](./单片机答疑数据库/062_蓝桥杯单片机第二十二周周报.md)
> 周期：蓝桥杯单片机第二十二周周报

### 整理记录

- **问题描述：** EEPROM 存储参数后掉电重启，数据未正确恢复
- **问题背景：** 日常课程练习
- **问题分析：** 存在两处错误：
- [a] **写入字节数不足：** `Voltage_Compare` 声明为 `idata unsigned int`（2 字节），但 `EEPROM_Write(&Voltage_Compare, 0, 1)` 第三个参数写了 1，只写入了低字节（1 字节），高字节丢失。读取时 `EEPROM_Read(&Voltage_Compare, 0, 1)` 同样只读了 1 字节，高字节为随机值，导致恢复的数据与存入时不一致
- [b] **局部变量未初始化：** `EEPROM_Temp` 未初始化，校验判断时使用随机值，可能导致校验意外通过或失败
- **解决方案：**
- [a] `unsigned int` 占 2 字节，读写长度参数应改为 2：`EEPROM_Write(&Voltage_Compare, 0, 2)` 和 `EEPROM_Read(&Voltage_Compare, 0, 2)`
- [b] 局部变量声明时务必初始化：`unsigned char EEPROM_Temp = 0;`

## 坑 11：BCD码与十进制混淆

> 来源：[063_蓝桥杯单片机第二十二周周报（二）.md](./单片机答疑数据库/063_蓝桥杯单片机第二十二周周报（二）.md)
> 周期：蓝桥杯单片机第二十二周周报（二）
> 关联模块：定时 / 计数

### 整理记录

- **问题描述：** EEPROM存入的时钟时和分数据错误（16时存成22，47分存成71）
- **问题背景：** 第15届省赛
- **问题分析：** DS1302寄存器返回BCD格式数据（0x16表示16时，字节值=22），代码未做转换直接存入EEPROM，评测器按十进制解释读到22和71
```
DS1302寄存器(BCD) → ucRtc[](BCD) → Timer_Record[](BCD) → EEPROM(BCD)
评测器期望：16、47（十进制）
实际存入：22、71（BCD原始字节值）
```
- **解决方案：** 写入EEPROM时做BCD→十进制转换：
```c
EEPROM_Write_Dat[0] = Timer_Record[0] / 16 * 10 + Timer_Record[0] % 16;
EEPROM_Write_Dat[1] = Timer_Record[1] / 16 * 10 + Timer_Record[1] % 16;
```
> **附注：** 题目中"HEX编码写入"仅针对输入数据（地址2和3），时间数据（地址0和1）存实际十进制数值即可。

## 坑 13：EEPROM 写入时机不符题意

> 来源：[067_蓝桥杯单片机第二十三周周报（二）.md](./单片机答疑数据库/067_蓝桥杯单片机第二十三周周报（二）.md)
> 周期：蓝桥杯单片机第二十三周周报（二）

### 整理记录

- **问题描述：** 评测系统检查EEPROM地址0的值与预期不符
- **问题背景：** 第11届省赛（本次对话）、第13届国赛
- **问题分析：** 题目明确要求"**从参数界面退出时**，将电压参数VP×10保存到EEPROM"，但代码在每次按S16/S17调整参数时都立即写入：
```c
// 错误：每次按键都写EEPROM
case 16:
Voltage_Standard += 5;
EEPROM_Write(&Voltage_Standard, 0, 1); // ← 不应在此处！
break;
```
另一个变体更严重——在主循环中无条件写入：
```c
// 第13届国赛：led_proc() 中每次调用都写
void led_proc()
{
// ...
EEPROM_Write(&delay_j, 0, 1); // 每次主循环迭代都写！
}
// 后果：EEPROM寿命被快速消耗 + 主循环被阻塞~50ms/次
```
- **解决方案：** 仅在按S12退出参数界面时执行一次写入：
```c
case 1: // 参数界面
if(Key_Down == 12)
{
EEPROM_Write(&Voltage_Standard, 0, 1); // 退出时才写
Seg_Disp_Mode = 2; // 切换到计数界面
}
// S16/S17只修改变量，不写EEPROM
```
> **规律总结：** EEPROM写入时机必须严格按题意执行。常见错误有两种：（1）每次修改参数就写，（2）在主循环中无条件写。正确做法是"修改时只改RAM变量，退出/切换时才写EEPROM"。

## 坑 14：第十三届国赛 EEPROM 上电能读但写不成功——保存策略设计不合理

> 来源：[069_蓝桥杯单片机第二十四周周报（二）.md](./单片机答疑数据库/069_蓝桥杯单片机第二十四周周报（二）.md)
> 周期：蓝桥杯单片机第二十四周周报（二）

### 整理记录

- **问题描述：** 上电后能从 EEPROM 读出继电器动作次数，但运行中次数变化后掉电再上电仍为旧值，写入不成功
- **问题背景：** 第十三届国赛题目
- **问题分析：** 问题不在 EEPROM 底层驱动，而在上层保存逻辑，三个原因叠加：
- [a] **旧值更新过早：** 程序在真正写 EEPROM 之前就把 `Relay_Num_Old = Relay_Num` 执行了，如果本次写入因锁标志校验未通过而没有执行，系统也不会再认为"数据发生了变化"，失去重试机会
- [b] **锁标志被当成运行时写入门禁：** 每次保存前都要先读取并校验锁标志，一旦锁标志异常，后续写入就会长期被阻断。锁标志更适合用于"上电判断数据是否有效"，不适合在运行时拦截每次写操作
- [c] **保存周期偏慢：** 保存任务是周期性执行（500ms），不是继电器变化后立刻保存，如果数据刚变化还没轮到保存任务就掉电，也会丢失
- **解决方案：**
- [a] 只在 EEPROM 写入成功后才同步旧值：先写数据，再写锁标志，最后才 `Relay_Num_Old = Relay_Num`
- [b] 运行时不再用锁标志拦截写入，只要数据变化就直接写，写完后顺带维护锁标志；锁标志仅在上电初始化时用于判断历史数据是否有效
- [c] 改进后的保存函数：
```c
void EEPROM_Save(void)
{
if(Relay_Num_Old != Relay_Num)
{
EEPROM_Write(&Relay_Num, 0, 1);
EEPROM_Write(&EEPROM_Lock, 22, 1);
Relay_Num_Old = Relay_Num;
}
}
```

## 坑 15：第十一届 EEPROM 变量类型与保存时机错误引发连锁失败

> 来源：[070_蓝桥杯单片机第二十五周周报.md](./单片机答疑数据库/070_蓝桥杯单片机第二十五周周报.md)
> 周期：蓝桥杯单片机第二十五周周报

### 整理记录

- **问题描述：** 第十一届省赛 4T 测评中，参数界面的加减显示和数值比较全部异常：进入参数界面后显示 `P 6.00` 而非 `P 1.00`，越限判断、越线计数、状态指示灯全部与预期不符
- **问题背景：** 第十一届省赛题目，EEPROM 存储电压阈值参数用于越线检测与越限判断
- **问题分析：** 第一份代码存在两个根本错误：
- [a] **EEPROM 存储变量与运行阈值混用：** `voltage_disp_10x` 定义为 `unsigned int`（2 字节），但 EEPROM 仅读写 1 字节内容，在 C51 环境下导致初值读歪，阈值从正常的 `10`（即 1.0V）变成了异常大值（约 6V），后续所有比较和越线计数均基于错误阈值运行
- [b] **EEPROM 保存时机错误：** 题目要求退出参数界面时保存（P → N），但代码写在了切回数据显示界面时保存（N → U），导致 4T 检测 EEPROM 地址 0 时读到的仍为旧值
- [c] **连锁反应：** 参数初值错误直接导致越限判断（L1 状态）、越线计数（Num）、清零测试等后续测评项全部失败，并非独立新 bug
- **解决方案：**
- [a] 将 EEPROM 读写变量改为 `unsigned char`（1 字节），同时新增独立的运行阈值变量（如 `Voltage_Control_100x`），实现"存储值、显示值、运行值"三者分离
- [b] 将 EEPROM 写回时机改为退出参数界面时（`Seg_Disp_Mode == 2` 时），而非切回数据显示界面时
- [c] 上电时先读 EEPROM 到显示变量，再拷贝到运行阈值变量：`Voltage_Control_100x = Voltage_Disp_10x * 10;`，后续采样比较统一使用运行阈值
