## 坑 4：串口指令不熟

> 来源：[026_蓝桥杯单片机第八周周报（五）.md](./单片机答疑数据库/026_蓝桥杯单片机第八周周报（五）.md)
> 周期：2025年09月02日 — 2025年09月28日

### 整理记录

* **问题描述**：学员提问：“如果输入一个10进制数字，然后以`%X`的形式打印，它就会自动转换成16进制输出吗？”
* **问题背景**：涉及串口通信的题目。
* **问题分析**：这是对C语言`printf`函数格式化输出字符的基本用法不熟悉。`%x`和`%X`就是用于将整数以十六进制形式输出的占位符。
* **解决方案**：明确告知学员，`printf`中使用`%x`会将整数以**小写**十六进制形式输出（如 `ff`），而使用`%X`会以**大写**十六进制形式输出（如 `FF`）。

## 坑 6：串口定时器与调度器选择冲突

> 来源：[035_蓝桥杯单片机第十周周报（四）.md](./单片机答疑数据库/035_蓝桥杯单片机第十周周报（四）.md)
> 周期：2025年11月17日 — 2025年11月23日

### 整理记录

* **问题描述**：程序中同时配置了串口通信和任务调度器，编译无报错，但串口始终无法接收到上位机发送的数据，排查后发现定时器中断未正常触发，串口接收中断也无响应。
* **问题背景**：串口采用定时器1作为波特率发生器，调度器基于定时器0实现定时任务。
* **问题分析**：

  + 定时器1（串口波特率发生器）与定时器0（调度器）的中断优先级未明确配置，默认优先级相同导致中断抢占冲突，调度器的定时中断频繁触发，抑制了串口接收中断的响应。
  + 串口初始化时未正确使能接收中断，仅配置了发送功能，导致接收数据时无中断触发。
* **解决方案**：

  + 明确配置中断优先级，将串口接收中断优先级设为高于调度器中断。

## 坑 7：IIC时序错误导致ADDA接收不到数据

> 来源：[035_蓝桥杯单片机第十周周报（四）.md](./单片机答疑数据库/035_蓝桥杯单片机第十周周报（四）.md)
> 周期：2025年11月17日 — 2025年11月23日

### 整理记录

* **问题描述**：基于IIC协议实现AD转换芯片（如PCF8591）与单片机通信，程序执行后ADDA接口无数据返回，单片机读取的寄存器值始终为0xFF或固定值，排查硬件接线无误。
* **问题背景**：PCF8591作为AD转换芯片
* **问题分析**：

  + IIC起始信号和停止信号时序不标准，起始信号中SCL未保持高电平就拉低SDA，停止信号中SDA未在SCL高电平时拉高，导致从机（PCF8591）无法识别通信起始和结束。
  + 数据传输时SCL高电平期间SDA发生跳变，违反IIC协议中“高电平数据稳定”的要求，从机读取到错误数据，无法正确响应单片机的读取指令。
* **解决方案**：

  + 修正IIC起始和停止信号的时序代码，确保起始信号为“SCL高电平→SDA拉低”，停止信号为“SCL高电平→SDA拉高”
  + 规范数据传输时序，在SCL拉低时修改SDA电平，SCL拉高时保持SDA稳定

## 坑 10：本周重点问题详解

> 来源：[064_蓝桥杯单片机第二十二周周报（三）.md](./单片机答疑数据库/064_蓝桥杯单片机第二十二周周报（三）.md)
> 周期：2026年3月9日 — 2026年3月15日
> 关联模块：C51 语法 / 基础逻辑

### 整理记录

**(1) DAC数学公式运算出现异常极大值**
* **问题描述**：选手利用折线公式计算DAC输出值时，原本期望输出几伏的电压，但Debug中浮点变量突变为异常极大值（如 `3278.45`），导致DAC完全失效。
* **问题背景**：第十五届省赛及其他包含斜率换算的ADC/DAC综合题。
* **问题分析**：公式中存在诸如 `(Distance - 90)` 的相减运算。由于 `Distance` 定义为 `unsigned int`（16位无符号整型），运算结果无法为负，发生底层反转（如 `23 - 90 = 65469`）。随后被转为 `float` 计算，导致整个等式逻辑崩溃。
* **解决方案**：运算前必须破坏无符号属性。可显式强制转换为有符号数 `(int)Distance - 90`，或将减数写为浮点格式 `90.0f` 交由C语言编译器自动向高精度提升类型。
**(2) 矩阵键盘组合键(S8+S9)极难触发**
* **问题描述**：赛题要求同时按下两个按键触发清零功能，但实际操作时极易误判为其中某个单按键，甚至双按毫无反应。
* **问题背景**：第十一届、第十六届省赛等含有组合按键操作的真题。
* **问题分析**：单片机循环扫描频率极高，而人类双指按下必然存在微秒级“时间差”。常规的 `if-else` 消抖判定，遇到首个闭合按键即刻锁定状态机并跳出，永远无法同时捕捉到第二个按键的闭合。
* **解决方案**：采用“宽容状态机（历史最大值法）”。在按下期间不再直接判定动作，而是利用按位或（`|=`）不断将当前按键状态累加入“历史账本”中；直到双指完全离开键盘的松手瞬间，再根据历史账本的最终值（如既按了S8又按了S9，值为3）结算判定动作。
**(3) 串口调用标准库解析导致死机/乱码**
* **问题描述**：串口完成定长接收后，单片机对字符串解析失败。有时伴随数码管第一位乱码闪烁，或系统无故重启卡死。
* **问题背景**：第十五届国赛或其他要求通过串口接收并解析特定指令（如坐标、指令）的赛题。
* **问题分析**：C语言底层字符串缺乏长度边界概念，完全依赖结束符 `\0` (0x00) 作为停止判断。若串口接收缓存数组未在数据末尾补上结束符，调用 `sscanf` / `strcmp` 时函数将犹如“盲人无刹车列车”，冲出接收数组并疯狂读取、覆盖内存中的相邻变量（如 `Seg_Buf[0]`），引发堆栈溢出或内存踩踏。
* **解决方案**：在接收超限处做数组阈值拦截（如 `index < MAX-1`）；并在进入解析逻辑前，人为在末尾强制补充结束防撞墙：`uart_buf[uart_index] = '\0';`，随后再调用库函数。

## 坑 11：AD_Read 缺少 I2CWaitAck() 导致电压读取卡在 5.00V

> 来源：[066_蓝桥杯单片机第二十三周周报.md](./单片机答疑数据库/066_蓝桥杯单片机第二十三周周报.md)
> 周期：蓝桥杯单片机第二十三周周报

### 整理记录

- **问题描述：** 学员二刷题目时改用官方赛点资源包的 IIC 文件，RB2 电压读取始终卡在 5.00V，但之前用课程提供的 IIC 文件没有问题
- **问题背景：** PCF8591 ADC 读取函数编写
- **问题分析：** `AD_Read` 函数中发送控制字节 `addr` 后缺少 `I2CWaitAck()`。学员自认为"比对了几遍都一样"，实际漏掉了第 123 行的 `I2CWaitAck()`。没有等待 ACK，主机直接发送 Repeated Start，PCF8591 未完成控制字节的处理，后续读取返回全 1（0xFF），换算为电压即 `255 * 100 / 51 ≈ 500`，显示 5.00V
- **解决方案：** 在发送 `addr` 后补上 `I2CWaitAck()`：
```c
unsigned char AD_Read(unsigned char addr) {
unsigned char temp = 0;
I2CStart();
I2CSendByte(0x90);
I2CWaitAck();
I2CSendByte(addr);
I2CWaitAck(); // 补上这一行
I2CStart();
I2CSendByte(0x91);
I2CWaitAck();
temp = I2CReceiveByte();
I2CSendAck(1);
I2CStop();
return temp;
}
```
