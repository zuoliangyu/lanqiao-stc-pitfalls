## 坑 5：超声波数据存储时间相同

> 来源：[033_蓝桥杯单片机第十周周报（二）.md](./单片机答疑数据库/033_蓝桥杯单片机第十周周报（二）.md)
> 周期：蓝桥杯单片机第十周周报（二）

### 整理记录

* **问题描述**:超声波数据还不能存储到数组里,显示的时间是一样的,每次存储的时间戳都相同。
* **问题背景**:十七届4T场次一模拟赛。
* **问题分析**:

  + 更新RTC时间的函数放在了一个特定界面的处理函数里
  + 只有在特定界面时才会更新时间
  + 在其他界面存储数据时,时间没有更新,所以每次存的时间都是一样的
* **解决方案**:

  + 将更新RTC时间的函数放到主循环或系统调度函数的开头,确保每次循环都执行到时间更新函数,这样存储数据时才能获取到实时更新的时间戳。

## 坑 6：超声波测距数值异常

> 来源：[039_蓝桥杯单片机第十一周周报 （三）.md](./单片机答疑数据库/039_蓝桥杯单片机第十一周周报 （三）.md)
> 周期：蓝桥杯单片机第十一周周报 （三）

### 整理记录

* **问题描述**:超声波测距数值异常,无法正常测距。
* **问题背景**:十三届省赛。
* **问题分析**:

  + 发送方波函数中连续两次写Tx=1
  + 应该是先Tx=1再Tx=0,形成方波
  + 并且在读取函数中超声波的发送函数也没有调用
  + 没有触发超声波发送,自然无法接收到回波
* **解决方案**:

  + 修改发送方波函数,Tx=1后应该跟着Tx=0,不是Tx=1。并且在超声波读取函数中要调用发送函数,正确的超声波发送流程为:

```
// 发送超声波脉冲
void Send_Wave(void)
{
    Tx = 1;  // 拉高
    Delay_12us;  
    Tx = 0;  // 拉低,形成脉冲
}

// 读取超声波数据
unsigned int Read_Distance(void)
{
    Send_Wave();  // 先发送超声波
    // 等待接收回波并计算距离
    // ...
}
```

## 坑 7：超声波只有个位变化

> 来源：[039_蓝桥杯单片机第十一周周报 （三）.md](./单片机答疑数据库/039_蓝桥杯单片机第十一周周报 （三）.md)

### 整理记录

* **问题描述**:超声波测距时,显示的数值只有个位在变化,十位、百位不变。
* **问题背景**:十三届省赛。
* **问题分析**:

  + 超声波读取函数中的时间变量类型不正确
  + 使用了unsigned char或其他小范围类型
  + 导致数值溢出,只能显示0-255范围内的个位变化
* **解决方案**:

  + 超声波读取函数中的时间变量需要使用int类型或unsigned int类型。确保变量能够存储完整的计数值,避免数据溢出。

```
// 错误示例
unsigned char time;  // 范围0-255,会溢出

// 正确示例
unsigned int time;  // 范围0-65535,足够存储超声波计数值
```

## 坑 11：超声波忙等循环阻塞主循环导致数码管在 000 和 RB2 值之间跳变

> 来源：[066_蓝桥杯单片机第二十三周周报.md](./单片机答疑数据库/066_蓝桥杯单片机第二十三周周报.md)
> 周期：蓝桥杯单片机第二十三周周报

### 整理记录

- **问题描述：** 模式1下数码管显示在 `000` 和 RB2 实际电压值之间反复跳变
- **问题背景：** 题目同时使用超声波测距和 AD 电压采集，调度器架构
- **问题分析：** `Get_Distance()` 调用 `Ut_Wave_Data()`，其内部有一个忙等循环 `while((US_RX==1)&&(CF==0));`，等待超声波回波或 PCA 溢出。如果没有回波返回（传感器未接好或超出量程），该循环会阻塞主循环长达 **65ms**（直到 PCA 计数器溢出）。在调度器架构下，超声波测距约每 120ms 调用一次，其中约 65ms 被忙等循环占用，导致：
- AD 采集函数在阻塞期间无法执行，采到的值为 0 或过期数据
- 数码管刷新被中断，显示不连贯
- 交替出现正常值（非阻塞期采到的 RB2 电压）和 `000`（阻塞期未更新的数据）
- **解决方案：** 将超声波测距的忙等循环改为非阻塞方式，利用定时器中断做超时判断，避免长时间占用主循环：
```c
// 原来（阻塞式，最长卡65ms）
void Ut_Wave_Data() {
// ... 发送触发脉冲 ...
while((US_RX == 1) && (CF == 0)); // 死等回波，阻塞主循环
// ... 计算距离 ...
}
// 改进方案：加超时计数保护
void Ut_Wave_Data() {
unsigned int timeout = 0;
// ... 发送触发脉冲 ...
while((US_RX == 1) && (CF == 0)) {
if(++timeout > 10000) break; // 超时跳出，避免长时间阻塞
}
// ... 计算距离 ...
}
```
或者将超声波测距放到独立的定时窗口中执行，与 AD 采集和数码管刷新在时间上错开，确保互不干扰。
> **附注：** 在调度器架构中，任何一个任务函数中的长时间阻塞都会影响所有其他任务的执行。超声波的忙等循环是典型的"一个阻塞拖垮全局"问题。凡是包含 `while` 等待硬件响应的驱动函数，都应加上超时保护或改为非阻塞设计。

## 坑 12：徘徊判定用错变量 + 距离参数未生效 + 温度采集间隔过短 + 超声波延时参数错误（6处修复）

> 来源：[066_蓝桥杯单片机第二十三周周报.md](./单片机答疑数据库/066_蓝桥杯单片机第二十三周周报.md)
> 周期：蓝桥杯单片机第二十三周周报

### 整理记录

- **问题描述：** 4T 测评 57/71，继电器能计次数但记不准，徘徊状态无法触发，修改距离参数后不生效。学员多次自查和使用 AI 辅助排查仍未定位到核心问题
- **问题背景：** 第十六届省赛，涉及超声波测距、运动状态判定（静止/徘徊/跑动）、继电器控制、温度采集
- **问题分析：**
- [a] **徘徊判定条件用错变量（核心问题）：** `else if(D_Distance >= 5 && Distance < 10)` 中第二个条件写成了 `Distance`（绝对距离，通常远大于 10cm），应为 `D_Distance`（两次距离差值）。`Distance < 10` 几乎永远为 `false`，导致所有 ΔL ≥ 5cm 的情况跳过徘徊，直接进入跑动判定，徘徊状态永远无法触发
- [b] **距离参数使用了未被修改的变量：** 代码定义了两套参数变量：`Dist_Para`（初始值 30，从未被按键修改）和 `Dist_Para_Set`（S8/S9 实际修改的变量）。"接近"判定 `Approached = (Distance < Dist_Para)` 用了前者，导致用户调整距离参数后完全不生效，继电器吸合逻辑、LED 控制、NC 统计计数全部出错。这是扣分最多的 Bug，影响约 10 个测评项
- [c] **温度采集间隔过短导致竞态条件：** 温度每 160ms 采集一次，距离每 999ms 采集一次。当测试条件中温度和距离同时变化时，温度先更新（160ms），距离还是旧值（等 999ms），造成短暂的"高温+接近"假触发，继电器多吸合一次，NC 计数偏多
- [d] **温度计时变量类型溢出：** `Temp_Slow_Down` 为 `unsigned char`（最大 255），将间隔改为 300ms 后需同步改为 `unsigned int`，否则条件永远为真，温度函数永不执行
- [e] **超声波 Delay12us 延时参数错误：** `i = 38` 远大于正确值 `i = 3`，导致发射频率严重偏离 40kHz，传感器无法正确谐振，距离测量值系统性偏小约 3cm
- [f] **IIC 延时函数循环体为空：** 官方模板的 `I2C_Delay` 循环体缺少 `_nop_()`，PCF8591 通信时序不稳定，ADC 读值偶发不准
- **解决方案：**
- [a] 将 `Distance < 10` 改为 `D_Distance < 10`：
```c
// 修改前（Distance 是绝对距离，几乎永远 >= 10）
else if(D_Distance >= 5 && Distance < 10)
Temp_Motion_State = 1;
// 修改后（用差值判定徘徊）
else if(D_Distance >= 5 && D_Distance < 10)
Temp_Motion_State = 1;
```
- [b] 将 `Dist_Para` 改为 `Dist_Para_Set`：
```c
Approached = (Distance < Dist_Para_Set); // 用被按键实际修改的变量
```
- [c] 温度采集间隔从 160ms 改为 300ms，与距离采集节奏匹配，消除竞态窗口
- [d] `Temp_Slow_Down` 类型从 `unsigned char` 改为 `unsigned int`
- [e] 超声波 `Delay12us` 中 `i = 38` 改为 `i = 3`
- [f] IIC 延时循环体添加 `_nop_()`
> **附注：** 本例的核心教训是**变量命名混淆**——`Distance` vs `D_Distance`、`Dist_Para` vs `Dist_Para_Set`，名字相近但含义完全不同，编译器不会报错，只能靠人工审查。建议：① 避免定义功能重叠的变量，只用一个变量管理参数；② 变量名应清晰体现含义差异（如用 `delta_dist` 代替 `D_Distance`）。

## 坑 13：第十六届省赛（第二次）数码管测距数据卡死——measure 标志位被立即清零

> 来源：[069_蓝桥杯单片机第二十四周周报（二）.md](./单片机答疑数据库/069_蓝桥杯单片机第二十四周周报（二）.md)
> 周期：蓝桥杯单片机第二十四周周报（二）

### 整理记录

- **问题描述：** 学员在超声波测距函数中加了 `if(measure==1)` 判断后，按键启动测距后数码管显示停在第一次测到的值，不再更新，看起来像"卡死"
- **问题背景：** 第十六届省赛第二次题目
- **问题分析：** 程序并非真正卡死，而是测距结果只更新了一次。按键按下后将 `measure` 置为 1，但在 `distance_get()` 函数内部，一次测距完成后立刻将 `measure` 清为 0：
```c
void distance_get() {
unsigned char dist_temp;
dist_temp = us_dat(v);
if(measure == 1) {
dist = dist_temp;
seg_model_flag = 1;
measure = 0; // ← 测完一次就清零，后续不再进入
seg_model_work = 1;
}
}
```
这样后续调度虽然还在正常运行、数码管扫描也正常，但 `dist` 不会再被更新，显示值就一直停在第一次测到的数上
- **解决方案：** 不要在测距函数内部清零 `measure`，应根据题目逻辑在需要停止测距时（如再次按键或切换模式时）才清零，保证测距期间 `measure` 持续为 1，距离值持续更新

## 坑 14：第十六届超声波测距数码管残留与测距显示异常

> 来源：[070_蓝桥杯单片机第二十五周周报.md](./单片机答疑数据库/070_蓝桥杯单片机第二十五周周报.md)
> 周期：蓝桥杯单片机第二十五周周报

### 整理记录

- **问题描述：** 测距功能异常：单次显示固定数值 231，累计显示始终为 0，数码管存在残留现象
- **问题背景：** 第十六届省赛题目，超声波测距涉及单次测量和累计测量两种模式
- **问题分析：** 学员原代码存在 3 个问题：
- [a] **数码管残留：** `Seg_Proc()` 中 `switch(range_show_mode)` 的 case 1 和 case 2 分支没有完整清零所有 Seg_Buf 位，case 1 只写了 Seg_Buf[4..7]，case 2 同样如此。当 `range_show_mode` 切换时，前一轮显示的数据会残留在数码管上
- [b] **距离变量类型过窄：** `Distance` 定义为 `unsigned char`（最大值 255），超声波测距结果可达三位数以上，超出范围会截断。累计显示需要 `Distance_sum` 变量但原代码未定义
- [c] **缺少累计功能支撑：** 超声波驱动 `Ut_Wave_Data()` 无参数，无法根据温度补偿声速，导致距离计算固定使用硬编码系数 0.017，结果不准确
- **解决方案：**
- [a] 完善 `Seg_Proc()` 中每个 `case` 分支的 Seg_Buf 清零逻辑，切换显示模式时完整覆写所有位。同时将 `range_show_mode` 初始值设为 1（默认显示距离界面），而非 0
- [b] 超声波驱动中 `Ut_Wave_Data()` 的返回值类型从 `unsigned char` 提升为 `unsigned int`；增加 `float speed` 参数支持温度补偿声速计算；距离计算公式改为 `(time * speed) / 20000.0f`
- [c] 新增 `Distance_sum` 变量记录累计距离，新增 `signal_flag`/`summation_flag`/`summation_index` 区分单次/累计模式；累计模式下按 `para_time_arr[para_time_index]` 控制采样间隔，`para_num_arr[para_num_index]` 控制采样次数

## 坑 15：超声波测距项目数码管显示状态错位与调度器越界

> 来源：[070_蓝桥杯单片机第二十五周周报.md](./单片机答疑数据库/070_蓝桥杯单片机第二十五周周报.md)
> 周期：蓝桥杯单片机第二十五周周报
> 关联模块：综合赛题 / 系统

### 整理记录

- **问题描述：** 在数码管显示函数中直接写死 `24` 可以正常显示，但在函数外部把变量定义为 `24` 时数码管不能稳定显示
- **问题背景：** 超声波测距项目，涉及单次/多次测距切换、参数界面调整、数码管多界面显示、调度器任务调度
- **问题分析：**
- [a] **默认显示界面不对：** `Seg_cheju = 0` 代表初始处于"首次界面"，该界面会把后四位数码管直接清空。`Seg_chejushu_dang` 虽然初始化为 `24`，但程序没有走到"显示 `24` 的分支"
- [b] **单次测量按键先清零再覆盖：** 单次测量按键处理中先执行 `Seg_chejushu_dang = 0;`，再调用测距函数用新结果覆盖初始化值，导致全局定义的 `24` 保不住
- [c] **多次测量标志自相矛盾：** 按键处理中先写 `Seg_duoci = 1;` 又马上写 `Seg_duoci = 0;`，多次测量标志刚打开就被自己关掉
- [d] **调度器数组越界写入：** 任务表只有 5 项，但代码访问了 `Scheduler_Task[6]`，属于数组越界，会写坏相邻内存，导致变量值异常和显示不稳定
- [e] **时间基准变量未更新：** 使用没有随时间更新的 `next` 作为任务时间基准，而非系统时基 `uwTick`
- **解决方案：**
- [a] 将 `Seg_cheju` 默认值改为 `1`，上电直接进入"单次测量显示界面"，让初始化值 `24` 有机会显示
- [b] 单次测量开始前不要先把 `Seg_chejushu_dang` 清零，改为测距结果直接覆盖
- [c] 测量标志位不能刚置 `1` 又立刻改回 `0`，测距完成 `Seg_danci = 0;` 由 `Get_Distance()` 内部控制
- [d] 使用 `#define` 定义任务索引常量（`TASK_LED`、`TASK_KEY` 等），避免越界；确保访问下标不超过任务表长度
- [e] 任务时间基准统一改为 `uwTick`，`Scheduler_Task[TASK_DISTANCE].last_ms = uwTick;`
- **经验教训：** 遇到"函数里写死正常，变量方式不正常"时，优先检查三件事：变量有没有用于当前显示分支、变量有没有被后续逻辑改掉、是否存在数组越界写坏内存
