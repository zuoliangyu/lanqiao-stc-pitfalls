## 坑 1：电位器通道先做浮点缩放导致显示跳零

> 第十六届模拟赛 2 · 发现者：谁无年少轻狂时

### 现象

对 `0x03` 电位器 ADC 通道做数据处理时，如果先写成 `temp * 20 / 51.0` 这类浮点运算，数码管会出现多次跳 0 的现象，测评会因此扣分。同题里的光敏电阻通道不太明显，更容易让人误以为是电位器通道本身有问题。

### 原因

8051 上浮点除法开销很大，放在高频刷新的 ADC 数据处理中，会明显拉长一次采样后的运算时间。电位器通道变化更快、显示刷新更频繁，所以更容易把瞬时显示节奏打乱，表现成“偶发跳 0”。

```c
// 错误写法：在高频 ADC 显示路径里直接做浮点除法
unsigned char temp;
temp = Ad_Read(0x03);
Value = temp * 20 / 51.0;
```

### 解决方法

显示用的数据尽量走整数或定点运算，先乘后除，最后再落到显示变量。确实需要小数时，也优先用“放大 10 倍/100 倍”的定点写法，而不是直接上浮点。

```c
// 正确写法：显示值直接用整数缩放
unsigned char temp;
temp = Ad_Read(0x03);
Value = temp * 20 / 51;
```

### 举一反三

- 8051 上的浮点、32 位除法都很贵，别随手塞进高频刷新路径
- 采样、显示、扫描这类“每轮都跑”的逻辑，优先用整数和定点
- 某个通道更容易出问题，很多时候不是通道特殊，而是该通道的处理路径更重

## 坑 2：PWM 开启后不屏蔽中断导致 AD 采样被污染

> 第十三届国赛 · 发现者：萌新也有国一梦

### 现象

程序一旦开启 PWM，并启用每 `100us` 触发一次的 Timer2 中断，`ADDA` 采样值就会乱跳，甚至周期性跳 0。单独看 ADC 代码没问题，但一和 PWM 同时开就炸。

### 原因

`Ad_Read()` 这类访问通常依赖软件时序。Timer2 中断频率高、优先级也高时，会在采样过程中硬插进来，把总线或采样时序切碎，最后读到被污染的数据。

```c
// 错误写法：采样过程中允许高频中断随时打断
temp = Ad_Read(0x43);
```

### 解决方法

在 AD 采样前后临时关总中断，保证一次采样过程连续完成。若题目允许，也可以改成“停止 PWM 后采样，采样完再恢复”。

```c
// 正确写法：把一次采样保护成原子操作
EA = 0;
Ad_Read(0x43);
Ad_Read(0x43);
temp = Ad_Read(0x43);
EA = 1;
```

### 举一反三

- 软件模拟 I2C、SPI、单总线时，最怕高频中断插队
- “单独运行正常，一开 PWM/定时器就乱跳”时，优先怀疑时序被中断打断
- 关中断要只包住最小关键区，别一关就是大段逻辑

> 来源：[014_蓝桥杯单片机第六周周报.md](./单片机答疑数据库/014_蓝桥杯单片机第六周周报.md)
> 周期：2025年10月19-2025年10月25

### 整理记录

* **问题描述**：使用ADC转换时，单独的光敏电阻可以正常显示数据，但与滑动变阻器一起显示时，滑动变阻器正常而光敏电阻无法显示。
* **问题背景**：无具体背景
* **问题分析**：

  + 多通道ADC切换时序问题
  + 可能存在通道切换延时不足
  + ADC转换完成标志处理不当
  + 数据读取函数调用顺序或频率问题
* **解决方案**：

  + 为光敏电阻添加独立的ADC读取函数，确保每个通道都有足够的转换时间和正确的数据读取流程。

## 坑 5：ADC采样值归零问题

> 来源：[019_蓝桥杯单片机第七周周报（二）.md](./单片机答疑数据库/019_蓝桥杯单片机第七周周报（二）.md)
> 周期：2024年10月27日 — 2024年11月02日

### 整理记录

* **问题描述**：读取滑动变阻器电压，老是出现归零的情况，只要把DA读取函数注释掉就不会出现这种情况。
* **问题背景**：无具体课程信息。
* **问题分析**：

  + DA和AD读取函数存在冲突
  + 两个函数在同一函数中读取会相互干扰
  + 时序冲突导致采样值异常
* **解决方案**：

  + 两个函数要分开读取，放在同一个函数里会跳0。需要将DA和AD读取操作分离到不同的函数或不同的执行时机。

## 坑 8：PCF8591上电假触发导致数据异常

> 来源：[059_蓝桥杯单片机第二十一周周报.md](./单片机答疑数据库/059_蓝桥杯单片机第二十一周周报.md)
> 周期：蓝桥杯单片机第二十一周周报

### 整理记录

-
- **问题描述**：单片机刚上电未进行任何操作，系统自动发生一次“由暗变亮”的光敏触发，且因NE555起振未完成，记录了错误数据导致后续平均值严重偏低。
- **问题背景**：涉及环境光阈值触发的历届真题。
- **问题分析**：PCF8591芯片上电后内部寄存器默认值为128（约2.51V），而真实明亮环境光电压通常低于80，导致程序捕获到一次虚假的128 -> 真实值下降沿。
- **解决方案**：在main函数进入while(1)主循环前，对PCF8591执行一次“空读/假读”操作以消耗掉默认的128，并正确初始化Old变量。

## 坑 9：串口乱码 + ADC 电压最大只到 1.30V（底层代码笔误）

> 来源：[062_蓝桥杯单片机第二十二周周报.md](./单片机答疑数据库/062_蓝桥杯单片机第二十二周周报.md)
> 周期：蓝桥杯单片机第二十二周周报
> 关联模块：串口

### 整理记录

- **问题描述：** 串口打印出现乱码，同时电位器 ADC 读数最大只能到 1.30V
- **问题背景：** 第十四届省赛模拟题2
- **问题分析：** 存在两处底层代码笔误：
- [a] **串口乱码：** `putchar()` 函数中 `while(TI == 0)` 后面漏写分号，导致 `TI = 0;` 被当作 while 循环体。TI 置 0 后循环条件重新成立，形成死循环，串口发送时序被破坏，输出乱码
```c
// 错误写法
while(TI == 0) // ← 缺少分号
TI = 0; // 被当作循环体，死循环
```
- [b] **ADC 上限异常：** `ad_da()` 中 ADC 换算公式写成了 `ad_read(0x43) * 51.0f / 100`（乘以 51），正确应为除以 51。ADC 返回值 0-255 对应 0-5.00V，换算公式应为 `ad_read(0x43) / 51.0f * 100`，乘以 51 导致结果远超正常范围，显示时被截断
- **解决方案：**
- [a] `while(TI == 0)` 后补上分号：`while(TI == 0);`
- [b] 将 `* 51.0f / 100` 改为 `/ 51.0f * 100`（或等价写法 `* 100 / 51.0f`）

## 坑 10：底层 Seg_Disp 操作 P2 时未保留低位，破坏 I2C 总线导致 AD 读数跳变

> 来源：[066_蓝桥杯单片机第二十三周周报.md](./单片机答疑数据库/066_蓝桥杯单片机第二十三周周报.md)
> 周期：蓝桥杯单片机第二十三周周报
> 关联模块：数码管 / 工具链 / 环境

### 整理记录

- **问题描述：** AD 读数在真实值和 5.00V 之间随机跳变，在 V4T 版本上没有此现象，移植到调度器架构后出现
- **问题背景：** 学员自己敲的 Seg_Disp / Led_Disp / Init 底层代码
- **问题分析：** PCF8591 的 I2C 数据线 SDA 和时钟线 SCL 接在 P2.0 和 P2.1，而数码管/LED 底层操作 P2 高位（选通 74HC573 锁存器）。学员底层代码写成 `temp = temp & 0x1f` 而非 `temp = P2 & 0x1f`，没有先读取 P2 当前值再保留低 5 位，而是对未初始化的局部变量 `temp` 做掩码。这导致写 P2 时低位（P2.0/P2.1）被随机值覆盖，正在进行的 I2C 通信时序被破坏，AD 读取偶发返回 0xFF（5.00V）。在 V4T 版本中由于代码执行时序不同，I2C 通信碰巧未被打断，所以没有暴露此问题
- **解决方案：** 底层操作 P2 高位时，必须先读取 P2 当前值保留低位：
```c
// 错误写法（temp 未初始化，低位被随机值覆盖）
temp = temp & 0x1f;
temp = temp | 0xe0;
P2 = temp;
// 正确写法（先读 P2，保留低5位不动）
temp = P2 & 0x1f;
temp = temp | 0xe0;
P2 = temp;
```
> **附注：** 凡是操作 P2 高位做锁存器选通的地方，都必须用 `P2 & 0x1f` 先保留低位，否则会破坏挂在 P2 低位上的 I2C 总线。

## 坑 11：第十六届省赛 光照等级无法正常变化——ADC 通道错误+等级赋值错误

> 来源：[069_蓝桥杯单片机第二十四周周报（二）.md](./单片机答疑数据库/069_蓝桥杯单片机第二十四周周报（二）.md)
> 周期：蓝桥杯单片机第二十四周周报（二）

### 整理记录

- **问题描述：** 光照强度等级 N1 不能正常变成 N2，等级始终不对
- **问题背景：** 第十六届省赛题目
- **问题分析：** 存在两个错误叠加：
- [a] **ADC 通道写错：** 光敏电阻的 I2C 地址应为 `0x41`，但代码中写成了 `0x43`（滑动变阻器通道），导致读取的不是光照值而是电位器值
- [b] **光照等级赋值错误：** 四个分段的 `Light_Level` 赋值写成了 `0, 1, 0, 0`，而非正确的 `0, 1, 2, 3`，导致即使 ADC 值正确，等级也无法区分：
```c
// 错误写法
if(AD_DA_100x >= 300) Light_Level = 0;
if(AD_DA_100x >= 200 && ...) Light_Level = 1;
if(AD_DA_100x >= 50 && ...) Light_Level = 0; // ← 应为 2
if(AD_DA_100x < 50) Light_Level = 0; // ← 应为 3
```
- **解决方案：**
- [a] 将 ADC 读取通道从 `0x43` 改为 `0x41`
- [b] 修正四个分段的等级赋值为 `0, 1, 2, 3`

## 坑 12：第十四届省赛光敏下降沿旧值未更新导致重复触发

> 来源：[070_蓝桥杯单片机第二十五周周报.md](./单片机答疑数据库/070_蓝桥杯单片机第二十五周周报.md)
> 周期：蓝桥杯单片机第二十五周周报

### 整理记录

- **问题描述：** 完成一次"由亮到暗"触发后，进入 3 秒显示期间环境仍保持较暗，3 秒结束后程序很快再次触发，只要现场一直较暗就会反复触发
- **问题背景：** 第十四届省赛题目，光敏下降沿触发采集，触发后进入 3 秒显示期，等待下一次亮暗变化
- **问题分析：**
- [a] **上一拍光照值更新时机错误：** 当前写法中，只有在未触发状态下才会更新 `light_value_old`。首次触发成功后，`light_value_old` 就停留在"亮"的状态不再刷新。3 秒结束后如果现场仍然较暗，则 `当前值 < 阈值 && 上一拍值 > 阈值` 这个条件再次成立，程序误以为又发生了一次新的亮暗变化
- [b] **边沿判断的 `old` 值被冻结：** 3 秒显示期间 `light_value_old` 没有被持续追踪最新采样值，导致显示结束后把同一次暗状态误判成新的边沿事件
- **解决方案：**
- [a] 将 `light_value_old = temp_ad;` 移到边沿判断之外，**不管是否已触发，每次采样后都更新上一拍光照值**。这样 3 秒结束前上一拍已经被更新成暗值，不再满足"由亮到暗"的边沿条件
- [b] 触发判断保持在 `tigger_flag == 0` 且 `temp_ad < LIGHT_DARK_THRESHOLD && light_value_old > LIGHT_DARK_THRESHOLD` 的条件不变
- 核心原则：边沿触发类题目中，上一拍采样值必须"每次都更新"，不能因为触发了就停止刷新

## 坑 13：AD采集异常跳变

> 来源：[071_蓝桥杯单片机第二十五周周报（二）.md](./单片机答疑数据库/071_蓝桥杯单片机第二十五周周报（二）.md)
> 周期：蓝桥杯单片机第二十五周周报（二）

### 整理记录

- **问题描述：** 学员反馈在使用 PCF8591 读取光敏/电位器电压时，AD 转换的值出现极其严重的无规律跳变（跳变幅度甚至高达 1V 左右），导致相关逻辑无法正常工作。
**问题背景：** 蓝桥杯单片机外设编程，在一个项目中同时使用了 PCF8591 的 AD 采集（数模转换）和 DA 输出（模数转换）功能。
**问题分析：** ：
- [a] **DA 使能位丢失（控制字节写错）：** 学员在读取 AD 通道 3 时，传入的控制字节是 `0x03`。但如果系统中同时用到了 DA 输出，读取 AD 时的控制字节**必须**是 `0x4X`（如 `0x43`）。写成 `0x03` 会在读取 AD 的瞬间强行关闭 DA 输出，导致芯片内部电压基准震荡，从而引起严重的 AD 采样跳变。
- [b] **I2C 时序未闭环（总线死锁隐患）：** 经检查学员源码，其 `Da_Write`（DA输出函数）在发送完数据后，**漏写了 `I2CStop();`**。这导致 I2C 总线一直处于被占用的 Busy 状态，接下来的 AD 读取时序自然会被严重破坏。
**解决方案：**
- [a] **规范控制字节：** 系统里启用了 DA 输出，后续所有的 AD 读取操作，控制位必须加上 `0x40`。即将 `AD_Read(0x03)` 必须改为 `AD_Read(0x43)`。
- [b] **补全 I2C 时序：** 在 `Da_Write` 函数的末尾补上 `I2CStop();`，释放 I2C 总线，确保每次通信都是完整的闭环。
